# CI and automation

verbatra is a development dependency, so the same commands you run locally run in CI. Install it, provide the provider key from your CI secret store as the matching environment variable, and run `verbatra translate`. Use `--dry-run` when you want to check for missing or stale translations without writing files.

```bash
pnpm add -D @verbatra/cli
# ANTHROPIC_API_KEY is provided by the CI secret store as an environment variable
verbatra translate --json
```

## Exit codes

verbatra uses its exit code as the contract a CI step branches on:

| Code | Meaning |
| --- | --- |
| `0` | success |
| `1` | `translate` finished but some locales failed (translate only) |
| `2` | could not run (a whole-run error, or a usage error) |
| `130` | `watch` was force-stopped by a second interrupt |

A per-run failure in `watch` is reported as a stream record, not an exit code; a single interrupt stops `watch` gracefully and exits `0`.

## JSON output

Both commands accept `--json` for machine-readable output.

`verbatra translate --json` prints the run summary as one JSON object with this shape:

```ts
interface RunSummary {
  dryRun: boolean;          // whether this was a dry run (no provider calls, no writes)
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
}
```

`verbatra watch --json` prints one record per run as NDJSON (one JSON object per line). Each record is either a success or a failure:

```ts
type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };
```

A failed run during `watch` is one of these records on the stream; it does not stop the watcher and does not set a non-zero exit code.

# El archivo de bloqueo

verbatra mantiene un único archivo confirmado, `verbatra.lock.json`, en tu proyecto. Es el registro de lo que verbatra ya ha traducido, y es lo que hace que una ejecución sea incremental: cada ejecución compara tu origen con esta línea base y llama al proveedor solo para las claves que realmente cambiaron.

## Qué registra

Para cada locale de destino, el archivo de bloqueo almacena un hash de contenido para cada clave de origen traducida. El hash viene del valor de origen, así que captura exactamente el texto que produjo la traducción existente. Sin traducciones, sin claves de API. Solo hashes.

## Cómo lo usa una ejecución

Cuando verbatra lee un locale de destino, compara el origen con ese locale usando el archivo de bloqueo como línea base de lo que se tradujo antes. Cada clave de origen se clasifica en uno de cuatro grupos:

- nueva: el destino aún no tiene la clave.
- modificada: el valor de origen difiere del hash registrado en el bloqueo.
- sin cambios: el valor de origen aún coincide con el hash registrado.
- huérfana: la clave está en el destino pero ya no en el origen.

Solo las claves nuevas y modificadas van al proveedor. Las claves sin cambios se dejan intactas, por eso una segunda ejecución sin ediciones de origen no llama al proveedor para nada.

El hash de contenido trata el Unicode canónicamente equivalente (NFC) y los finales de línea distintos (CRLF frente a LF) como el mismo contenido. Así que volver a guardar un archivo de origen con una forma de normalización o finales de línea diferentes no marca una clave como modificada ni desencadena una retraducción.

## Cómo se actualiza

Después de escribir un locale, verbatra actualiza el bloqueo con los hashes de origen actuales, con una excepción deliberada. Una clave retenida en esta ejecución (porque su traducción falló la comprobación de marcadores de posición o su ICU de origen no era válido) conserva su hash anterior. La siguiente ejecución la sigue viendo como que necesita trabajo y la reintenta, en lugar de registrarla como hecha. Las claves huérfanas no obtienen entrada.

## Primera ejecución, bloqueo ausente, bloqueo corrupto

Un archivo de bloqueo ausente cuenta como una primera ejecución: cada clave de destino es nueva. Un bloqueo que está presente pero corrupto o más grande que el límite de tamaño no se sobrescribe silenciosamente. La ejecución falla con `LOCK_FILE_INVALID` para que puedas inspeccionarlo en lugar de perder la línea base.

## Confírmalo

Confirma `verbatra.lock.json` junto con tus archivos de locale. Es la línea base compartida con la que comparan tu equipo y tu CI, así que mantenerlo en el control de versiones es lo que hace que las ejecuciones incrementales sean reproducibles entre máquinas. Consulta [Cómo funciona](/docs/how-it-works) para ver la canalización completa de la ejecución, y [Seguridad de la traducción](/docs/translation-safety) para las comprobaciones que deciden qué se retiene.

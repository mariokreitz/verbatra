title: Traduction humaine
description: Exporte les chaînes non traduites vers un classeur Excel, fais-les traduire par un humain, puis réimporte le résultat, avec les mêmes vérifications de sûreté qu'une exécution automatisée.
---
Toutes les chaînes n'appartiennent pas à un provider. Les textes marketing, les textes juridiques et les noms de produits veulent souvent un traducteur humain, et certaines équipes gardent un pro dans la boucle pour les langues qui comptent le plus. verbatra gère ça avec un aller-retour de classeur : **exporte** les chaînes qui ont besoin d'être traduites dans un fichier Excel stylisé, remets-le à un traducteur, puis **importe** le fichier rempli dans tes fichiers de locale.

Le même diff pilote les deux directions. L'export sélectionne exactement les clés qu'une exécution automatisée traduirait (les nouvelles et les modifiées), de sorte qu'une locale traduite manuellement reste synchronisée avec ta source de la même manière qu'une locale traduite par IA. L'import exécute les mêmes vérifications de placeholders et d'ICU que `translate`, donc si un traducteur saisit à la main une valeur qui casse un placeholder, verbatra la retient et te le signale plutôt que de l'écrire.

## L'aller-retour

```bash
# 1. Export the strings that need translating
verbatra export

# 2. A translator fills the Translation column and sends the file back

# 3. Import the filled workbook
verbatra import verbatra-translations.xlsx
```

Par défaut, l'export dépose `verbatra-translations.xlsx` dans le répertoire de travail. L'import le relit, valide chaque valeur, écrit celles qui passent dans tes fichiers de locale, et met à jour le [fichier de verrouillage](/docs/the-lock-file) pour les clés qu'il a acceptées, exactement comme le ferait une exécution `translate`.

## Le classeur

Le fichier s'ouvre sur une feuille **Instructions**, suivie d'une feuille de données par locale cible, chacune nommée d'après sa locale (`de`, `fr`, ...). Chaque feuille de données partage les mêmes colonnes :

| Colonne | Modifiable | Objet |
| --- | --- | --- |
| `Key` | non | Le chemin de clé pointé. La seule identité qui rattache une ligne à une chaîne. |
| `Source` | non | La valeur de la locale source, pour référence. |
| `Current translation` | non | La valeur cible existante, le cas échéant. |
| `Status` | non | `new` (pas encore de traduction), `changed` (la source a changé et la traduction doit être mise à jour) ou `unchanged` (déjà à jour, incluse seulement avec `--include-unchanged`). |
| `Translation` | **oui** | La seule cellule que le traducteur remplit. |
| `Source hash` | non | Masquée. Le contenu source capturé à l'export, utilisé pour détecter une source qui a changé après l'export. |
| `Context` | non | Contexte développeur pour la chaîne, quand le format source en porte un (`@key.description` de Flutter ARB, `<note>` de XLIFF). Affiché pour référence seulement, jamais importé comme traduction. |
| `Review status` | non | `ok` ou `review` : si les heuristiques de révision de verbatra ont signalé la traduction de cette ligne pour un second regard. Lecture seule et consultatif, jamais un verrou à l'import. |
| `Review reasons` | non | Une liste d'étiquettes de motifs séparées par des virgules qui l'expliquent (par exemple `length-ratio-outlier, equals-source`). Vide quand `Review status` vaut `ok`. |

Chaque colonne sauf `Translation` est verrouillée et la feuille est protégée, de sorte qu'un traducteur ne peut modifier que la seule colonne qui compte. La ligne d'en-tête est figée et la colonne `Source hash` est masquée. `Context`, `Review status` et `Review reasons` sont ajoutées après `Source hash` plutôt qu'insérées plus tôt, de sorte que `Translation` garde sa position pour quiconque écrit des scripts contre la forme du classeur. Trie ou filtre les lignes comme tu veux : le rattachement se fait par `Key`, jamais par position.

Une cellule `Translation` vide signifie « pas encore traduite ». L'import l'ignore et n'écrit jamais une chaîne vide, de sorte qu'un traducteur peut renvoyer un classeur à moitié rempli et terminer le reste plus tard.

## Sélectionner ce qui est exporté

Par défaut, `export` n'inclut que les chaînes manquantes et modifiées, le travail qui doit réellement être fait. Deux flags ajustent la sélection :

```bash
# Only the German and French sheets
verbatra export --locales de,fr

# Also include strings that are already up to date
verbatra export --include-unchanged

# Write to a specific path
verbatra export --out handoff/round-2.xlsx
```

Utilise `--include-unchanged` quand un traducteur veut le contexte complet d'une locale, pas seulement le delta. Voir la référence [`verbatra export`](/docs/cli/export) pour chaque flag.

## Sûreté à l'import

L'import n'est pas un collage aveugle. Avant que verbatra n'écrive quoi que ce soit, chaque valeur saisie par le traducteur passe par les mêmes vérifications qu'une traduction automatisée :

- **Intégrité des placeholders** : une traduction doit porter exactement les mêmes placeholders que sa source. Supprime `{name}` ou invente `{total}` et la valeur est retenue.
- **Validité ICU** : un message ICU doit rester structurellement valide, avec les mêmes noms d'argument. Seul le texte lisible par l'humain peut changer.
- **Dérive de la source** : le `Source hash` masqué permet à l'import de repérer une chaîne source qui a changé après l'export du classeur. Une ligne périmée est retenue, de sorte que tu n'écrases jamais une source actuelle par une traduction d'une ancienne.
- **Clés inconnues** : une ligne dont la `Key` n'est pas dans la source actuelle (par exemple, une qu'un traducteur a ajoutée à la main) est rejetée.
- **`Context` n'est jamais une source de traduction** : `Context` est un texte de référence en lecture seule. L'import l'ignore entièrement, et un classeur exporté avant l'existence de cette colonne s'importe toujours normalement.

Une valeur qui échoue à une vérification est **retenue** (non écrite) et signalée dans le résumé de l'exécution, de sorte que tu peux la corriger et la réimporter. Les clés qui passent sont écrites et verrouillées ; celles qui échouent laissent le fichier de locale intact. Ajoute `--dry-run` pour valider un classeur retourné et prévisualiser ce qui serait écrit sans toucher à un seul fichier :

```bash
verbatra import verbatra-translations.xlsx --dry-run
```

L'import partage le contrat des codes de sortie de `translate` : il renvoie `0` quand toutes les locales réussissent, `1` quand une locale échoue, et `2` quand l'exécution n'a pas pu démarrer. Cela signifie qu'un transfert manuel s'intègre aux mêmes vérifications CI qu'un transfert automatisé. Voir [CI et automatisation](/docs/ci-and-exit-codes).

## Signaux de révision

Les vérifications ci-dessus décident de ce que l'import écrit. `Review status` et `Review reasons` sont une couche différente, non bloquante, par-dessus : les propres heuristiques de verbatra pour repérer une traduction structurellement correcte mais qui mérite un second regard. Elles ne retiennent jamais une ligne et n'affectent jamais ce que l'import accepte. Un traducteur ne peut pas effacer un signal en modifiant une cellule ; le seul moyen de l'effacer est de corriger la valeur de `Translation` et de réexporter.

Les deux colonnes sont recalculées à chaque export à partir des valeurs source et cible actuelles, pas reportées d'une exécution `translate`. `Review reasons` liste les étiquettes qui s'appliquent, séparées par des virgules :

- `length-ratio-outlier` : la traduction est nettement plus courte ou plus longue que sa source.
- `equals-source` : la traduction est identique à la source.
- `glossary-term-missed` : un terme de glossaire configuré n'a pas fait son chemin dans la traduction.
- `integrity-reordered` : l'ensemble des placeholders correspondait mais est arrivé dans un ordre différent de celui de la source.
- `provider-degraded` : le lot dont vient cette clé portait un avis de dégradation DeepL au moment de la traduction. Comme l'export n'appelle jamais de provider, ce motif n'apparaît jamais sur une ligne exportée ; il n'apparaît que dans la sortie `translate --json` ou `watch --json`. Une clé qui n'a déclenché que ce motif au moment de la traduction exporte `ok`, sauf si elle déclenche aussi indépendamment l'un des quatre motifs ci-dessus, auquel cas la ligne exportée porte ce motif à la place.

Une ligne sans rien de traduit encore exporte toujours `ok` : il n'y a rien à réviser. Voir [Sûreté de la traduction](/docs/translation-safety) pour ce que signifie chaque motif et comment ces signaux se rapportent aux vérifications de placeholders et d'ICU ci-dessus.

## Depuis le SDK

Les commandes de la CLI enveloppent deux fonctions du SDK, `exportWorkbook` et `importWorkbook`. Appelle-les directement quand tu intègres la traduction manuelle dans tes propres outils :

```ts
import { exportWorkbook, importWorkbook, loadConfig } from "@verbatra/sdk";

const config = await loadConfig();

// Write a workbook of the strings that need translating
await exportWorkbook({ config });

// ...later, import the filled file back
const summary = await importWorkbook({ config, workbook: "verbatra-translations.xlsx" });
```

Les deux prennent la config validée de [`loadConfig`](/docs/sdk), et `importWorkbook` renvoie le même `RunSummary` que `translate`. Le classeur lui-même est construit et analysé par `@verbatra/exchange`, un package interne qui détient un modèle de ligne neutre, agnostique vis-à-vis du format, et le `.xlsx` stylisé vers et depuis lequel il sérialise. Le SDK le compose pour toi : tu l'atteins via `exportWorkbook` / `importWorkbook`, jamais directement.

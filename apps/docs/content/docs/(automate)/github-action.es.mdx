title: GitHub Action

¿Quieres que las traducciones se ejecuten en cada push? verbatra incluye una GitHub Action compuesta que hace exactamente eso. Ejecuta `verbatra translate --json` en CI, anota cualquier fallo, escribe un resumen del job y sale con el código de salida de la CLI. La conectas con `uses:`. No se publica en npm.

## Disponibilidad

Esta action vive en el repositorio, así que la referencias directamente desde ahí con `uses: mariokreitz/verbatra/packages/github-action@<commit-sha>`, fijada a un SHA de commit. No hay paquete de npm, ni ficha en el GitHub Marketplace, ni etiqueta de action versionada. La referencia `uses:` basada en ruta y fijada a un SHA de commit es la única forma soportada de consumirla.

## Uso

```yaml
name: translate
on:
  push:
    branches: [main]

permissions:
  contents: read

jobs:
  verbatra:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<commit-sha>
      - uses: mariokreitz/verbatra/packages/github-action@<commit-sha>
        with:
          version: 0.3.0 # pin @verbatra/cli to an exact version
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

La action ejecuta la CLI, así que lee la clave de API del proveedor desde el entorno. Pásala desde los secrets de tu repositorio bajo la variable que espere tu proveedor (mira [Proveedores](/docs/providers)): `ANTHROPIC_API_KEY` en el ejemplo de arriba, `OPENAI_API_KEY` si ejecutas OpenAI, y así sucesivamente.

## Entradas

| Entrada | Obligatoria | Por defecto | Descripción |
| --- | --- | --- | --- |
| `version` | sí | - | la versión de `@verbatra/cli` a ejecutar, p. ej. `0.3.0`. Debe ser una versión semver exacta; el paso falla de inmediato con un error claro ante una etiqueta como `latest`, un rango, o un prefijo `^`/`~` |
| `config-path` | no | `""` | archivo de configuración explícito a cargar (mapea a `--config`); vacío usa la búsqueda de configuración normal |
| `working-directory` | no | `""` | directorio contra el que resolver la configuración y los archivos de locale (mapea a `--cwd`) |
| `dry-run` | no | `"false"` | ponlo a `"true"` para informar de qué cambiaría sin llamar a un proveedor ni escribir (mapea a `--dry-run`) |
| `node-version` | no | `"24"` | versión de Node.js a configurar para ejecutar la CLI |

La action siempre ejecuta `translate --json`. Nunca ejecuta `init` ni `watch`.

## Comportamiento de salida

La action captura el stdout y el código de salida de la CLI sin abortar antes de tiempo. Anota los fallos, escribe el resumen del job y solo entonces propaga el código de salida de la CLI. Así que el paso falla con `1` (algunos locales fallaron) o `2` (no se pudo ejecutar), pero no antes de que obtengas las anotaciones y el resumen. Los [códigos de salida de la CLI](/docs/cli#exit-codes) detallan lo que significa cada código.

## Persistir las traducciones

Sin `dry-run`, `translate` escribe los archivos de locale actualizados en el checkout del runner, pero la action se detiene ahí: no los confirma. Para conservar los cambios, añade tu propio paso que confirme y haga push, o que abra un pull request. Cuando solo quieras que CI señale las traducciones que faltan o están desactualizadas sin escribir nada, pon `dry-run: "true"`.

## Seguridad

Fija tanto la action como la CLI a referencias exactas. Fija `uses:` a un SHA de commit, y pon la entrada `version` a una release exacta de `@verbatra/cli` en lugar de a una etiqueta flotante como `latest`. Eso mantiene una ejecución reproducible e impide que arrastre automáticamente una release comprometida. Dale al workflow solo el privilegio que necesite: `contents: read` para una comprobación, más `contents: write` y `pull-requests: write` solo cuando un paso posterior confirma o abre un pull request. Las entradas llegan a la CLI como datos, nunca insertadas en el shell, así que un valor de entrada manipulado sigue siendo un argumento y nunca se convierte en código ejecutable.

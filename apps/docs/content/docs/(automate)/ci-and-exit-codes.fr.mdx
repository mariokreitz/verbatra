---
title: CI et codes de sortie
description: Lance verbatra en CI, lis ses codes de sortie et consomme sa sortie JSON.
---
verbatra est une dépendance de développement, donc les commandes que tu lances localement tournent de la même façon en CI. Installe-le, passe la clé du provider depuis ton magasin de secrets CI sous la forme de la variable d'environnement correspondante, et lance `verbatra translate`.

```bash
pnpm add -D @verbatra/cli
# ANTHROPIC_API_KEY is provided by the CI secret store as an environment variable
verbatra translate --json
```

Tu veux faire échouer un pipeline quand les traductions ont dérivé, sans appeler de provider ni même avoir besoin d'une clé ? Lance [`verbatra check`](/docs/cli/check) : il sort avec `1` quand une locale est désynchronisée.

```bash
# read-only gate: exit 1 if any locale is missing or stale
verbatra check
```

Tu as besoin des clés exactes derrière cette dérive (par exemple pour les publier dans un commentaire de pull request) ? Lance [`verbatra diff`](/docs/cli/diff) : même garde-fou en lecture seule, mais au lieu de seulement compter, il liste les clés qui seraient ajoutées, retraduites ou sont orphelines.

```bash
# read-only gate with the key lists: exit 1 if any locale has pending changes
verbatra diff
```

## Codes de sortie

Le code de sortie est le contrat sur lequel ton étape de CI se branche :

| Code | Signification |
| --- | --- |
| `0` | succès (pour `check`, chaque locale est synchronisée ; pour `diff`, aucun changement en attente) |
| `1` | `translate` ou `import` s'est terminé mais certaines locales ont échoué, `check` a trouvé une locale désynchronisée, ou `diff` a trouvé des changements en attente |
| `2` | impossible de lancer (une erreur sur toute l'exécution, ou une erreur d'utilisation, y compris une valeur `--locales` inconnue ou vide) |
| `130` | `watch` ou `ui` a été arrêté de force par une seconde interruption |

`watch` est l'exception : un échec par exécution apparaît comme un enregistrement de flux, pas comme un code de sortie, et une seule interruption arrête `watch` proprement et sort avec `0`.

## Sortie JSON

Chaque commande (`translate`, `watch`, `check`, `diff`, `export` et `import`) accepte `--json` pour une sortie lisible par machine.

`verbatra translate --json` affiche le résumé de l'exécution sous forme d'un seul objet JSON ayant cette forme :

```ts
interface RunSummary {
  dryRun: boolean;          // whether this was a dry run (no provider calls, no writes)
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
}
```

`verbatra watch --json` affiche un enregistrement par exécution au format NDJSON (un objet JSON par ligne). Chaque enregistrement est un succès ou un échec :

```ts
type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };
```

Une exécution échouée pendant `watch` arrive comme l'un de ces enregistrements sur le flux. Elle n'arrête pas le surveillant et ne définit pas de code de sortie non nul.

`verbatra check --json` affiche un document de statut en lecture seule sous forme d'un seul objet JSON. Le `inSync` de niveau supérieur est vrai exactement quand chaque locale est synchronisée, ce qui est exactement quand la commande sort avec `0` :

```ts
interface CheckSummary {
  inSync: boolean;             // true exactly when the command exits 0
  locales: LocaleCheckSummary[];
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;             // in source, absent from target
  stale: number;               // source changed since last translated
  upToDate: number;            // target matches the current source
  inSync: boolean;             // missing === 0 && stale === 0
}
```

`verbatra diff --json` exécute le même calcul en lecture seule mais te donne des listes de clés au lieu de décomptes. Le `hasPendingChanges` de niveau supérieur est vrai exactement quand au moins une locale a une clé à ajouter ou à retraduire, ce qui est exactement quand la commande sort avec `1`. Les clés orphelines apparaissent dans les listes mais ne définissent jamais `hasPendingChanges` :

```ts
interface DiffSummary {
  hasPendingChanges: boolean;  // true exactly when the command exits 1
  locales: LocaleDiff[];
}

interface LocaleDiff {
  locale: string;
  missing: string[];           // in source, absent from target: would be added
  changed: string[];           // source changed since last translated: would be re-translated
  orphaned: string[];          // in target, absent from source: reported only
  hasPendingChanges: boolean;  // missing.length > 0 || changed.length > 0
}
```

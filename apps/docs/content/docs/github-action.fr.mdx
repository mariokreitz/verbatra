# GitHub Action

verbatra fournit une GitHub Action composite qui exécute `verbatra translate --json` en CI, annote tout échec, écrit un résumé de tâche, puis sort avec le code de sortie de la CLI. Elle est consommée directement depuis le dépôt avec `uses:` ; elle n'est pas publiée sur npm.

## Disponibilité

Cette action n'existe que dans le dépôt. Tu la référence directement depuis le dépôt avec `uses: mariokreitz/verbatra/packages/github-action@<commit-sha>`, épinglée à un SHA de commit. Elle n'est pas publiée sur npm et n'a ni fiche GitHub Marketplace ni étiquette d'action versionnée ; la seule manière prise en charge de la consommer est la référence `uses:` basée sur le chemin et épinglée à un SHA de commit.

## Utilisation

```yaml
name: translate
on:
  push:
    branches: [main]

permissions:
  contents: read

jobs:
  verbatra:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<commit-sha>
      - uses: mariokreitz/verbatra/packages/github-action@<commit-sha>
        with:
          version: 0.3.0 # pin @verbatra/cli to an exact version
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

L'action exécute la CLI, donc la clé API du fournisseur est lue depuis l'environnement. Fournis-la depuis les secrets de ton dépôt sous la forme de la variable correspondant à ton fournisseur (voir [Fournisseurs](/docs/providers)) - par exemple `ANTHROPIC_API_KEY` ci-dessus.

## Entrées

| Entrée | Requis | Par défaut | Description |
| --- | --- | --- | --- |
| `version` | oui | - | la version de `@verbatra/cli` à exécuter, par exemple `0.3.0`. Épingle-la à une version exacte ; n'utilise pas une étiquette flottante telle que `latest` |
| `config-path` | non | `""` | fichier de configuration explicite à charger (mappé sur `--config`) ; vide utilise la recherche de configuration normale |
| `working-directory` | non | `""` | répertoire par rapport auquel résoudre la configuration et les fichiers de locale (mappé sur `--cwd`) |
| `dry-run` | non | `"false"` | définis sur `"true"` pour signaler ce qui changerait sans appeler de fournisseur ni écrire (mappé sur `--dry-run`) |
| `node-version` | non | `"24"` | version de Node.js à configurer pour exécuter la CLI |

L'action exécute toujours `translate --json` ; elle n'exécute ni `init` ni `watch`.

## Comportement de sortie

L'action capture la sortie standard et le code de sortie de la CLI sans échouer prématurément, annote les échecs et écrit le résumé de tâche, puis propage le code de sortie de la CLI - de sorte que l'étape échoue sur `1` (certaines locales ont échoué) ou `2` (impossible de s'exécuter) seulement après que les annotations et le résumé sont produits. Les [codes de sortie de la CLI](/docs/cli#exit-codes) définissent la signification de chaque code.

## Persister les traductions

Sans `dry-run`, `translate` écrit les fichiers de locale mis à jour dans le checkout du runner, mais l'action ne les commite pas. Pour conserver les changements, ajoute ta propre étape pour commiter et pousser, ou ouvre une pull request. Utilise `dry-run: "true"` lorsque tu veux seulement que la CI vérifie les traductions manquantes ou obsolètes.

## Sécurité

Épingle à la fois l'action et la CLI à des références exactes : épingle `uses:` à un SHA de commit, et épingle l'entrée `version` à une release exacte de `@verbatra/cli` plutôt qu'à une étiquette flottante, afin qu'une exécution soit reproductible et ne puisse pas récupérer automatiquement une release compromise. Donne au workflow le moindre privilège dont il a besoin - `contents: read` pour une vérification ; ajoute `contents: write` et `pull-requests: write` seulement si une étape ultérieure commite ou ouvre une pull request. Les entrées sont passées à la CLI comme données, jamais insérées dans le shell, de sorte qu'une valeur d'entrée forgée reste un argument plutôt que du code exécutable.

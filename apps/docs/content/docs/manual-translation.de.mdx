# Manuelle Übersetzung

Nicht jeder String gehört zu einem Provider. Marketing-Texte, juristische Texte und Produktnamen wollen meist einen menschlichen Übersetzer, und manche Teams halten für die wichtigsten Sprachen einen Profi im Loop. verbatra löst das mit einem Arbeitsmappen-Round-Trip: **exportiere** die zu übersetzenden Strings in eine gestylte Excel-Datei, gib sie einem Übersetzer und **importiere** dann die ausgefüllte Datei zurück in deine Locale-Dateien.

Derselbe Diff treibt beide Richtungen. Der Export wählt genau die Keys aus, die ein automatisierter Lauf übersetzen würde (die neuen und geänderten), sodass eine manuell übersetzte Locale genauso mit deiner Quelle synchron bleibt wie eine KI-übersetzte. Der Import führt dieselben Platzhalter- und ICU-Prüfungen aus wie `translate`, sodass verbatra einen Wert zurückhält und dir Bescheid gibt, statt ihn zu schreiben, falls ein Übersetzer einen Wert von Hand eintippt, der einen Platzhalter zerstört.

## Der Round-Trip

```bash
# 1. Export the strings that need translating
verbatra export

# 2. A translator fills the Translation column and sends the file back

# 3. Import the filled workbook
verbatra import verbatra-translations.xlsx
```

Standardmäßig legt der Export `verbatra-translations.xlsx` im Arbeitsverzeichnis ab. Der Import liest sie zurück, validiert jeden Wert, schreibt die bestandenen in deine Locale-Dateien und aktualisiert die [lock file](/docs/the-lock-file) für die akzeptierten Keys, genau wie es ein `translate`-Lauf täte.

## Die Arbeitsmappe

Die Datei beginnt mit einem **Instructions**-Blatt, gefolgt von einem Datenblatt pro Ziel-Locale, jeweils nach seiner Locale benannt (`de`, `fr`, ...). Jedes Datenblatt teilt sich dieselben Spalten:

| Spalte | Editierbar | Zweck |
| --- | --- | --- |
| `Key` | nein | Der gepunktete Key-Pfad. Die einzige Identität, die eine Zeile auf einen String zurückabbildet. |
| `Source` | nein | Der Wert der Quell-Locale, zur Referenz. |
| `Current translation` | nein | Der vorhandene Zielwert, falls vorhanden. |
| `Status` | nein | `new` (noch keine Übersetzung) oder `changed` (die Quelle hat sich geändert und die Übersetzung muss aktualisiert werden). |
| `Translation` | **ja** | Die einzige Zelle, die der Übersetzer ausfüllt. |
| `Source hash` | nein | Ausgeblendet. Der beim Export erfasste Quellinhalt, genutzt um eine Quelle zu erkennen, die sich nach dem Export geändert hat. |

Jede Spalte außer `Translation` ist gesperrt und das Blatt ist geschützt, sodass ein Übersetzer nur die eine Spalte bearbeiten kann, die zählt. Die Kopfzeile ist fixiert und die Spalte `Source hash` ist ausgeblendet. Sortiere oder filtere die Zeilen, wie du möchtest: die Zuordnung erfolgt über `Key`, niemals über die Position.

Eine leere `Translation`-Zelle bedeutet "noch nicht übersetzt". Der Import überspringt sie und schreibt niemals einen leeren String, sodass ein Übersetzer eine halb ausgefüllte Arbeitsmappe zurücksenden und den Rest später fertigstellen kann.

## Auswählen, was exportiert wird

Standardmäßig schließt `export` nur die fehlenden und geänderten Strings ein, die Arbeit, die tatsächlich zu tun ist. Zwei Flags stellen die Auswahl ein:

```bash
# Only the German and French sheets
verbatra export --locales de,fr

# Also include strings that are already up to date
verbatra export --include-unchanged

# Write to a specific path
verbatra export --out handoff/round-2.xlsx
```

Greif zu `--include-unchanged`, wenn ein Übersetzer den vollen Kontext einer Locale will, nicht nur das Delta. Siehe die [`verbatra export`](/docs/cli/export)-Referenz für jedes Flag.

## Sicherheit beim Import

Der Import ist kein blindes Einfügen. Bevor verbatra irgendetwas schreibt, läuft jeder vom Übersetzer eingetippte Wert durch dieselben Prüfungen wie eine automatisierte Übersetzung:

- **Platzhalter-Integrität**: eine Übersetzung muss exakt dieselben Platzhalter tragen wie ihre Quelle. Lass `{name}` weg oder erfinde `{total}`, und der Wert wird zurückgehalten.
- **ICU-Gültigkeit**: eine ICU-Nachricht muss strukturell gültig bleiben, mit denselben Argumentnamen. Nur der menschenlesbare Text darf sich ändern.
- **Quell-Drift**: der ausgeblendete `Source hash` lässt den Import einen Quell-String erkennen, der sich nach dem Export der Arbeitsmappe geändert hat. Eine veraltete Zeile wird zurückgehalten, sodass du nie eine aktuelle Quelle mit der Übersetzung einer alten überschreibst.
- **Unbekannte Keys**: eine Zeile, deren `Key` nicht in der aktuellen Quelle ist (etwa einer, den ein Übersetzer von Hand hinzugefügt hat), wird abgelehnt.

Ein Wert, der irgendeine Prüfung nicht besteht, wird **zurückgehalten** (nicht geschrieben) und in der Lauf-Zusammenfassung gemeldet, sodass du ihn korrigieren und erneut importieren kannst. Keys, die bestehen, werden geschrieben und festgeschrieben; die, die scheitern, lassen die Locale-Datei unberührt. Füge `--dry-run` hinzu, um eine zurückgegebene Arbeitsmappe zu validieren und eine Vorschau dessen zu sehen, was geschrieben würde, ohne eine einzige Datei anzurühren:

```bash
verbatra import verbatra-translations.xlsx --dry-run
```

Der Import teilt den Exit-Code-Vertrag von `translate`: er gibt `0` zurück, wenn jede Locale erfolgreich ist, `1`, wenn eine Locale fehlschlägt, und `2`, wenn der Lauf nicht starten konnte. Das heißt, eine manuelle Übergabe fällt in dieselben CI-Prüfungen wie eine automatisierte. Siehe [CI and automation](/docs/ci-and-exit-codes).

## Aus dem SDK

Die CLI-Befehle umschließen zwei SDK-Funktionen, `exportWorkbook` und `importWorkbook`. Rufe sie direkt auf, wenn du manuelle Übersetzung in dein eigenes Tooling einbaust:

```ts
import { exportWorkbook, importWorkbook, loadConfig } from "@verbatra/sdk";

const config = await loadConfig();

// Write a workbook of the strings that need translating
await exportWorkbook({ config });

// ...later, import the filled file back
const summary = await importWorkbook({ config, workbook: "verbatra-translations.xlsx" });
```

Beide nehmen die validierte Konfiguration von [`loadConfig`](/docs/sdk), und `importWorkbook` gibt dieselbe `RunSummary` zurück wie `translate`. Die Arbeitsmappe selbst wird von `@verbatra/exchange` gebaut und geparst, einem internen Paket, das ein neutrales, formatunabhängiges Zeilenmodell und die gestylte `.xlsx` besitzt, in die und aus der es serialisiert. Das SDK komponiert es für dich: du erreichst es über `exportWorkbook` / `importWorkbook`, niemals direkt.

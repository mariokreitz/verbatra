# Manuelle Übersetzung

Nicht jeder String sollte durch einen Provider laufen. Marketing-Texte, juristische Texte und Produktnamen benötigen oft einen menschlichen Übersetzer, und manche Teams behalten für die wichtigsten Sprachen einen Profi in der Schleife. verbatra unterstützt dies mit einem Arbeitsmappen-Hin-und-Zurück: **exportiere** die Strings, die übersetzt werden müssen, in eine gestaltete Excel-Datei, übergib sie einem Übersetzer und **importiere** die ausgefüllte Datei zurück in deine Locale-Dateien.

Derselbe Vergleich treibt beide Abläufe an. Der Export wählt genau die Schlüssel aus, die ein automatisierter Lauf übersetzen würde - die neuen und geänderten -, sodass ein manuelles Locale auf dieselbe Weise mit deiner Quelle synchron bleibt wie ein KI-übersetztes. Der Import führt dieselben Platzhalter- und ICU-Prüfungen aus wie `translate`, sodass ein von Hand getippter Wert, der einen Platzhalter bricht, zurückgehalten und gemeldet statt geschrieben wird.

## Das Hin-und-Zurück

```bash
# 1. Export the strings that need translating
verbatra export

# 2. A translator fills the Translation column and sends the file back

# 3. Import the filled workbook
verbatra import verbatra-translations.xlsx
```

Der Export schreibt `verbatra-translations.xlsx` standardmäßig in das Arbeitsverzeichnis. Der Import liest sie zurück, validiert jeden Wert, schreibt die bestandenen in deine Locale-Dateien und aktualisiert die [Lock-Datei](/docs/the-lock-file) für die akzeptierten Schlüssel - genau wie ein `translate`-Lauf es täte.

## Die Arbeitsmappe

Die Datei hat ein **Instructions**-Blatt, gefolgt von einem Datenblatt pro Ziel-Locale, jeweils nach seinem Locale benannt (`de`, `fr`, ...). Jedes Datenblatt hat dieselben Spalten:

| Spalte | Bearbeitbar | Zweck |
| --- | --- | --- |
| `Key` | nein | Der Punkt-getrennte Schlüsselpfad. Die alleinige Identität, die eine Zeile auf einen String zurückführt. |
| `Source` | nein | Der Wert des Quell-Locales, zur Referenz. |
| `Current translation` | nein | Der vorhandene Zielwert, falls vorhanden. |
| `Status` | nein | `new` (noch keine Übersetzung) oder `changed` (die Quelle hat sich geändert und die Übersetzung muss aktualisiert werden). |
| `Translation` | **ja** | Die einzige Zelle, die der Übersetzer ausfüllt. |
| `Source hash` | nein | Ausgeblendet. Der beim Export erfasste Quellinhalt, verwendet, um eine Quelle zu erkennen, die sich nach dem Export geändert hat. |

Jede Spalte außer `Translation` ist gesperrt, und das Blatt ist geschützt, sodass ein Übersetzer nur die eine Spalte bearbeiten kann, auf die es ankommt. Die Kopfzeile ist fixiert und die Spalte `Source hash` ist ausgeblendet. Zeilen können frei sortiert oder gefiltert werden - die Zuordnung erfolgt über `Key`, niemals über die Position.

Eine leere `Translation`-Zelle bedeutet "noch nicht übersetzt": sie wird beim Import übersprungen, niemals als leerer String geschrieben. Das ermöglicht es einem Übersetzer, eine teilweise ausgefüllte Arbeitsmappe zurückzugeben und den Rest später aufzunehmen.

## Auswählen, was exportiert wird

Standardmäßig schließt `export` nur die fehlenden und geänderten Strings ein - die Arbeit, die tatsächlich getan werden muss. Zwei Flags passen die Auswahl an:

```bash
# Only the German and French sheets
verbatra export --locales de,fr

# Also include strings that are already up to date
verbatra export --include-unchanged

# Write to a specific path
verbatra export --out handoff/round-2.xlsx
```

`--include-unchanged` ist nützlich, wenn ein Übersetzer den vollen Kontext eines Locales statt nur des Deltas möchte. Siehe die [`verbatra export`](/docs/cli/export)-Referenz für jedes Flag.

## Sicherheit beim Import

Der Import ist kein blindes Einfügen. Jeder Wert, den der Übersetzer getippt hat, durchläuft dieselben Prüfungen wie eine automatisierte Übersetzung, bevor er geschrieben wird:

- **Platzhalter-Integrität** - eine Übersetzung muss genau dieselben Platzhalter tragen wie ihre Quelle. Ein Wert, der `{name}` weglässt oder `{total}` erfindet, wird zurückgehalten.
- **ICU-Gültigkeit** - eine ICU-Nachricht muss strukturell gültig bleiben, mit denselben Argumentnamen. Nur der menschenlesbare Text darf sich ändern.
- **Quell-Abweichung** - der ausgeblendete `Source hash` lässt den Import einen Quell-String erkennen, der sich geändert hat, nachdem die Arbeitsmappe exportiert wurde. Eine veraltete Zeile wird zurückgehalten, sodass du niemals eine aktuelle Quelle mit einer Übersetzung einer alten überschreibst.
- **Unbekannte Schlüssel** - eine Zeile, deren `Key` nicht in der aktuellen Quelle ist (zum Beispiel eine, die ein Übersetzer von Hand hinzugefügt hat), wird abgelehnt.

Ein Wert, der eine Prüfung nicht besteht, wird **zurückgehalten** - nicht geschrieben - und in der Lauf-Zusammenfassung gemeldet, sodass du ihn beheben und erneut importieren kannst. Die Schlüssel, die bestehen, werden geschrieben und festgeschrieben; die, die nicht bestehen, lassen die Locale-Datei unberührt. Führe mit `--dry-run` aus, um eine zurückgegebene Arbeitsmappe zu validieren und zu sehen, was geschrieben würde, ohne eine Datei zu berühren:

```bash
verbatra import verbatra-translations.xlsx --dry-run
```

Da der Import den `translate`-Exit-Code-Vertrag teilt, gibt er `0` zurück, wenn jedes Locale erfolgreich ist, `1`, wenn ein Locale fehlschlägt, und `2`, wenn der Lauf nicht starten konnte - sodass eine manuelle Übergabe in dieselben CI-Prüfungen passt wie eine automatisierte. Siehe [CI und Automatisierung](/docs/ci-and-exit-codes).

## Aus dem SDK

Die CLI-Befehle umschließen zwei SDK-Funktionen, `exportWorkbook` und `importWorkbook`, die du direkt aufrufen kannst, wenn du manuelle Übersetzung in deine eigenen Werkzeuge einbaust:

```ts
import { exportWorkbook, importWorkbook, loadConfig } from "@verbatra/sdk";

const config = await loadConfig();

// Write a workbook of the strings that need translating
await exportWorkbook({ config });

// ...later, import the filled file back
const summary = await importWorkbook({ config, workbook: "verbatra-translations.xlsx" });
```

Beide nehmen die validierte Konfiguration von [`loadConfig`](/docs/sdk), und `importWorkbook` gibt dieselbe `RunSummary` zurück wie `translate`. Die Arbeitsmappe selbst wird von `@verbatra/exchange` erstellt und geparst, einem internen Paket, das ein neutrales, formatunabhängiges Zeilenmodell und die gestaltete `.xlsx` besitzt, in die und aus der es serialisiert. Das SDK setzt sie zusammen; du interagierst über `exportWorkbook` / `importWorkbook` mit ihr, nicht direkt.

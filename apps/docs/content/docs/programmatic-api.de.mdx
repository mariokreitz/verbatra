# Programmatische API

Die [SDK-Referenz](/docs/sdk) dokumentiert jede Funktion und jeden Typ. Diese Seite ist der aufgabenorientierte Begleiter: ein paar End-to-End-Muster für die Verwendung von verbatra im Code. Greife zum SDK, wenn du Übersetzung in einem Skript, einem Build-Schritt oder einem CI-Job möchtest, mit voller Kontrolle über alles, was darum herum geschieht.

## In einem Skript übersetzen

Lade die Konfiguration, führe eine Übersetzung aus und melde die Schlagzeile. Probleme des gesamten Laufs werfen einen `SdkError`; Ergebnisse pro Locale sind Daten in der Zusammenfassung.

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig();
const summary = await translate({ config });

console.log(`${summary.succeeded.length} ok, ${summary.failed.length} failed`);
```

## Einen Build- oder CI-Schritt am Ergebnis ausrichten

Die Zusammenfassung sagt dir, was sich geändert hat und was fehlgeschlagen ist, ohne Ausgabe zu parsen. Lass den Schritt fehlschlagen, wenn ein Locale fehlgeschlagen ist, und inspiziere das Detail pro Locale, um den Grund zu protokollieren.

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const summary = await translate({ config: await loadConfig() });

for (const locale of summary.locales) {
  if (locale.status === "failed") {
    console.error(`${locale.locale}: ${locale.error?.code} ${locale.error?.message}`);
  }
}

if (summary.failed.length > 0) {
  process.exitCode = 1;
}
```

## Prüfen ohne zu schreiben

`check` liest und vergleicht, ohne einen Provider aufzurufen oder Dateien zu schreiben, und meldet die Abweichung pro Locale aufgeteilt in fehlende, veraltete und aktuelle Anzahlen. Sein `inSync`-Flag ist genau dann wahr, wenn nichts fehlend oder veraltet ist, sodass es das natürliche CI-Gate ist.

```ts
import { check, loadConfig } from "@verbatra/sdk";

const summary = await check({ config: await loadConfig() });

if (!summary.inSync) {
  const drifted = summary.locales.filter((locale) => !locale.inSync).map((locale) => locale.locale);
  console.error(`Out of sync: ${drifted.join(", ")}. Run verbatra translate.`);
  process.exitCode = 1;
}
```

`translate({ dryRun: true })` ist eine Alternative, wenn du die vollständige Lauf-Zusammenfassung möchtest (die Schlüssel, die übersetzt würden), statt nur der Abweichungs-Anzahlen.

`diff` ist das schreibgeschützte Geschwister von `check`: Es führt dieselbe Berechnung aus, gibt aber die tatsächlichen Schlüssellisten (`missing`, `changed`, `orphaned`) pro Locale statt Anzahlen zurück, mit demselben `hasPendingChanges`-Gate. Greife dazu, wenn du wissen musst, welche Schlüssel, nicht nur wie viele.

```ts
import { diff, loadConfig } from "@verbatra/sdk";

const summary = await diff({ config: await loadConfig() });

if (summary.hasPendingChanges) {
  for (const locale of summary.locales) {
    if (locale.hasPendingChanges) {
      console.log(`${locale.locale}: add ${locale.missing.join(", ")}; re-translate ${locale.changed.join(", ")}`);
    }
  }
  process.exitCode = 1;
}
```

## Strings an einen menschlichen Übersetzer übergeben

Exportiere die Strings, die übersetzt werden müssen, in eine Excel-Arbeitsmappe und importiere dann die ausgefüllte Datei zurück. `importWorkbook` gibt dieselbe `RunSummary` wie `translate` zurück und führt dieselben Platzhalter- und ICU-Prüfungen aus, sodass eine manuelle Übergabe in dasselbe Reporting und Gating wie ein automatisierter Lauf passt. Siehe [Manuelle Übersetzung](/docs/manual-translation).

```ts
import { exportWorkbook, importWorkbook, loadConfig } from "@verbatra/sdk";

const config = await loadConfig();

// Write a workbook of the strings that need translating.
const exported = await exportWorkbook({ config });
console.log(`Wrote ${exported.path}`);

// ...later, after a translator fills and returns the file, import it back.
const summary = await importWorkbook({ config, workbook: exported.path });
console.log(`${summary.succeeded.length} ok, ${summary.failed.length} failed`);
```

## Neu übersetzen, während sich die Quelle ändert

`watch` ruft dein `onRun` einmal pro entprellter Änderung auf und gibt einen Controller zurück, den du sauber stoppst.

```ts
import { loadConfig, watch } from "@verbatra/sdk";

const controller = await watch({
  config: await loadConfig(),
  onRun: (result) => {
    if (result.status === "failed") {
      console.error(`run failed: ${result.error.code}`);
    }
  },
});

process.on("SIGINT", () => {
  void controller.stop();
});
```

## Eine Konfiguration ohne Datei

`loadConfig` akzeptiert ein In-Memory-`configOverride`, sodass du verbatra vollständig aus dem Code heraus steuern kannst, ohne eine `verbatra.config.ts` auf der Festplatte.

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig({
  configOverride: {
    sourceLocale: "en",
    targetLocales: ["de", "fr"],
    format: "i18next-json",
    files: { pattern: "locales/{locale}.json" },
    provider: { id: "anthropic", options: { model: "claude-sonnet-4-6", maxTokens: 4096 } },
  },
});

await translate({ config });
```

Für die vollständige Liste der Funktionen, die `RunSummary`- und `LocaleSummary`-Strukturen und jeden `SdkError`-Code siehe die [SDK-Referenz](/docs/sdk). Für den Exit-Code-Vertrag, wenn du stattdessen das CLI in CI ausführst, siehe [CI und Exit-Codes](/docs/ci-and-exit-codes).

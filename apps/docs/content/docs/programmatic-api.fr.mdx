# API programmatique

La [référence du SDK](/docs/sdk) couvre chaque fonction et chaque type. Cette page en est le compagnon pratique : une poignée de modèles de bout en bout pour brancher verbatra dans du code. Tourne-toi vers le SDK quand tu veux de la traduction à l'intérieur d'un script, d'une étape de build ou d'une tâche CI, avec un contrôle total sur tout ce qui l'entoure.

## Traduire dans un script

Charge la config, exécute une traduction, affiche l'essentiel. Les problèmes sur toute l'exécution lèvent une `SdkError` ; les résultats par locale sont des données sur le résumé.

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig();
const summary = await translate({ config });

console.log(`${summary.succeeded.length} ok, ${summary.failed.length} failed`);
```

## Conditionner une étape de build ou de CI au résultat

Le résumé t'indique ce qui a changé et ce qui a échoué, sans analyse de sortie requise. Fais échouer l'étape quand une locale a échoué, et inspecte le détail par locale pour journaliser pourquoi.

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const summary = await translate({ config: await loadConfig() });

for (const locale of summary.locales) {
  if (locale.status === "failed") {
    console.error(`${locale.locale}: ${locale.error?.code} ${locale.error?.message}`);
  }
}

if (summary.failed.length > 0) {
  process.exitCode = 1;
}
```

## Vérifier sans écrire

`check` lit et diffuse sans appeler de provider ni écrire de fichiers, et signale la dérive par locale répartie en décomptes de clés manquantes, obsolètes et à jour. Son flag `inSync` est vrai exactement quand rien n'est manquant ni obsolète, ce qui en fait ton garde-fou CI naturel.

```ts
import { check, loadConfig } from "@verbatra/sdk";

const summary = await check({ config: await loadConfig() });

if (!summary.inSync) {
  const drifted = summary.locales.filter((locale) => !locale.inSync).map((locale) => locale.locale);
  console.error(`Out of sync: ${drifted.join(", ")}. Run verbatra translate.`);
  process.exitCode = 1;
}
```

Tu veux le résumé d'exécution complet (les clés qui seraient traduites) plutôt que seulement les décomptes de dérive ? Utilise `translate({ dryRun: true })`.

`diff` est la fonction sœur en lecture seule de `check` : même calcul, mais elle renvoie les listes de clés réelles (`missing`, `changed`, `orphaned`) par locale au lieu de décomptes, avec le même garde-fou `hasPendingChanges`. Tourne-toi vers elle quand tu as besoin de savoir quelles clés, et pas seulement combien.

```ts
import { diff, loadConfig } from "@verbatra/sdk";

const summary = await diff({ config: await loadConfig() });

if (summary.hasPendingChanges) {
  for (const locale of summary.locales) {
    if (locale.hasPendingChanges) {
      console.log(`${locale.locale}: add ${locale.missing.join(", ")}; re-translate ${locale.changed.join(", ")}`);
    }
  }
  process.exitCode = 1;
}
```

## Confier des chaînes à un traducteur humain

Exporte les chaînes qui ont besoin d'être traduites dans un classeur Excel, puis réimporte le fichier rempli. `importWorkbook` renvoie le même `RunSummary` que `translate` et exécute les mêmes vérifications de placeholders et d'ICU, de sorte qu'un transfert manuel s'insère exactement dans le même reporting et les mêmes garde-fous que tu utilises pour une exécution automatisée. Voir [Traduction manuelle](/docs/manual-translation).

```ts
import { exportWorkbook, importWorkbook, loadConfig } from "@verbatra/sdk";

const config = await loadConfig();

// Write a workbook of the strings that need translating.
const exported = await exportWorkbook({ config });
console.log(`Wrote ${exported.path}`);

// ...later, after a translator fills and returns the file, import it back.
const summary = await importWorkbook({ config, workbook: exported.path });
console.log(`${summary.succeeded.length} ok, ${summary.failed.length} failed`);
```

## Retraduire au fur et à mesure que la source change

`watch` déclenche ton `onRun` une fois par changement débondi et renvoie un contrôleur que tu arrêtes proprement.

```ts
import { loadConfig, watch } from "@verbatra/sdk";

const controller = await watch({
  config: await loadConfig(),
  onRun: (result) => {
    if (result.status === "failed") {
      console.error(`run failed: ${result.error.code}`);
    }
  },
});

process.on("SIGINT", () => {
  void controller.stop();
});
```

## Une config sans fichier

`loadConfig` accepte un `configOverride` en mémoire, de sorte que tu peux piloter verbatra entièrement depuis le code sans aucun `verbatra.config.ts` sur le disque.

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig({
  configOverride: {
    sourceLocale: "en",
    targetLocales: ["de", "fr"],
    format: "i18next-json",
    files: { pattern: "locales/{locale}.json" },
    provider: { id: "anthropic", options: { model: "claude-sonnet-4-6", maxTokens: 4096 } },
  },
});

await translate({ config });
```

Pour la liste complète des fonctions, les formes `RunSummary` et `LocaleSummary`, et chaque code `SdkError`, voir la [référence du SDK](/docs/sdk). Tu lances plutôt la CLI en CI ? Voir le contrat des codes de sortie dans [CI et codes de sortie](/docs/ci-and-exit-codes).

---
title: Translation safety
description: The integrity checks and security boundaries that decide what verbatra writes and what it withholds.
---
verbatra doesn't trust machine output by default. It runs every translation through integrity checks
before it writes, and it keeps untrusted text away from the instructions it sends the provider. This
page covers what those checks are and what they protect.

## Placeholder integrity

A translated value has to carry the same placeholders as its source. After translation, verbatra
compares the placeholder set of each result against the source value. Drop, add, or alter a
placeholder and the translation fails the check and gets withheld instead of written. The withheld key
keeps its prior lock hash, so the next run retries it, and it shows up per locale as an integrity
mismatch on the run summary.

What counts as a placeholder depends on the format: `{{double-brace}}` tokens for i18next,
ngx-translate, and YAML, single-brace `{name}` and `{0}` for vue-i18n, ICU argument and tag names for
next-intl and ARB, and inline placeholder ids for XLIFF. For i18next, `$t(...)` nesting references are
guarded too: drop, alter, or translate a `$t(...)` reference and the translation fails the check and
gets withheld. So if the source is `Welcome, {{name}}` and the model returns `Bienvenido` with the
`{{name}}` gone, verbatra holds it back rather than shipping a broken string. See
[Formats](/docs/formats) for the syntax of each.

For next-intl and ARB, the comparison is branch-aware: a plural or select argument is checked category
by category against its matching source branch, instead of flattening the whole message into one set
first. A placeholder fabricated inside a single branch is caught even if that same placeholder
legitimately exists in another branch of the source.

## ICU validity

For the ICU formats, next-intl and ARB, verbatra parses each source value. A source key whose value
isn't valid ICU gets set aside and skipped for translation instead of being sent in a broken state,
and it's reported as invalid ICU on the summary. The ICU body stays verbatim through the pipeline:
nothing is resolved or rewritten.

The same parser also checks the other side of the run: once a translation comes back, verbatra
validates the candidate value itself against the format's message syntax, not just its placeholder
set. A candidate that keeps every placeholder intact but breaks the ICU plural or select structure
still fails the check and gets withheld exactly like a placeholder mismatch, reported the same way on
the run summary. This applies to `translate`, `watch`, workbook import, and Studio's [retranslate
action](/docs/cli/studio).

## Review flags

Passing the checks above only proves a translation is structurally safe: nothing is broken, but it might still be wrong. verbatra runs a second, non-blocking layer on top of that: a pure heuristic that flags a key for human review without ever withholding it. A review flag is advisory only. It never withholds a key, and nothing on the workbook or the run summary can be edited to clear it; the only way to clear a flag is to fix the underlying translation.

Five reason codes, and a flagged key can carry more than one:

- `LENGTH_RATIO_OUTLIER`: the translated value's trimmed length is far shorter or longer than the source's (below 0.35x or above 3.0x, skipped when the trimmed source is under 12 UTF-16 code units). This is a raw length-ratio check on UTF-16 length, so it can over-flag a legitimately verbose target language (German) or a dense CJK target; treat it as a nudge to look, not a verdict.
- `EQUALS_SOURCE`: the translated value equals the source value verbatim, in a target locale that differs from the source locale, and the source actually had something translatable (at least one Unicode letter). A placeholder-only, numeric-only, or punctuation-only source is never flagged this way.
- `GLOSSARY_TERM_MISSED`: a configured glossary source term occurs in the source value but its mapped target term does not occur in the translation. Only evaluated when a glossary is configured.
- `INTEGRITY_REORDERED`: the placeholder set matched exactly (the placeholder-integrity check above already accepted the translation) but landed in a different order than the source. This flag never overlaps with a placeholder-integrity failure: it only fires on a translation that already passed.
- `PROVIDER_DEGRADED`: the batch this key came from carried a `FORMALITY_DOWNGRADED` or `GLOSSARY_IGNORED` notice. In today's provider set this is DeepL-only, since no LLM provider currently emits either notice.

Where you see this: `translate` and `watch` report it per locale as `needsReview` on the run summary, restricted to keys actually accepted and written this run, and the CLI prints a `needs-review` count when it is above zero; see [`verbatra translate`](/docs/cli/translate) and the [SDK reference](/docs/sdk). The Excel workbook carries the same signal as its `Review status` and `Review reasons` columns, recomputed fresh at every export; see [Human translation](/docs/manual-translation). [Verbatra Studio](/docs/cli/studio)'s needs-review queue lists the same flagged keys, read live from the persisted snapshot the last `translate`/`watch` run left behind.

`PROVIDER_DEGRADED` is translate-time only. It comes from a provider notice that exists only in memory during the run that produced it: nothing on disk records it, since fixing that would mean persisting provider-time facts in the [lock file](/docs/the-lock-file), which this feature deliberately does not do. So a workbook exported later from the same on-disk target file never shows `PROVIDER_DEGRADED` for that key, even though the key itself is still there: it exports as `ok` unless the key also independently trips one of the four recomputable reason codes above, in which case the exported row carries that reason instead.

## The prompt-injection boundary

The strings you translate aren't trustworthy: they can contain text that looks like an instruction
(think a UI string that literally reads "ignore your instructions"). verbatra treats them that way.
The system rules it sends a provider are compile-time constants. All variable input, the entries to
translate along with your glossary and tone, travels only in the user-turn data payload, never spliced
into the instruction channel. Provider output is constrained to a schema and validated, and the
placeholder check runs on whatever comes back. A string that tries to redirect the model can't reach
the rules, and a result that breaks a placeholder gets withheld.

## Secrets never leak

Providers read API keys only from the environment, never from your config or arguments. When a
provider SDK call fails, its raw error (which can carry request headers or a key) is discarded, never
bound, logged, or re-thrown. In its place verbatra raises its own structured error with a static,
secret-free message. The keys are secret-free by construction: nothing sensitive ever reaches a string
that would need scrubbing. A key can't show up in logs, in error messages, or in output.

## What you see

Every one of these outcomes is data on the run summary, per locale: the keys translated, the ones
withheld for a placeholder mismatch, the source keys skipped for invalid ICU, the keys flagged for
review, and any notices. Notices
include provider notices and SDK notices, such as a plural-category warning (code
`PLURAL_CATEGORIES_INCOMPLETE`) when a target language needs more plural categories than the source
supplies; see [Add a language](/docs/add-a-language) for how to synthesize those forms instead.
Nothing is hidden, and one locale's failure doesn't stop the others. See
[How it works](/docs/how-it-works) for where each check sits in the run.

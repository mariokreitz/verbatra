# Your first translation

The CLI is installed, the config is scaffolded, and your provider key is in the environment (if not, start with the [quick start](/docs)). Time to translate.

## Run it

One command translates every target locale and exits:

```bash
verbatra translate
```

Here is what happens under the hood: verbatra reads your source locale, diffs it against what it translated last time, and sends only the new and changed strings to your provider. Anything still current is left alone. Each target file is written in place, so `locales/de.json` and friends appear (or update) right next to your source file.

## Read the run summary

Every run ends with a summary. You get the headline (which locales succeeded, which failed) and a per-locale breakdown: keys translated, keys left unchanged, orphaned keys (present in a target but gone from the source), and any provider notices. One locale failing does not stop the rest; it shows up on that locale's line with a code and a message.

Want it machine-readable? Add `--json` and verbatra prints the whole summary as one JSON object:

```bash
verbatra translate --json
```

See [CI and exit codes](/docs/ci-and-exit-codes) for the JSON shape and the exit-code contract.

## Preview before you spend a token

To see exactly what a run would do without calling a provider or writing a single file, do a dry run:

```bash
verbatra translate --dry-run
```

You get the same summary, listing the keys that would be translated, but nothing is sent and nothing is written. It is the safe way to check a run before it costs you anything, which is especially handy on a free-tier key.

## Re-runs only touch what changed

This is the part that makes verbatra cheap to run often. Run `verbatra translate` again without editing your source and nothing goes to the provider: every key is already current. Change one source string and only that one key gets re-translated next run.

The trick is a committed lock file (`verbatra.lock.json`) that records what was translated. Each run diffs against it, so the provider is called only for what actually changed. See [How it works](/docs/how-it-works) for the full pipeline.

<Callout title="Commit the lock file">
verbatra writes `verbatra.lock.json` on the first run. Commit it alongside your locale files so every machine, and your CI, diffs against the same baseline. See [The lock file](/docs/the-lock-file).
</Callout>

## Translate while you work

Do not want to re-run by hand? Watch the source and let verbatra re-translate on every change:

```bash
verbatra watch
```

It keeps running until you stop it with Ctrl-C.

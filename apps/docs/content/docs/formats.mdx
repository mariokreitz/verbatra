# Formats

verbatra works on locale files through a format adapter. Every adapter reads a file into one format-neutral representation that the diff, hashing, and integrity checks operate on, and writes it back in the file's original shape. You select the format with the `format` field in your config. Four JSON formats shipped in v1, and three more (ARB, YAML, and XLIFF) are now available alongside them.

## The four JSON formats

| `format` | For | Placeholders | Plurals | ICU |
| --- | --- | --- | --- | --- |
| `i18next-json` | i18next | `{{name}}` and `$t(...)` nesting references | CLDR plural suffix on the key | no |
| `vue-i18n-json` | vue-i18n | `{name}`, `{0}` | pipe in the value | no |
| `next-intl-json` | next-intl | ICU argument and tag names | ICU plural or selectordinal | yes |
| `ngx-translate-json` | ngx-translate | `{{name}}` | none | no |

i18next and ngx-translate share the same `{{double-brace}}` interpolation. i18next additionally extracts and guards its `$t()` nesting references (which splice another key's content into the value), for example `$t(common.foo)` and `$t(common.foo, { options })`, as placeholders, so a translation that drops or alters one fails the integrity check. Two limitations apply: nested parentheses inside the `$t()` options are not supported, and only the default `$t(` prefix is recognized. ngx-translate uses the same double-brace interpolation but has no `$t()` nesting. vue-i18n uses single braces and decides plural from a pipe in the value. next-intl values are ICU MessageFormat: the placeholders are the ICU argument and rich-text tag names, plural follows an ICU plural or selectordinal argument, and the ICU body is kept verbatim through the pipeline. ngx-translate has no built-in plural or ICU, and its files may be flat with dotted keys or nested; verbatra preserves whichever style the file uses on write.

## ARB, YAML, and XLIFF

Three more formats are now available beyond the JSON family. They use the same adapter pipeline, the same format-neutral representation, and the same placeholder and ICU integrity checks. Select each with its `format` tag.

| `format` | For | Extensions | Placeholders | Notable behavior |
| --- | --- | --- | --- | --- |
| `arb` | Flutter | `.arb` | ICU | `@`-prefixed metadata is preserved and never translated |
| `yaml` | YAML i18n | `.yml`, `.yaml` | `{{name}}` | nested tree, i18next-style interpolation |
| `xliff` | XLIFF (XML) | `.xlf`, `.xliff` | inline placeholder ids | updates existing `<target>` elements in place; the target file must already exist |

ARB (Flutter Application Resource Bundle) is JSON-based, with `@`-prefixed metadata keys (`@key` per-message metadata and `@@`-prefixed globals such as `@@locale`) sitting alongside the message keys. verbatra preserves that metadata and writes it back unchanged: it is never sent for translation. ARB message values are ICU MessageFormat, so the placeholders, plural handling, and ICU validation work exactly as they do for next-intl.

YAML is a nested tree in YAML syntax, the same shape as a nested JSON file. The `yaml` format assumes i18next-compatible `{{double-brace}}` interpolation. A round-trip preserves the tree and its values; YAML comments are not carried across a write, the same way JSON has no comment concept.

XLIFF is XML, a flat list of trans-units rather than a nested tree. verbatra reads both XLIFF 1.2 (`file/body/trans-unit`) and XLIFF 2.0 (`file/unit/segment`), keying each entry by its trans-unit `id` and taking the value from `<target>` when present, otherwise `<source>`. On write it updates each `<target>` in place and leaves `<source>`, attributes, and `<note>` elements untouched, so the document round-trips. Because a single flat key/value map cannot reconstruct an XLIFF document, the destination file must already exist: verbatra updates targets in a pre-seeded target file (the standard XLIFF workflow) and does not create a missing target file. Inline placeholder elements (`x`, `g`, `bx`, `ex`, `ph`, `it`, `mrk`) are extracted by id and guarded across translation.

## Why the format matters

The format tells verbatra two things it needs for a safe run. First, the placeholder syntax, so the [placeholder integrity check](/docs/translation-safety) knows what to compare before and after translation. Second, for ICU, which values to validate, so an invalid ICU source key is skipped rather than sent in a broken state. The wrong format compares the wrong tokens, so match it to your i18n library.

## How files are read and written

verbatra caps input size and nesting depth on read, and resists a file changing underneath it. It writes atomically: it writes to a temporary file, then renames it into place, so an interrupted write never leaves a half-finished locale file. When verbatra cannot handle a file, it raises a structured, secret-free error rather than a raw one, for example on invalid JSON, a structure too deep, or, for ngx-translate only, a file that mixes flat dotted keys with nested objects.

## Dotted keys

For i18next, vue-i18n, and next-intl, a literal dotted leaf (a key such as `"foo.bar"` used as a single leaf) round-trips losslessly: verbatra reads it and writes it back with its on-disk shape preserved, not re-nested into `foo` then `bar`. Real nested paths stay nested. The one case that still fails by design is a genuine collision, where one file expresses the same effective path both as a literal dotted leaf and as a real nested path (for example `"foo.bar"` alongside `"foo": { "bar": ... }`); that run errors with `INVALID_STRUCTURE` rather than guessing or corrupting data. ngx-translate is unchanged: a dotted key there denotes a nested path, so its existing flat-vs-nested handling applies and a file that mixes flat dotted keys with nested objects still errors.

## Choosing a format

Match the format to your framework: `i18next-json` for i18next, `vue-i18n-json` for vue-i18n, `next-intl-json` for next-intl, and `ngx-translate-json` for ngx-translate. For non-JSON sources, use `arb` for Flutter, `yaml` for YAML-based i18n, and `xliff` for XLIFF files. See [Config file](/docs/config-file) for where `format` sits in the config.

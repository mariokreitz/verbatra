# Formats

verbatra works on JSON locale files through a format adapter. Every adapter reads a file into one format-neutral representation that the diff, hashing, and integrity checks operate on, and writes it back in the file's original shape. v1 supports four JSON formats, selected with the `format` field in your config.

## The four formats

| `format` | For | Placeholders | Plurals | ICU |
| --- | --- | --- | --- | --- |
| `i18next-json` | i18next | `{{name}}` | CLDR plural suffix on the key | no |
| `vue-i18n-json` | vue-i18n | `{name}`, `{0}` | pipe in the value | no |
| `next-intl-json` | next-intl | ICU argument and tag names | ICU plural or selectordinal | yes |
| `ngx-translate-json` | ngx-translate | `{{name}}` | none | no |

i18next and ngx-translate share the same `{{double-brace}}` interpolation. vue-i18n uses single braces and decides plural from a pipe in the value. next-intl values are ICU MessageFormat: the placeholders are the ICU argument and rich-text tag names, plural follows an ICU plural or selectordinal argument, and the ICU body is kept verbatim through the pipeline. ngx-translate has no built-in plural or ICU, and its files may be flat with dotted keys or nested; verbatra preserves whichever style the file uses on write.

## Why the format matters

The format tells verbatra two things it needs for a safe run. First, the placeholder syntax, so the [placeholder integrity check](/docs/translation-safety) knows what to compare before and after translation. Second, for ICU, which values to validate, so an invalid ICU source key is skipped rather than sent in a broken state. The wrong format compares the wrong tokens, so match it to your i18n library.

## How files are read and written

verbatra caps input size and nesting depth on read, and resists a file changing underneath it. It writes atomically: it writes to a temporary file, then renames it into place, so an interrupted write never leaves a half-finished locale file. When verbatra cannot handle a file, it raises a structured, secret-free error rather than a raw one, for example on invalid JSON, a structure too deep, or, for ngx-translate only, a file that mixes flat dotted keys with nested objects.

## Dotted keys

For i18next, vue-i18n, and next-intl, a literal dotted leaf (a key such as `"foo.bar"` used as a single leaf) round-trips losslessly: verbatra reads it and writes it back with its on-disk shape preserved, not re-nested into `foo` then `bar`. Real nested paths stay nested. The one case that still fails by design is a genuine collision, where one file expresses the same effective path both as a literal dotted leaf and as a real nested path (for example `"foo.bar"` alongside `"foo": { "bar": ... }`); that run errors with `INVALID_STRUCTURE` rather than guessing or corrupting data. ngx-translate is unchanged: a dotted key there denotes a nested path, so its existing flat-vs-nested handling applies and a file that mixes flat dotted keys with nested objects still errors.

## Choosing a format

Match the format to your framework: `i18next-json` for i18next, `vue-i18n-json` for vue-i18n, `next-intl-json` for next-intl, and `ngx-translate-json` for ngx-translate. XLIFF, YAML, and ARB are planned for later releases. See [Config file](/docs/config-file) for where `format` sits in the config.

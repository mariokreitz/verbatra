---
title: Translation safety
description: The integrity checks and security boundaries that decide what verbatra writes and what it withholds.
---
verbatra doesn't trust machine output by default. It runs every translation through integrity checks
before it writes, and it keeps untrusted text away from the instructions it sends the provider. This
page covers what those checks are and what they protect.

## Placeholder integrity

A translated value has to carry the same placeholders as its source. After translation, verbatra
compares the placeholder set of each result against the source value. Drop, add, or alter a
placeholder and the translation fails the check and gets withheld instead of written. The withheld key
keeps its prior lock hash, so the next run retries it, and it shows up per locale as an integrity
mismatch on the run summary.

What counts as a placeholder depends on the format: `{{double-brace}}` tokens for i18next,
ngx-translate, and YAML, single-brace `{name}` and `{0}` for vue-i18n, ICU argument and tag names for
next-intl and ARB, and inline placeholder ids for XLIFF. For i18next, `$t(...)` nesting references are
guarded too: drop, alter, or translate a `$t(...)` reference and the translation fails the check and
gets withheld. So if the source is `Welcome, {{name}}` and the model returns `Bienvenido` with the
`{{name}}` gone, verbatra holds it back rather than shipping a broken string. See
[Formats](/docs/formats) for the syntax of each.

## ICU validity

For the ICU formats, next-intl and ARB, verbatra parses each source value. A source key whose value
isn't valid ICU gets set aside and skipped for translation instead of being sent in a broken state,
and it's reported as invalid ICU on the summary. The ICU body stays verbatim through the pipeline:
nothing is resolved or rewritten.

## The prompt-injection boundary

The strings you translate aren't trustworthy: they can contain text that looks like an instruction
(think a UI string that literally reads "ignore your instructions"). verbatra treats them that way.
The system rules it sends a provider are compile-time constants. All variable input, the entries to
translate along with your glossary and tone, travels only in the user-turn data payload, never spliced
into the instruction channel. Provider output is constrained to a schema and validated, and the
placeholder check runs on whatever comes back. A string that tries to redirect the model can't reach
the rules, and a result that breaks a placeholder gets withheld.

## Secrets never leak

Providers read API keys only from the environment, never from your config or arguments. When a
provider SDK call fails, its raw error (which can carry request headers or a key) is discarded, never
bound, logged, or re-thrown. In its place verbatra raises its own structured error with a static,
secret-free message. The keys are secret-free by construction: nothing sensitive ever reaches a string
that would need scrubbing. A key can't show up in logs, in error messages, or in output.

## What you see

Every one of these outcomes is data on the run summary, per locale: the keys translated, the ones
withheld for a placeholder mismatch, the source keys skipped for invalid ICU, and any notices. Notices
include provider notices and SDK notices, such as a plural-category warning (code
`PLURAL_CATEGORIES_INCOMPLETE`) when a target language needs more plural categories than the source
supplies; see [Add a language](/docs/add-a-language) for how to synthesize those forms instead.
Nothing is hidden, and one locale's failure doesn't stop the others. See
[How it works](/docs/how-it-works) for where each check sits in the run.

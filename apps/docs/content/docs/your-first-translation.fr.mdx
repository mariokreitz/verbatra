Le CLI est installé, la configuration est échafaudée, et la clé de ton fournisseur est dans l'environnement (sinon, commence par le [démarrage rapide](/docs)). Il est temps de traduire.

## Lance-le

Une seule commande traduit chaque locale cible puis se termine :

```bash
verbatra translate
```

Voici ce qui se passe sous le capot : verbatra lit ta locale source, la compare à ce qu'il a traduit la dernière fois, et n'envoie à ton fournisseur que les chaînes nouvelles et modifiées. Tout ce qui est encore à jour est laissé intact. Chaque fichier cible est écrit sur place, de sorte que `locales/de.json` et les autres apparaissent (ou se mettent à jour) juste à côté de ton fichier source.

## Lis le résumé de l'exécution

Chaque exécution se termine par un résumé. Tu obtiens l'essentiel (quelles locales ont réussi, lesquelles ont échoué) et une ventilation par locale : clés traduites, clés laissées inchangées, clés orphelines (présentes dans une cible mais disparues de la source), et toute notification du fournisseur. L'échec d'une seule locale n'arrête pas les autres ; il est signalé sur la ligne de cette locale avec un code et un message.

Tu le veux lisible par machine ? Ajoute `--json` et verbatra affiche tout le résumé sous forme d'un seul objet JSON :

```bash
verbatra translate --json
```

Voir [CI et codes de sortie](/docs/ci-and-exit-codes) pour la structure du JSON et le contrat des codes de sortie.

## Prévisualise avant de dépenser un token

Pour voir exactement ce qu'une exécution ferait sans appeler de fournisseur ni écrire un seul fichier, fais une exécution à blanc :

```bash
verbatra translate --dry-run
```

Tu obtiens le même résumé, listant les clés qui seraient traduites, mais rien n'est envoyé et rien n'est écrit. C'est le moyen sûr de vérifier une exécution avant qu'elle ne te coûte quoi que ce soit, ce qui est particulièrement pratique avec une clé de niveau gratuit.

## Les réexécutions ne touchent que ce qui a changé

C'est ce qui rend verbatra économique à lancer souvent. Relance `verbatra translate` sans modifier ta source et rien ne part vers le fournisseur : chaque clé est déjà à jour. Modifie une seule chaîne source et seule cette clé est retraduite à la prochaine exécution.

L'astuce est un fichier de verrouillage versionné (`verbatra.lock.json`) qui enregistre ce qui a été traduit. Chaque exécution le compare, de sorte que le fournisseur n'est appelé que pour ce qui a réellement changé. Voir [Comment ça marche](/docs/how-it-works) pour le pipeline complet.

<Callout title="Commite le fichier de verrouillage">
verbatra écrit `verbatra.lock.json` lors de la première exécution. Commite-le avec tes fichiers de locale afin que chaque machine, et ta CI, se comparent à la même base de référence. Voir [Le fichier de verrouillage](/docs/the-lock-file).
</Callout>

## Traduis pendant que tu travailles

Tu ne veux pas relancer à la main ? Surveille la source et laisse verbatra retraduire à chaque modification :

```bash
verbatra watch
```

Il continue de tourner jusqu'à ce que tu l'arrêtes avec Ctrl-C.

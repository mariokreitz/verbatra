# Konfigurationsdatei

verbatra liest ein kleines, nicht geheimes Konfigurationsobjekt und validiert es, bevor es irgendeine Arbeit verrichtet. Hier ist kein API-Schlüssel enthalten: Der Provider liest seinen Schlüssel aus der Umgebung, sodass diese Datei sicher zu committen ist.

<Callout title="Schlüssel kommen aus der Umgebung">
Aus dieser Datei wird niemals ein API-Schlüssel gelesen. Jeder Provider liest seinen Schlüssel aus einer Umgebungsvariable, sodass die Konfiguration sicher zu committen bleibt. Siehe [Provider](/docs/providers).
</Callout>

## Eine minimale Konfiguration

```ts
import { defineConfig } from "@verbatra/sdk";

export default defineConfig({
  sourceLocale: "en",
  targetLocales: ["de", "fr"],
  format: "i18next-json",
  files: {
    pattern: "locales/{locale}.json",
  },
  provider: {
    id: "gemini",
    options: {
      model: "gemini-2.5-flash", // replace with your provider's model id
      maxOutputTokens: 4096,
    },
  },
});
```

`defineConfig` ist ein Identitäts-Helfer aus `@verbatra/sdk`. Es gibt sein Argument unverändert zurück und existiert nur, um dir Typinferenz und Editor-Autovervollständigung zu geben.

<Callout title="Modell-Autovervollständigung in TypeScript">
Wenn du die Konfiguration in TypeScript schreibst, bietet dein Editor die bekannten Modell-IDs des gewählten Providers für `options.model` an, eingegrenzt durch die `id` des Providers, den du gewählt hast. Die Liste stammt aus dem eigenen SDK des Providers (`@anthropic-ai/sdk`, `openai`, `@google/genai`), sodass sie mit der installierten SDK-Version aktuell bleibt. Das Feld ist auf die Modelle dieses Providers beschränkt: Ein Modell eines anderen Providers (zum Beispiel ein Claude-Modell unter `id: "gemini"`) ist ein Typfehler. Die Laufzeit validiert `model` nur als nicht-leeren String, sodass ein brandneues Modell, das das SDK noch nicht gelistet hat, zur Laufzeit trotzdem funktioniert, auch wenn der Editor es markiert.
</Callout>

## Felder

- **`sourceLocale`** (string): das Locale, in dem deine Quell-Strings geschrieben sind.
- **`targetLocales`** (String-Array): die Locales, in die übersetzt werden soll. Es darf `sourceLocale` nicht enthalten.
- **`format`** (string): eines von `i18next-json`, `vue-i18n-json`, `next-intl-json`, `ngx-translate-json`, `arb`, `yaml` oder `xliff`. Siehe [Formate](/docs/formats) dafür, was jedes erkennt und wie es mit Platzhaltern umgeht.
- **`files.pattern`** (string): der Pfad zu jeder Locale-Datei. Er muss das `{locale}`-Token enthalten, das verbatra durch jedes Locale ersetzt (zum Beispiel wird `locales/{locale}.json` zu `locales/de.json` aufgelöst).
- **`provider`** (object): der Übersetzungs-Provider, ausgewählt über `id`. Jeder Provider nimmt seine eigenen `options`, und in TypeScript ist das `model`-Feld auf die bekannten Modell-IDs des gewählten Providers beschränkt (ein Modell eines anderen Providers ist ein Typfehler). Siehe [Provider](/docs/providers).
- **`glossary`** (optional, object): eine Zuordnung von Quellbegriffen zu bevorzugten Zielbegriffen. Die LLM-Provider (Anthropic, OpenAI, Gemini) erhalten sie mit der Anfrage und werden angewiesen, ihre Begriffe als bindend zu behandeln. DeepL wendet diese Begriffszuordnung nicht an; es vermerkt einen Hinweis und wendet stattdessen ein über die ID konfiguriertes Glossar an (siehe [Provider](/docs/providers)).
- **`tone`** (optional): eines von `"formal"`, `"informal"` oder `"neutral"`. Die LLM-Provider erhalten es und werden angewiesen, es zu berücksichtigen. DeepL ordnet es der Formalität zu: `formal` wird formeller, `informal` wird weniger formell, und `neutral` oder ein fehlender Ton wird weggelassen. Bei einem DeepL-Schlüssel der kostenlosen Stufe wird ein nicht-standardmäßiger Ton nicht unterstützt und mit einem Hinweis auf den Standardwert herabgestuft.
- **`prune`** (optional, boolean, Standard `false`): wenn wahr, werden Schlüssel, die in einer Zieldatei vorhanden, aber nicht in der Quelle vorhanden sind (die verwaisten Schlüssel des Diffs), aus der geschriebenen Datei und der Lock-Datei entfernt. Eine Pro-Lauf-`prune`-Option bei `translate` (das CLI-Flag `--prune`) überschreibt dies. Es werden immer nur verwaiste Schlüssel entfernt.
- **`generatePlurals`** (optional, boolean, Standard `false`): wenn wahr, synthetisiert verbatra die CLDR-Pluralformen, die eine Zielsprache benötigt, die Quelle aber nicht liefert (zum Beispiel Polnisch few/many). Wird nur für i18next-JSON-Projekte unterstützt, die von einem LLM-Provider übersetzt werden; DeepL, Nicht-i18next-Formate und unbekannte Zielsprachen fallen auf die Plural-Warnung pro Locale zurück. Anders als bei `prune` gibt es kein `--generate-plurals`-CLI-Flag, also setze dies in der Konfiguration; die SDK-`translate()`-Eingabe akzeptiert jedoch eine Pro-Lauf-`generatePlurals`-Überschreibung, die Vorrang hat.
- **`maxBatchSize`** (optional, positive Ganzzahl, Standard `50`): die fehlend-plus-geänderten Einträge eines Locales werden in sequenzielle Teil-Batches aufgeteilt, die nicht größer als dieser Wert sind, sodass eine einzige übergroße Provider-Anfrage nicht das gesamte Locale fehlschlagen lassen kann. Ein fehlgeschlagener Teil-Batch wird zurückgehalten und beim nächsten Lauf erneut versucht, während die anderen weiterhin Fortschritt machen. Dies ist nur über die Konfiguration möglich: Es gibt kein CLI-Flag und keine Pro-Lauf-`translate()`-Überschreibung, also setze es in der Konfiguration.

Unbekannte Schlüssel auf oberster Ebene werden abgelehnt. Das ist auch der Grund, warum ein API-Schlüssel nicht in der Konfiguration leben kann: Schlüssel kommen aus der Umgebung, nicht aus der Datei.

Eine Konfiguration, die die optionalen Felder verwendet:

```ts
import { defineConfig } from "@verbatra/sdk";

export default defineConfig({
  sourceLocale: "en",
  targetLocales: ["de", "fr"],
  format: "i18next-json",
  files: {
    pattern: "locales/{locale}.json",
  },
  glossary: {
    "Sign in": "Anmelden",
  },
  tone: "formal",
  maxBatchSize: 25,
  provider: {
    id: "anthropic",
    options: {
      model: "claude-sonnet-4-6",
      maxTokens: 4096,
    },
  },
});
```

## Erkennungsreihenfolge

verbatra sucht vom Arbeitsverzeichnis aus aufwärts und verwendet die erste Quelle, die es findet. Mehrere vorhandene Quellen sind kein Fehler; der erste Treffer gewinnt. Die Suchreihenfolge ist:

1. `package.json` (eine `"verbatra"`-Eigenschaft)
2. `.verbatrarc`
3. `.verbatrarc.json`
4. `.verbatrarc.yaml`
5. `.verbatrarc.yml`
6. `.verbatrarc.js`
7. `.verbatrarc.cjs`
8. `.verbatrarc.ts`
9. `verbatra.config.js`
10. `verbatra.config.cjs`
11. `verbatra.config.ts`

Um eine explizite Datei zu laden, anstatt zu suchen, übergib `--config <path>`.

## Das Arbeitsverzeichnis und .env

Führe verbatra aus deinem Projekt-Stammverzeichnis aus, oder übergib `--cwd`, das darauf zeigt. Das Arbeitsverzeichnis ist der einzige Anker für alles, was verbatra auflöst: Die Konfigurationssuche beginnt dort und läuft aufwärts, `.env.local` und `.env` werden von dort geladen, und jeder Locale-Dateipfad wird dagegen aufgelöst. Alle drei am selben Ort zu halten, ist das, was ein einfaches `verbatra translate` funktionieren lässt.

Die Umgebungspräzedenz ist, höchste zuerst: eine Variable, die bereits in deiner echten Umgebung gesetzt ist, dann `.env.local`, dann `.env`. Eine bereits vorhandene Variable wird niemals überschrieben.

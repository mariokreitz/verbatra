---
title: GitHub Action
description: "Run verbatra in GitHub Actions: translate in CI, annotate failures, and write a job summary."
---
Want translations to run on every push? verbatra ships a composite GitHub Action that does exactly that. It runs `verbatra translate --json` in CI, annotates any failures, writes a job summary, and exits with the CLI's exit code. You wire it in with `uses:`. It is not published to npm.

## Availability

This action lives in the repository, so you reference it straight from there with `uses: mariokreitz/verbatra/packages/github-action@<commit-sha>`, pinned to a commit SHA. There is no npm package, no GitHub Marketplace listing, and no versioned action tag. The path-based `uses:` reference pinned to a commit SHA is the only supported way to consume it.

## Usage

```yaml
name: translate
on:
  push:
    branches: [main]

permissions:
  contents: read

jobs:
  verbatra:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<commit-sha>
      - uses: mariokreitz/verbatra/packages/github-action@<commit-sha>
        with:
          version: 0.3.0 # pin @verbatra/cli to an exact version
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

The action runs the CLI, so it reads the provider's API key from the environment. Pass it in from your repository secrets under the variable your provider expects (see [Providers](/docs/providers)): `ANTHROPIC_API_KEY` in the example above, `OPENAI_API_KEY` if you run OpenAI, and so on.

## Inputs

| Input | Required | Default | Description |
| --- | --- | --- | --- |
| `version` | yes | - | the `@verbatra/cli` version to run, e.g. `0.3.0`. Pin this to an exact version; do not use a floating tag such as `latest` |
| `config-path` | no | `""` | explicit config file to load (maps to `--config`); empty uses the normal config search |
| `working-directory` | no | `""` | directory to resolve config and locale files against (maps to `--cwd`) |
| `dry-run` | no | `"false"` | set to `"true"` to report what would change without calling a provider or writing (maps to `--dry-run`) |
| `node-version` | no | `"24"` | Node.js version to set up for running the CLI |

The action always runs `translate --json`. It never runs `init` or `watch`.

## Exit behavior

The action grabs the CLI's stdout and exit code without bailing early. It annotates failures, writes the job summary, and only then propagates the CLI's exit code. So the step fails on `1` (some locales failed) or `2` (could not run), but not before you get the annotations and summary. The [CLI exit codes](/docs/cli#exit-codes) spell out what each code means.

## Persisting translations

Without `dry-run`, `translate` writes updated locale files into the runner's checkout, but the action stops there: it does not commit them. To keep the changes, add your own step that commits and pushes, or opens a pull request. When you only want CI to flag missing or stale translations without writing anything, set `dry-run: "true"`.

## Security

Pin both the action and the CLI to exact references. Pin `uses:` to a commit SHA, and set the `version` input to an exact `@verbatra/cli` release instead of a floating tag like `latest`. That keeps a run reproducible and stops it from auto-pulling a compromised release. Give the workflow only the privilege it needs: `contents: read` for a check, plus `contents: write` and `pull-requests: write` only when a later step commits or opens a pull request. Inputs reach the CLI as data, never spliced into the shell, so a crafted input value stays an argument and never becomes executable code.

# verbatra diff

Zeigt pro Ziel-Locale die tatsächlich ausstehenden Übersetzungsänderungen als Schlüssellisten: die Schlüssel, die hinzugefügt würden, die Schlüssel, die neu übersetzt würden, und die Schlüssel, die jetzt im Ziel verwaist sind. `diff` führt dieselbe Lese- und Diff-Operation aus, die [`verbatra translate`](/docs/cli/translate) macht, bevor es einen Provider aufrufen würde, hört aber dort auf: Es ruft keinen Provider auf, schreibt keine Dateien und berührt niemals die [Lock-Datei](/docs/the-lock-file).

```bash
verbatra diff
```

`diff` ist das detaillierte Geschwister von [`verbatra check`](/docs/cli/check). Wo `check` mit Anzahlen pro Locale die Frage "wie viel ist abgewichen" beantwortet, beantwortet `diff` mit den expliziten Listen die Frage "welche Schlüssel". Beide führen dieselbe schreibgeschützte Berechnung aus.

## Die drei Schlüssellisten

Für jedes Ziel-Locale meldet `diff` drei Gruppen:

- **add** (`missing`): Schlüssel, die im Quell-Locale vorhanden, aber im Ziel nicht vorhanden sind. Ein Lauf würde diese hinzufügen.
- **re-translate** (`changed`): Schlüssel, die in beiden vorhanden sind, bei denen sich der Quell-String geändert hat, seit er zuletzt übersetzt wurde. Ein Lauf würde diese neu übersetzen.
- **orphaned**: Schlüssel, die im Ziel vorhanden, aber in der Quelle nicht vorhanden sind. Diese werden nur gemeldet. Ein standardmäßiger `translate`-Lauf entfernt sie nicht, daher zählen verwaiste Schlüssel für sich allein nicht als ausstehende Arbeit (nur [`verbatra translate --prune`](/docs/cli/translate) entfernt sie).

Ein Locale hat ausstehende Änderungen, wenn es mindestens einen Schlüssel zum Hinzufügen oder Neuübersetzen hat. Verwaiste Schlüssel allein machen ein Locale nicht ausstehend.

## Optionen

| Option | Beschreibung |
| --- | --- |
| `--cwd <path>` | löst Konfigurations- und Locale-Dateien aus diesem Verzeichnis auf |
| `--config <path>` | lädt diese Konfigurationsdatei, anstatt nach einer zu suchen |
| `--locales <list>` | kommagetrennte Teilmenge der Ziel-Locales (Standard alle konfigurierten) |
| `--json` | gibt die Diff-Zusammenfassung als JSON aus |

Es gibt kein `--dry-run`: `diff` ist von Natur aus ein trockener Lesevorgang. Es benötigt keinen API-Schlüssel.

## Ausgabe

Ohne `--json` gibt `diff` eine Kopfzeile mit Anzahlen pro Locale aus, dann die Schlüssel gruppiert unter add, re-translate und orphaned. Ein Locale ohne etwas zum Hinzufügen, Neuübersetzen oder Verwaisen wird auf eine einzelne Zeile reduziert. Jeder Schlüssel wird aufgelistet, ohne Kürzung; leite an einen Pager weiter, wenn du einen möchtest.

```
verbatra diff
  de: 3 to add, 1 to re-translate, 2 orphaned
    add:          app.title, nav.home, nav.about
    re-translate: footer.copyright
    orphaned:     legacy.banner, legacy.promo
  fr: no pending changes
2 locales, pending changes
```

Mit `--json` gibt es die Zusammenfassung als ein JSON-Objekt auf stdout aus. Das oberste `hasPendingChanges` ist genau dann wahr, wenn mindestens ein Locale einen Schlüssel zum Hinzufügen oder Neuübersetzen hat (und genau dann, wenn sich der Befehl mit `1` beendet):

```json
{
  "hasPendingChanges": true,
  "locales": [
    {
      "locale": "de",
      "missing": ["app.title", "nav.home", "nav.about"],
      "changed": ["footer.copyright"],
      "orphaned": ["legacy.banner", "legacy.promo"],
      "hasPendingChanges": true
    },
    {
      "locale": "fr",
      "missing": [],
      "changed": [],
      "orphaned": [],
      "hasPendingChanges": false
    }
  ]
}
```

Das `hasPendingChanges` jedes Locales ist `missing.length > 0 || changed.length > 0`; `orphaned` ist ausgeschlossen. Ein Fehler des gesamten Laufs wird auf stderr geschrieben, sodass stdout sauber und unter `--json` pipebar bleibt.

## Beispiele

```bash
# list the keys that would be added, re-translated, or are orphaned (exit 1 if pending)
verbatra diff

# only diff the German and French locales
verbatra diff --locales de,fr

# machine-readable key lists on stdout for tooling or a PR comment
verbatra diff --json
```

## Exit-Codes

| Code | Bedeutung |
| --- | --- |
| `0` | keine ausstehenden Änderungen: kein Locale hat einen Schlüssel zum Hinzufügen oder Neuübersetzen |
| `1` | ausstehende Änderungen: mindestens ein Locale hat einen fehlenden oder geänderten Schlüssel (der vollständige Bericht wird trotzdem ausgegeben) |
| `2` | konnte nicht ausgeführt werden: ein Fehler des gesamten Laufs (fehlende oder ungültige Konfiguration, nicht lesbare Quelle, unbekanntes Format oder eine beschädigte Lock-Datei) oder ein Nutzungsfehler |

Verwaiste Schlüssel allein erzeugen niemals Exit `1`, weil ein standardmäßiger Lauf sie nicht entfernt. Bei `diff` bedeutet Exit `1` "es gibt ausstehende Arbeit", nicht einen Fehlschlag eines einzelnen Locales: Der Lauf selbst war erfolgreich. Siehe [CI und Automatisierung](/docs/ci-and-exit-codes) für das Verzweigen auf diese Codes.

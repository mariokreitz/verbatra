`@verbatra/cli` incluye el binario `verbatra`, una envoltura fina sobre [`@verbatra/sdk`](/docs/sdk). Ocho comandos hacen el trabajo:

| Comando | Qué hace |
| --- | --- |
| [`init`](/docs/cli/init) | crea una configuración de verbatra y un `.env.example` para este proyecto |
| [`translate`](/docs/cli/translate) | traduce cada locale de destino una vez y luego sale |
| [`watch`](/docs/cli/watch) | retraduce en cada cambio del origen hasta que lo interrumpas |
| [`check`](/docs/cli/check) | informa qué claves faltan o están obsoletas por locale, solo lectura |
| [`diff`](/docs/cli/diff) | lista las claves exactas que se añadirían, se retraducirían o están huérfanas por locale, solo lectura |
| [`export`](/docs/cli/export) | exporta las cadenas sin traducir a un libro de Excel para un traductor humano |
| [`import`](/docs/cli/import) | importa un libro rellenado de vuelta a los archivos de locale |
| [`studio`](/docs/cli/studio) | arranca Verbatra Studio, el panel local de traducción |

## Instala e invoca

Instala `@verbatra/cli` como dependencia de desarrollo y ejecuta el binario a través de tu gestor de paquetes:

```bash
pnpm add -D @verbatra/cli
pnpm verbatra translate      # or: npx verbatra translate, yarn verbatra translate
```

Necesitas Node.js `>=22.14.0`.

## Convenciones compartidas

- `--cwd <path>` resuelve la configuración y los archivos de locale desde ese directorio en lugar del actual. Todos los comandos lo aceptan.
- `--config <path>` carga ese archivo de configuración en lugar de buscar uno. Todos los comandos excepto `init` lo aceptan; el orden de búsqueda se describe en [Archivo de configuración](/docs/config-file).
- `--json` imprime un resultado legible por máquina en stdout, manteniendo stdout limpio para encadenar: los errores van a stderr. Lo admiten `translate`, `watch`, `check`, `diff`, `export` e `import`; `init` y `studio` no.
- `--help` y `--version` imprimen y salen con `0`. Un comando o flag desconocido sale con `2`.

## Archivos de entorno

`translate`, `watch` y `studio` cargan `.env.local` y luego `.env` desde el directorio de trabajo antes de ejecutarse. Una variable ya definida en tu entorno real siempre gana. Las claves API se leen solo del entorno, nunca del archivo de configuración ni de un flag: consulta [Proveedores](/docs/providers). Los comandos de solo lectura (`check`, `diff`) y los comandos de libro (`export`, `import`) nunca llaman a un proveedor y no cargan archivos `.env`.

## Códigos de salida

El código de salida es el contrato sobre el que ramifica un paso de CI o un script:

| Código | Significado |
| --- | --- |
| `0` | éxito (también `--help` y `--version`) |
| `1` | `translate` o `import` terminó pero algunos locales fallaron, `check` encontró un locale desincronizado, o `diff` encontró cambios pendientes |
| `2` | no se pudo ejecutar: un error de toda la ejecución (configuración, origen, proveedor, bloqueo) o un error de uso |
| `130` | `watch` o `studio` fue detenido a la fuerza por una segunda interrupción |

La página de cada comando detalla cómo se le aplican estos códigos. Consulta [CI y códigos de salida](/docs/ci-and-exit-codes) para conectarlos a una canalización.

`@verbatra/cli` te da el binario `verbatra`, un envoltorio ligero sobre [`@verbatra/sdk`](/docs/sdk). Siete comandos hacen el trabajo:

- [`init`](/docs/cli/init): genera una configuración y un `.env.example`.
- [`translate`](/docs/cli/translate): traduce cada locale de destino una vez y luego sale.
- [`watch`](/docs/cli/watch): vuelve a traducir con cada cambio del origen hasta que lo interrumpes.
- [`check`](/docs/cli/check): informa qué locales tienen cadenas faltantes o desactualizadas, de solo lectura, sin llamar a un proveedor.
- [`diff`](/docs/cli/diff): lista las claves exactas que se añadirían, se volverían a traducir o están huérfanas por locale, de solo lectura.
- [`export`](/docs/cli/export): vuelca las cadenas sin traducir a un libro de Excel para un traductor humano.
- [`import`](/docs/cli/import): lee un libro rellenado de vuelta a tus archivos de locale.

## Instalar e invocar

Instala verbatra como dependencia de desarrollo y luego ejecuta el binario a través de tu gestor de paquetes:

```bash
pnpm add -D @verbatra/cli
# pnpm verbatra ..., npx verbatra ..., o yarn verbatra ...
```

Necesitas Node.js `>=22.14.0`.

## Opciones globales

| Opción | Qué hace |
| --- | --- |
| `--help` | imprime el uso del programa o de un comando, luego sale con `0` |
| `--version` | imprime la versión instalada de `@verbatra/cli`, luego sale con `0` |

Ejecuta `verbatra <command> --help` para ver la lista completa de opciones y ejemplos de un comando: la misma referencia que estas páginas.

## El directorio de trabajo y .env

Antes de ejecutarse, `translate` y `watch` cargan `.env.local` y luego `.env` del directorio de trabajo. Una variable ya definida en tu entorno real siempre gana. ¿Invocas verbatra desde otro lugar que no sea la raíz de tu proyecto? Pasa `--cwd` para apuntar a ella. Las claves de API se leen solo del entorno, nunca de la configuración ni de una opción: consulta [Proveedores](/docs/providers).

## Códigos de salida

El código de salida de verbatra es el contrato sobre el que bifurca un paso de CI o un script:

| Código | Significado |
| --- | --- |
| `0` | éxito (también `--help` y `--version`) |
| `1` | `translate` o `import` terminó pero algunos locales fallaron, `check` encontró un locale fuera de sincronización, o `diff` encontró cambios pendientes |
| `2` | no se pudo ejecutar: un error de toda la ejecución, o un error de uso |
| `130` | `watch` fue detenido a la fuerza por una segunda interrupción |

Un fallo por ejecución en `watch` aparece como un registro de flujo, no como un código de salida. La página de cada comando detalla cómo se aplican estos códigos en él, y [CI y automatización](/docs/ci-and-exit-codes) cubre cómo conectarlos a una canalización.

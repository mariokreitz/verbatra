`@verbatra/cli` gives you the `verbatra` binary, a thin wrapper over [`@verbatra/sdk`](/docs/sdk). Eight commands do the work:

- [`init`](/docs/cli/init): scaffold a config and a `.env.example`.
- [`translate`](/docs/cli/translate): translate every target locale once, then exit.
- [`watch`](/docs/cli/watch): re-translate on every source change until you interrupt it.
- [`check`](/docs/cli/check): report which locales have missing or stale strings, read-only, without calling a provider.
- [`diff`](/docs/cli/diff): list the exact keys that would be added, re-translated, or are orphaned per locale, read-only.
- [`export`](/docs/cli/export): dump untranslated strings into an Excel workbook for a human translator.
- [`import`](/docs/cli/import): read a filled workbook back into your locale files.
- [`ui`](/docs/cli/ui): start Verbatra Studio, a local, read-only translation dashboard.

## Install and invoke

Install verbatra as a dev dependency, then run the binary through your package manager:

```bash
pnpm add -D @verbatra/cli
# pnpm verbatra ..., npx verbatra ..., or yarn verbatra ...
```

You need Node.js `>=22.14.0`.

## Global flags

| Flag | What it does |
| --- | --- |
| `--help` | print usage for the program or a command, then exit `0` |
| `--version` | print the installed `@verbatra/cli` version, then exit `0` |

Run `verbatra <command> --help` for a command's full option list and examples: the same reference as these pages.

## The working directory and .env

Before they run, `translate` and `watch` load `.env.local` and then `.env` from the working directory. A variable already set in your real environment always wins. Invoking verbatra from somewhere other than your project root? Pass `--cwd` to point at it. API keys are read only from the environment, never from the config or a flag: see [Providers](/docs/providers).

## Exit codes

verbatra's exit code is the contract a CI step or script branches on:

| Code | Meaning |
| --- | --- |
| `0` | success (also `--help` and `--version`) |
| `1` | `translate` or `import` finished but some locales failed, `check` found a locale out of sync, or `diff` found pending changes |
| `2` | could not run: a whole-run error, or a usage error |
| `130` | `watch` or `ui` was force-stopped by a second interrupt |

A per-run failure in `watch` shows up as a stream record, not an exit code. Each command page spells out how these codes apply to it, and [CI and automation](/docs/ci-and-exit-codes) covers wiring them into a pipeline.

---
title: verbatra translate
description: Übersetzt jedes Ziel-Locale einmal und beendet sich dann.
---
Führt den Ablauf aus Lesen, Diff, Übersetzen, Schreiben über jedes Ziel-Locale einmal aus und beendet sich dann. verbatra liest das Quell-Locale, vergleicht es mit dem, was es zuletzt übersetzt hat, und sendet nur die neuen oder geänderten Strings an den Provider. Alles, was noch aktuell ist, bleibt unberührt, sodass du niemals dafür zahlst, das neu zu übersetzen, was sich nicht bewegt hat.

```bash
verbatra translate
```

## Optionen

| Option | Beschreibung |
| --- | --- |
| `--cwd <path>` | löst Konfigurations- und Locale-Dateien aus diesem Verzeichnis auf |
| `--config <path>` | lädt diese Konfigurationsdatei, anstatt nach einer zu suchen |
| `--dry-run` | zeigt Änderungen in der Vorschau, ohne einen Provider aufzurufen oder Dateien zu schreiben |
| `--prune` | entfernt verwaiste Schlüssel (in einer Zieldatei, aber nicht in der Quelle vorhanden) aus der geschriebenen Datei und der Lock-Datei |
| `--json` | gibt die Lauf-Zusammenfassung als JSON aus |

Standardmäßig löscht ein Lauf nichts. Verwaiste Schlüssel (in einer Zieldatei vorhanden, aber nicht mehr in der Quelle) werden gemeldet und bleiben an Ort und Stelle. Übergib `--prune`, um genau diese Schlüssel aus der geschriebenen Datei und der Lock-Datei zu entfernen und sonst nichts. Dieses Pro-Lauf-Flag überschreibt die `prune`-Option in deiner [Konfiguration](/docs/config-file); kombiniere es mit `--dry-run`, um die Schlüssel in der Vorschau zu sehen, die entfernt würden. Du willst sehen, welche Schlüssel vor einem Lauf verwaist sind? Verwende [`verbatra diff`](/docs/cli/diff).

## Ausgabe

Ohne `--json` gibt `translate` eine menschenlesbare Lauf-Zusammenfassung aus. Mit `--json` gibt es die Lauf-Zusammenfassung als ein JSON-Objekt auf stdout aus: siehe die [`RunSummary`-Struktur in der SDK-Referenz](/docs/sdk). Ein Fehler des gesamten Laufs wird auf stderr geschrieben, sodass stdout sauber und unter `--json` pipebar bleibt.

Eine Locale-Zeile zeigt eine `needs-review`-Zahl nur, wenn sie größer als null ist: verbatras eigene Heuristiken, die einen übersetzten Key markieren, der einen zweiten Blick wert ist (eine verdächtig kurze oder lange Ausgabe, ein mit der Quelle identischer Wert, ein verpasster Glossarbegriff oder eine neu geordnete Platzhaltermenge). Sie hält von sich aus nichts zurück; siehe [Übersetzungssicherheit](/docs/translation-safety) dafür, was jeder Grund bedeutet. Die `--json`-Ausgabe trägt immer das volle Detail als `needsReview` auf jeder Locale.

## Beispiele

```bash
# translate once using the config it finds
verbatra translate

# preview changes without calling a provider or writing files
verbatra translate --dry-run

# also remove orphaned keys from target files
verbatra translate --prune

# preview the keys that would be pruned
verbatra translate --prune --dry-run

# run against a project in another directory
verbatra translate --cwd packages/app

# load a specific config instead of searching for one
verbatra translate --config config/verbatra.config.ts

# machine-readable summary on stdout
verbatra translate --json
```

## Exit-Codes

| Code | Bedeutung |
| --- | --- |
| `0` | jedes Ziel-Locale war erfolgreich |
| `1` | der Lauf wurde beendet, aber eines oder mehrere Locales sind fehlgeschlagen |
| `2` | konnte nicht ausgeführt werden: ein Fehler des gesamten Laufs (fehlende oder ungültige Konfiguration, nicht lesbare Quelle, unbekanntes Format, ein Provider, der nicht gebaut werden konnte, einschließlich eines fehlenden API-Schlüssels, oder eine beschädigte Lock-Datei) oder ein Nutzungsfehler |

Ein einzelnes fehlschlagendes Locale ist nicht fatal: Es ergibt Exit `1`, und das Detail pro Locale steht in der Zusammenfassung. Siehe [CI und Automatisierung](/docs/ci-and-exit-codes) für das Verzweigen auf diese Codes.

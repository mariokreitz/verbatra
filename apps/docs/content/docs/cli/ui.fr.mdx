Verbatra Studio est un tableau de bord web local sur ton projet verbatra : une vue en direct du statut de traduction, des changements en attente, de la configuration, de la dérive du fichier de verrouillage et de l'historique, sans jamais appeler de fournisseur ni écrire de fichier. Démarre-le avec :

```bash
verbatra ui
```

## Uniquement local et en lecture seule

Studio se lie uniquement à `127.0.0.1` : il n'est jamais accessible depuis une autre machine, et il n'y a aucune option pour changer cela. Chaque requête est vérifiée contre une liste stricte de Host et Origin autorisés, et chaque page et chaque appel API nécessitent le jeton de session affiché dans l'URL de démarrage, comparé en temps constant pour qu'une tentative incorrecte ne puisse pas être devinée par chronométrage. Studio n'envoie aucun en-tête CORS, si bien qu'un onglet de navigateur sur une autre origine ne pourrait pas le lire même en essayant. Il n'y a aucun endpoint qui modifie l'état dans Studio : il n'appelle jamais `translate`, n'exécute jamais `watch`, et n'importe ni n'exporte jamais un classeur. Tout ce que tu vois provient des mêmes calculs en lecture seule que [`verbatra check`](/docs/cli/check) et [`verbatra diff`](/docs/cli/diff).

## Référence de la commande

| Option | Description |
| --- | --- |
| `--cwd <path>` | résoudre la configuration et les fichiers de locale depuis ce répertoire |
| `--config <path>` | charger ce fichier de configuration au lieu d'en rechercher un |
| `--port <n>` | remplacer le port par défaut de Studio (doit être un entier de 1 à 65535) |

`verbatra ui` charge `.env.local` et `.env` exactement comme toute autre commande, puis résout ta configuration une seule fois, avant même de toucher le réseau. Une configuration manquante ou invalide quitte avec `2` et la même erreur affichée que toute autre commande ; Studio lui-même n'est alors jamais démarré.

## L'URL affichée

Une fois que Studio écoute, il affiche une ligne sur stdout :

```
Verbatra Studio running at http://127.0.0.1:5849/?token=<token>
```

Ouvre exactement cette URL. Le jeton dans la chaîne de requête amorce un cookie de session pour l'onglet du navigateur ; sans lui (ou avec le mauvais) toute requête est rejetée. Rien n'est jamais affiché avant que le serveur n'ait réussi à lier son port, si bien qu'un démarrage échoué ne divulgue jamais d'URL ni de jeton.

## Redémarrage lors d'un changement de configuration

La configuration est résolue une seule fois, au démarrage, et chaque vue, pendant toute la durée de vie de ce processus, s'appuie sur cette même configuration résolue. Si tu modifies `verbatra.config.ts` (une nouvelle locale cible, un fournisseur différent, un motif de fichier différent), arrête Studio et redémarre-le pour appliquer le changement. Les fichiers du projet eux-mêmes (tes fichiers de locale source et cible, ainsi que le fichier de verrouillage) sont relus à chaque requête, si bien que les modifier puis recharger la page affiche toujours les données actuelles ; Studio ne pousse pas encore de rafraîchissement automatique lorsqu'un fichier change sur le disque.

## Les vues

- **Statut** : les décomptes de clés manquantes, obsolètes et à jour par locale, le même calcul que `verbatra check`.
- **Diff** : les clés exactes qui seraient ajoutées, retraduites ou orphelines par locale, le même calcul que `verbatra diff`. Les longues listes de clés sont plafonnées dans la vue (avec une mention "affiche N sur le total") pour qu'un très grand catalogue reste réactif ; le filtrage continue de parcourir la liste complète.
- **Configuration et glossaire** : une projection expurgée de ta configuration résolue (locales source et cible, format, motif de fichier, id du fournisseur et les options scalaires sûres) ainsi que tes termes de glossaire, s'il y en a. Les valeurs en texte libre qui pourraient contenir un secret collé ne sont jamais incluses.
- **Dérive du verrouillage** : si un fichier de verrouillage existe, sa version et sa dérive par locale par rapport aux fichiers source et cible actuels, sans y écrire.
- **Historique** : l'historique des commits pour tes fichiers de locale surveillés, lu via `git log` avec un nombre de commits plafonné et sans `--follow`. Un projet sans dépôt git, ou avec git indisponible, affiche l'historique comme non disponible plutôt qu'une erreur. L'historique tronqué d'un clone superficiel n'est pas non plus une erreur : il affiche simplement ce que le clone possède.

## Installation

`@verbatra/ui` est le paquet propre de Studio. S'il n'est pas installé, `verbatra ui` quitte avec `2` et :

```
Verbatra Studio requires @verbatra/ui. Install it with: pnpm add -D @verbatra/ui
```

## Codes de sortie

| Code | Signification |
| --- | --- |
| `0` | Studio s'est arrêté proprement suite à un seul interrupt (Ctrl-C) |
| `2` | impossible de s'exécuter : une erreur d'exécution globale (configuration manquante ou invalide, `@verbatra/ui` non installé, ou le port est déjà utilisé), ou une erreur d'usage (un `--port` hors plage ou qui n'est pas un entier) |
| `130` | arrêté de force par un second interrupt |

# CLI

`@verbatra/cli` fournit le binaire `verbatra`, un mince wrapper autour de [`@verbatra/sdk`](/docs/sdk). Il embarque trois commandes :

- [`init`](/docs/cli/init) : échafaudez une configuration et un `.env.example`.
- [`translate`](/docs/cli/translate) : traduisez chaque locale cible une fois, puis quittez.
- [`watch`](/docs/cli/watch) : retraduisez à chaque modification de la source jusqu'à interruption.

## Installer et invoquer

verbatra est une dépendance de développement. Installez-le, puis exécutez le binaire via votre gestionnaire de paquets :

```bash
pnpm add -D @verbatra/cli
# pnpm verbatra ..., npx verbatra ..., ou yarn verbatra ...
```

Nécessite Node.js `>=22.14.0`.

## Options globales

| Option | Ce qu'elle fait |
| --- | --- |
| `--help` | affiche l'usage du programme ou d'une commande, puis quitte avec `0` |
| `--version` | affiche la version installée de `@verbatra/cli`, puis quitte avec `0` |

Exécutez `verbatra <command> --help` pour la liste complète des options et des exemples d'une commande - la même référence que ces pages.

## Le répertoire de travail et .env

`translate` et `watch` chargent `.env.local` puis `.env` depuis le répertoire de travail avant de s'exécuter ; une variable déjà définie dans votre environnement réel l'emporte toujours. Passez `--cwd` pour désigner la racine de votre projet si vous invoquez verbatra depuis un autre endroit. Les clés API ne sont lues que depuis l'environnement, jamais depuis la configuration ni une option - voir [Fournisseurs](/docs/providers).

## Codes de sortie

verbatra utilise son code de sortie comme le contrat sur lequel une étape de CI ou un script s'embranche :

| Code | Signification |
| --- | --- |
| `0` | succès (également `--help` et `--version`) |
| `1` | `translate` s'est terminé mais certaines locales ont échoué (translate uniquement) |
| `2` | impossible de s'exécuter : une erreur d'exécution globale, ou une erreur d'usage |
| `130` | `watch` a été arrêté de force par une seconde interruption |

L'échec d'une exécution dans `watch` est signalé comme un enregistrement de flux, pas comme un code de sortie. Chaque page de commande indique comment ces codes s'y appliquent, et [CI et automatisation](/docs/ci-and-exit-codes) couvre leur utilisation dans un pipeline.

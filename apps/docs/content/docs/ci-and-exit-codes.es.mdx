# CI y códigos de salida

verbatra es una dependencia de desarrollo, así que los comandos que ejecutas localmente se ejecutan igual en CI. Instálalo, pasa la clave del proveedor desde tu almacén de secretos de CI como la variable de entorno correspondiente y ejecuta `verbatra translate`.

```bash
pnpm add -D @verbatra/cli
# ANTHROPIC_API_KEY is provided by the CI secret store as an environment variable
verbatra translate --json
```

¿Quieres hacer fallar una canalización cuando las traducciones se han desviado, sin llamar a un proveedor ni siquiera necesitar una clave? Ejecuta [`verbatra check`](/docs/cli/check): sale con `1` cuando algún locale está fuera de sincronización.

```bash
# read-only gate: exit 1 if any locale is missing or stale
verbatra check
```

¿Necesitas las claves exactas detrás de esa desviación (por ejemplo, para publicarlas en un comentario de pull request)? Ejecuta [`verbatra diff`](/docs/cli/diff): la misma puerta de solo lectura, pero en lugar de solo contar, lista las claves que se añadirían, se volverían a traducir o están huérfanas.

```bash
# read-only gate with the key lists: exit 1 if any locale has pending changes
verbatra diff
```

## Códigos de salida

El código de salida es el contrato sobre el que se ramifica tu paso de CI:

| Código | Significado |
| --- | --- |
| `0` | éxito (para `check`, todos los locales están sincronizados; para `diff`, sin cambios pendientes) |
| `1` | `translate` o `import` terminó pero algunos locales fallaron, `check` encontró un locale fuera de sincronización, o `diff` encontró cambios pendientes |
| `2` | no se pudo ejecutar (un error de toda la ejecución o un error de uso) |
| `130` | `watch` fue detenido forzosamente por una segunda interrupción |

`watch` es la excepción: un fallo por ejecución aparece como un registro de flujo, no como un código de salida, y una sola interrupción detiene `watch` de forma elegante y sale con `0`.

## Salida JSON

Cada comando (`translate`, `watch`, `check`, `diff`, `export` e `import`) acepta `--json` para una salida legible por máquina.

`verbatra translate --json` imprime el resumen de la ejecución como un único objeto JSON con esta forma:

```ts
interface RunSummary {
  dryRun: boolean;          // whether this was a dry run (no provider calls, no writes)
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
}
```

`verbatra watch --json` imprime un registro por ejecución como NDJSON (un objeto JSON por línea). Cada registro es un éxito o un fallo:

```ts
type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };
```

Una ejecución fallida durante `watch` cae como uno de estos registros en el flujo. No detiene el observador y no establece un código de salida distinto de cero.

`verbatra check --json` imprime un documento de estado de solo lectura como un único objeto JSON. El `inSync` de nivel superior es true exactamente cuando todos los locales están sincronizados, que es exactamente cuando el comando sale con `0`:

```ts
interface CheckSummary {
  inSync: boolean;             // true exactly when the command exits 0
  locales: LocaleCheckSummary[];
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;             // in source, absent from target
  stale: number;               // source changed since last translated
  upToDate: number;            // target matches the current source
  inSync: boolean;             // missing === 0 && stale === 0
}
```

`verbatra diff --json` ejecuta el mismo cálculo de solo lectura pero te da listas de claves en lugar de recuentos. El `hasPendingChanges` de nivel superior es true exactamente cuando al menos un locale tiene una clave para añadir o volver a traducir, que es exactamente cuando el comando sale con `1`. Las claves huérfanas aparecen en las listas pero nunca establecen `hasPendingChanges`:

```ts
interface DiffSummary {
  hasPendingChanges: boolean;  // true exactly when the command exits 1
  locales: LocaleDiff[];
}

interface LocaleDiff {
  locale: string;
  missing: string[];           // in source, absent from target: would be added
  changed: string[];           // source changed since last translated: would be re-translated
  orphaned: string[];          // in target, absent from source: reported only
  hasPendingChanges: boolean;  // missing.length > 0 || changed.length > 0
}
```

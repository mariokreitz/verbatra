# Comment ça marche

Une exécution de verbatra est un court pipeline : charger la configuration, lire la locale source, puis pour chaque
locale cible diffuser par rapport à ce qui a été traduit la dernière fois, n'envoyer au fournisseur que ce qui a changé, vérifier le
résultat et écrire. Cette page parcourt chaque étape afin que tu saches exactement ce que fait une exécution.

## Découverte et chargement de la configuration

Une exécution commence par le chargement de la configuration. verbatra effectue une recherche vers le haut à partir du répertoire de travail et
utilise la première source qu'il trouve : une clé `"verbatra"` dans `package.json`, puis l'un des
fichiers `.verbatrarc.*`, puis un fichier `verbatra.config.*`. La configuration est validée avant que tout travail ne commence. Si rien n'est
trouvé, l'exécution échoue avec `CONFIG_NOT_FOUND` ; si une configuration est présente mais invalide, elle échoue avec
`CONFIG_INVALID`. La configuration ne porte aucune clé API ; le fournisseur lit sa clé depuis l'environnement.

## Lecture de la locale source

verbatra résout le fichier source à partir de `files.pattern` en substituant le jeton `{locale}` par
`sourceLocale`, par rapport au répertoire de travail. Si ce fichier est manquant, l'exécution échoue avec
`SOURCE_UNREADABLE` ; s'il ne peut pas être analysé, elle échoue avec `SOURCE_INVALID`. Pendant la lecture, l'adaptateur de format
note également toute clé source dont les valeurs sont invalides pour la syntaxe de message du format (ICU
invalide). Ces clés sont transportées et ignorées pour la traduction plutôt qu'envoyées dans un état défectueux.

## L'exécution par locale

Chaque locale cible est traitée séparément. Un échec dans une locale n'interrompt pas les autres.

### Diff par rapport à la base de référence du verrouillage

verbatra lit le fichier cible existant (un fichier absent est traité comme vide) et diffuse la source
par rapport à lui, en utilisant le fichier de verrouillage comme base de référence de ce qui a été traduit auparavant. Le diff trie chaque
clé source dans l'un de quatre paniers :

- les nouvelles clés que la cible ne possède pas encore,
- les clés modifiées dont le texte source diffère de la base de référence enregistrée,
- les clés inchangées qui sont déjà à jour,
- les clés orphelines présentes dans la cible mais qui ne sont plus dans la source.

Seules les clés nouvelles et modifiées sont candidates à la traduction. Les clés inchangées sont laissées intactes, et
les clés orphelines sont signalées et, par défaut, laissées en place. Pour opter pour leur suppression, passe `--prune`
(ou définis `prune: true` dans la configuration) : verbatra supprime alors exactement les clés orphelines du fichier
cible et du verrouillage, et rien d'autre. Toute candidate marquée ICU invalide est également mise de côté.
C'est ce qui maintient une exécution incrémentielle : un fournisseur n'est appelé que pour les chaînes qui ont réellement changé.

### Appel du fournisseur

Les entrées candidates sont envoyées au fournisseur configuré en un seul lot, accompagnées des options facultatives
`glossary` et `tone` de ta configuration. La façon dont elles sont appliquées dépend du fournisseur.

- Les fournisseurs LLM (Anthropic, OpenAI, Gemini) reçoivent les entrées, le glossaire et la tonalité dans une
  charge utile de données structurée, et sont invités à traiter les termes du glossaire comme contraignants et à respecter la
  tonalité. Le modèle est contraint de renvoyer une valeur par clé demandée.
- DeepL est un service de traduction automatique et mappe les mêmes entrées sur ses propres options. La tonalité devient
  formalité : `formal` demande un résultat plus formel, `informal` un résultat moins formel, et `neutral`
  ou une tonalité absente est omise. Un `glossaryId` configuré est passé nativement.

DeepL peut aussi signaler des notifications lorsqu'il se dégrade gracieusement. Avec une clé de niveau gratuit, une tonalité non par défaut n'est
pas prise en charge, donc il revient à la valeur par défaut et émet une notification `FORMALITY_DOWNGRADED`. Une carte de termes générique
`glossary` n'est pas appliquée par DeepL (il utilise plutôt un glossaire par id), donc il émet une
notification `GLOSSARY_IGNORED`. Ce sont des données sur le résultat, pas des erreurs. Les fournisseurs LLM n'émettent aucune
notification.

### Vérification des espaces réservés et écriture

Après la traduction, verbatra compare l'ensemble des espaces réservés de chaque valeur traduite à sa source.
Une traduction qui supprime ou altère un espace réservé échoue à cette vérification d'intégrité et est retenue plutôt
qu'écrite. Les traductions acceptées sont fusionnées dans la locale cible et écrites de façon atomique, de sorte qu'une
écriture interrompue ne laisse jamais un fichier à moitié terminé.

## Le fichier de verrouillage et le travail incrémentiel

verbatra conserve un fichier de verrouillage commité (`verbatra.lock.json`) qui enregistre, par locale, un hachage de contenu
pour chaque clé source traduite. Cet enregistrement est la base de référence par rapport à laquelle l'exécution suivante diffuse. Après qu'une
locale est écrite, le verrouillage est rafraîchi avec les hachages source actuels, avec une exception
délibérée : les clés qui ont été retenues lors de cette exécution (une non-concordance d'espace réservé ou un ICU invalide) conservent leur hachage
précédent, de sorte que l'exécution suivante les voit comme nécessitant encore du travail et les réessaie. Les clés orphelines n'obtiennent aucune entrée.
Un fichier de verrouillage manquant est traité comme une première exécution ; un fichier corrompu échoue avec `LOCK_FILE_INVALID` plutôt
que d'être silencieusement écrasé.

## Le résumé de l'exécution

Une exécution renvoie un résumé pour toutes les locales cibles : les locales qui ont réussi, celles qui ont échoué,
et une ventilation par locale. Chaque entrée de locale signale les clés traduites, les clés laissées inchangées,
les clés orphelines, les clés orphelines réellement élaguées lors de cette exécution (vide sauf si l'élagage est activé), toute clé source
ignorée pour ICU invalide, toute traduction retenue pour une non-concordance d'espace réservé, toute forme plurielle
générée lors de cette exécution (vide sauf si la génération de pluriels est activée), et toute notification. Les notifications couvrent à la fois les
notifications du fournisseur (telles que les cas DeepL ci-dessus) et les notifications du SDK, y compris un avertissement de catégorie de pluriel
(code `PLURAL_CATEGORIES_INCOMPLETE`) lorsqu'une langue cible a besoin de plus de catégories de pluriel CLDR que
la source n'en fournit. Une exécution à blanc produit le même type de résumé, listant les clés qui seraient traduites,
sans appeler de fournisseur ni écrire quoi que ce soit.

## Erreurs

verbatra trace une ligne claire entre deux types d'échec. Un problème sur toute l'exécution, tel qu'une configuration manquante ou
invalide, une source illisible, un format inconnu, un fournisseur qui ne peut pas être construit
(y compris une clé API manquante), ou un fichier de verrouillage corrompu, arrête l'exécution avec une erreur structurée. L'échec
d'une seule locale n'est pas fatal : il est enregistré sur l'entrée de cette locale avec un code et un message,
et le reste des locales s'exécute quand même.

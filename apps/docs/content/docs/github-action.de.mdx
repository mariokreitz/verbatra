# GitHub Action

Sollen Übersetzungen bei jedem Push laufen? verbatra bringt eine zusammengesetzte GitHub Action mit, die genau das tut. Sie führt `verbatra translate --json` in CI aus, annotiert alle Fehler, schreibt eine Job-Zusammenfassung und beendet sich mit dem Exit-Code der CLI. Du bindest sie mit `uses:` ein. Sie wird nicht auf npm veröffentlicht.

## Verfügbarkeit

Diese Action liegt im Repository, also referenzierst du sie direkt von dort mit `uses: mariokreitz/verbatra/packages/github-action@<commit-sha>`, angepinnt an einen Commit-SHA. Es gibt kein npm-Paket, keinen Eintrag im GitHub Marketplace und kein versioniertes Action-Tag. Die pfadbasierte `uses:`-Referenz, angepinnt an einen Commit-SHA, ist der einzige unterstützte Weg, sie zu nutzen.

## Verwendung

```yaml
name: translate
on:
  push:
    branches: [main]

permissions:
  contents: read

jobs:
  verbatra:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<commit-sha>
      - uses: mariokreitz/verbatra/packages/github-action@<commit-sha>
        with:
          version: 0.3.0 # pin @verbatra/cli to an exact version
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

Die Action führt die CLI aus, also liest sie den API-Schlüssel des Providers aus der Umgebung. Übergib ihn aus deinen Repository-Secrets unter der Variable, die dein Provider erwartet (siehe [Providers](/docs/providers)): `ANTHROPIC_API_KEY` im Beispiel oben, `OPENAI_API_KEY`, wenn du OpenAI nutzt, und so weiter.

## Inputs

| Input | Erforderlich | Standard | Beschreibung |
| --- | --- | --- | --- |
| `version` | ja | - | die auszuführende `@verbatra/cli`-Version, z. B. `0.3.0`. Pinne dies auf eine exakte Version an; verwende kein bewegliches Tag wie `latest` |
| `config-path` | nein | `""` | explizite Konfigurationsdatei zum Laden (bildet auf `--config` ab); leer nutzt die normale Konfigurationssuche |
| `working-directory` | nein | `""` | Verzeichnis, gegen das Konfiguration und Locale-Dateien aufgelöst werden (bildet auf `--cwd` ab) |
| `dry-run` | nein | `"false"` | auf `"true"` setzen, um zu melden, was sich ändern würde, ohne einen Provider aufzurufen oder zu schreiben (bildet auf `--dry-run` ab) |
| `node-version` | nein | `"24"` | Node.js-Version, die zum Ausführen der CLI eingerichtet wird |

Die Action führt immer `translate --json` aus. Sie führt niemals `init` oder `watch` aus.

## Exit-Verhalten

Die Action greift stdout und Exit-Code der CLI ab, ohne vorzeitig abzubrechen. Sie annotiert Fehler, schreibt die Job-Zusammenfassung und propagiert erst dann den Exit-Code der CLI. Der Schritt schlägt also bei `1` (einige Locales sind fehlgeschlagen) oder `2` (konnte nicht laufen) fehl, aber nicht bevor du die Annotationen und die Zusammenfassung bekommst. Die [CLI exit codes](/docs/cli#exit-codes) erklären, was jeder Code bedeutet.

## Übersetzungen persistieren

Ohne `dry-run` schreibt `translate` aktualisierte Locale-Dateien in den Checkout des Runners, aber die Action hört dort auf: sie committet sie nicht. Um die Änderungen zu behalten, füge einen eigenen Schritt hinzu, der committet und pusht oder einen Pull Request öffnet. Wenn du nur willst, dass CI fehlende oder veraltete Übersetzungen kennzeichnet, ohne etwas zu schreiben, setze `dry-run: "true"`.

## Sicherheit

Pinne sowohl die Action als auch die CLI auf exakte Referenzen an. Pinne `uses:` auf einen Commit-SHA an und setze den `version`-Input auf ein exaktes `@verbatra/cli`-Release statt auf ein bewegliches Tag wie `latest`. Das hält einen Lauf reproduzierbar und verhindert, dass er automatisch ein kompromittiertes Release zieht. Gib dem Workflow nur die Berechtigung, die er braucht: `contents: read` für eine Prüfung, plus `contents: write` und `pull-requests: write` nur, wenn ein späterer Schritt committet oder einen Pull Request öffnet. Inputs erreichen die CLI als Daten, niemals in die Shell eingefügt, sodass ein manipulierter Input-Wert ein Argument bleibt und niemals zu ausführbarem Code wird.

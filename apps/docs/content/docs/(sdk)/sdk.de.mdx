title: SDK-Referenz
description: Steuere verbatra aus deinen eigenen Skripten, CI-Jobs und Werkzeugen mit der @verbatra/sdk-API.
---
`@verbatra/sdk` ist die Engine, auf der der `verbatra`-Befehl läuft. Die CLI ist ein dünner Wrapper, sodass
alles, was die Kommandozeile tut, du auch im Code tun kannst: eine Konfiguration laden und validieren, eine einmalige Übersetzung
ausführen oder die Quelle überwachen und bei jeder Änderung neu übersetzen. Greif zum SDK, wenn du Übersetzung in
einem Skript, einem CI-Job oder deinem eigenen Werkzeug möchtest.

## Installation

```bash
pnpm add -D @verbatra/sdk
# npm
npm install -D @verbatra/sdk
# yarn
yarn add -D @verbatra/sdk
```

Erfordert Node.js `>=22.14.0`.

## Die öffentliche Oberfläche

Acht Funktionen, plus die Typen, die du berührst, wenn du sie aufrufst.

- `defineConfig(config): VerbatraConfig` gibt dein Argument unverändert zurück. Seine einzige Aufgabe ist
  Typinferenz und Autovervollständigung, wenn du eine Konfiguration in TypeScript schreibst. Das `model`-Feld ist auf die
  bekannten Modelle des gewählten Providers angepinnt, sodass das Modell eines anderen Providers ein Typfehler ist.
- `loadConfig(options?: LoadConfigOptions): Promise<VerbatraConfig>` findet und validiert die Konfiguration.
  `LoadConfigOptions` nimmt `cwd`, einen expliziten `configPath` oder ein In-Memory-`configOverride`. Es
  wirft einen `SdkError`, wenn keine Konfiguration zu finden ist oder die Validierung fehlschlägt.
- `translate(input: TranslateInput): Promise<RunSummary>` führt den Lesen-, Diff-, Übersetzen-, Schreiben-Ablauf
  über jede Ziel-Locale aus. `TranslateInput` ist `{ config, cwd?, dryRun?, prune?, generatePlurals? }`.
  `prune` (standardmäßig aus) entfernt verwaiste Keys aus der geschriebenen Datei und dem Lock. `generatePlurals`
  (standardmäßig aus) synthetisiert die CLDR-Pluralformen, die eine Zielsprache braucht, der Quelle aber fehlen
  (nur i18next-JSON mit einem LLM-Provider). Setze eines der Pro-Lauf-Flags, und es überschreibt die passende
  Konfigurationsoption für diesen Lauf. Du bekommst eine `RunSummary` zurück: Gesamtlauf-Probleme werfen einen `SdkError`,
  Ergebnisse pro Locale reisen in der Zusammenfassung mit.
- `watch(input: WatchInput): Promise<WatchController>` überwacht die Quelldatei und läuft bei jeder
  entprellten Änderung neu. `WatchInput` ist `{ config, cwd?, debounceMs?, onRun }`, und `onRun` feuert einmal pro
  Lauf mit einem `WatchRunResult`. Der zurückgegebene `WatchController` hat eine Methode, `stop()`, die
  den Watcher schließt und auf den laufenden Lauf wartet.
- `exportWorkbook(input: ExportWorkbookInput): Promise<ExportWorkbookResult>` schreibt die Strings, die
  noch übersetzt werden müssen, in eine gestylte Excel-Arbeitsmappe für einen menschlichen Übersetzer. `ExportWorkbookInput` ist
  `{ config, cwd?, out?, locales?, includeUnchanged? }`, und du bekommst ein `ExportWorkbookResult` zurück:
  den absoluten geschriebenen `path` plus `rows`-Anzahlen pro Locale. Kein Provider-Aufruf, keine Lock-Datei. Siehe
  [Menschliche Übersetzung](/docs/manual-translation).
- `importWorkbook(input: ImportWorkbookInput): Promise<RunSummary>` liest eine ausgefüllte Arbeitsmappe zurück in
  die Locale-Dateien und führt dieselben Platzhalter- und ICU-Prüfungen aus wie `translate`. `ImportWorkbookInput`
  ist `{ config, workbook, cwd?, dryRun? }`, und es löst zur selben `RunSummary` auf wie `translate`,
  sodass ein Fehlschlag pro Locale Daten in der Zusammenfassung ist, kein Wurf. `DEFAULT_WORKBOOK_PATH` ist der standardmäßige
  Export-Pfad (`verbatra-translations.xlsx`).
- `check(input: CheckInput): Promise<CheckSummary>` führt dasselbe Lesen und Diff aus wie `translate`, hört dann
  vor jedem Provider-Aufruf auf: es meldet die Abweichung pro Locale und schreibt nichts. `CheckInput` ist
  `{ config, cwd?, locales? }`, und seine `CheckSummary` setzt `inSync` genau dann auf true, wenn jede geprüfte
  Locale keine fehlenden und keine veralteten Keys hat. Kein Provider, kein Dateischreiben, keine Lock-Berührung. Siehe
  [`verbatra check`](/docs/cli/check). Willst du die Keys hinter den Anzahlen? Nutze `diff`, unten.
- `diff(input: DiffInput): Promise<DiffSummary>` ist das schreibgeschützte Geschwister von `check`: dasselbe Lesen und
  Diff, aber es gibt dir Key-Listen statt Anzahlen. `DiffInput` ist `{ config, cwd?, locales? }`. Die
  `DiffSummary` gibt jeder Locale `missing` (Keys, die hinzugefügt würden), `changed` (Keys, die
  neu übersetzt würden) und `orphaned` (Keys im Ziel, aber nicht in der Quelle, nur gemeldet). Sein
  `hasPendingChanges` ist genau dann true, wenn eine Locale einen fehlenden oder geänderten Key hat; verwaiste Keys
  allein lösen es nie aus. Kein Provider, kein Dateischreiben, keine Lock-Berührung. Siehe [`verbatra diff`](/docs/cli/diff).

Die Ergebnistypen:

```ts
interface RunSummary {
  dryRun: boolean;          // true when nothing was written and no provider was called
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
  usage?: UsageSummary;     // summed input/output tokens across every locale; absent if none reported usage
  budget?: RunBudget;       // present only when maxTokens is configured
}

interface LocaleSummary {
  locale: string;
  status: "succeeded" | "failed";
  translated: string[];          // keys translated this run (in dry-run, keys that would be)
  unchanged: string[];           // keys already up to date
  orphaned: string[];            // target keys with no matching source key (reported regardless of pruning)
  pruned: string[];              // orphaned keys removed this run; empty unless pruning is on (subset of orphaned)
  invalidIcuSource: string[];    // source keys skipped for invalid ICU
  integrityMismatches: string[]; // translations withheld for a placeholder mismatch, never a provider failure
  providerFailures: string[];    // keys withheld because the provider call itself failed (see the notices for why)
  budgetWithheld: string[];      // keys never sent because a maxTokens budget already tripped in "stop" mode
  generated: string[];           // CLDR plural forms synthesized this run; empty unless generation is on
  notices: LocaleNotice[];       // provider notices (e.g. DeepL degradation) and SDK notices (e.g. PLURAL_CATEGORIES_INCOMPLETE)
  needsReview: { key: string; reasons: string[] }[]; // accepted keys the review heuristics flagged for a second look, sorted by key; empty when nothing was flagged
  usage?: UsageSummary;          // summed input/output tokens for this locale; absent if nothing reported usage
  error?: { code: string; message: string }; // present only when status is "failed"
}

interface UsageSummary {
  inputTokens: number;
  outputTokens: number;
}

interface RunBudget {
  maxTokens: number;             // the configured ceiling
  behavior: "warn" | "stop";     // the configured behavior once the ceiling is reached
  supported: boolean;            // false when the configured provider never reported usage this run
  tokensUsed: number;            // cumulative input plus output tokens across the run so far
  exceeded: boolean;             // true once tokensUsed reached or passed maxTokens
}

// A notice is either a provider-emitted notice or an SDK-emitted one.
type LocaleNotice = ProviderNotice | SdkNotice;

type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };

interface CheckSummary {
  inSync: boolean;                 // true exactly when every locale is in sync
  locales: LocaleCheckSummary[];   // one entry per checked locale, in config order
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;                 // in source, absent from target
  stale: number;                   // source changed since last translated
  upToDate: number;                // target matches the current source
  inSync: boolean;                 // missing === 0 && stale === 0
}

interface DiffSummary {
  hasPendingChanges: boolean;      // true when some locale has a missing or changed key
  locales: LocaleDiff[];           // one entry per checked locale, in config order
}

interface LocaleDiff {
  locale: string;
  missing: string[];               // in source, absent from target: would be added
  changed: string[];               // source changed since last translated: would be re-translated
  orphaned: string[];              // in target, absent from source: reported only
  hasPendingChanges: boolean;      // missing.length > 0 || changed.length > 0
}
```

Hinweise leben pro Locale, auf `LocaleSummary.notices`. Sie decken Provider-Hinweise (etwa eine DeepL-
Herabstufung) und SDK-Hinweise ab (etwa eine Zielsprache, die mehr CLDR-Plural-Kategorien braucht
als die Quelle liefert, Code `PLURAL_CATEGORIES_INCOMPLETE`, oder ein überschrittenes Token-Budget, Code
`BUDGET_TOKENS_EXCEEDED`). Es gibt keine Hinweis-Liste auf oberster Ebene in
`RunSummary`, also lies sie immer von der Locale ab.

`needsReview` listet jeden akzeptierten Key, den die Review-Heuristiken für einen zweiten Blick markiert
haben, jeweils mit den zutreffenden Grund-Codes: `LENGTH_RATIO_OUTLIER`, `EQUALS_SOURCE`,
`GLOSSARY_TERM_MISSED`, `INTEGRITY_REORDERED` und `PROVIDER_DEGRADED`. Das ist ein eigenständiges, nicht
blockierendes Signal, getrennt von der Platzhalter- und ICU-Integritätssperre: ein Review-Flag hält
niemals einen Key zurück, es markiert ihn nur als sehenswert, sodass ein Key nie aus demselben Grund
sowohl in `needsReview` als auch in `integrityMismatches` erscheint. Es ist auf der von `importWorkbook`
zurückgegebenen `RunSummary` immer leer, da ein Arbeitsmappen-Import nie einen Provider aufruft oder
Flags auf seinem eigenen Pfad neu berechnet. Siehe [Übersetzungssicherheit](/docs/translation-safety) dafür,
was jeder Code bedeutet, und [Menschliche Übersetzung](/docs/manual-translation) für dasselbe Signal in den
Arbeitsmappen-Spalten `Review status` / `Review reasons`.

`usage` ist `undefined`, niemals ein erfundenes `{ inputTokens: 0, outputTokens: 0 }`, wann immer nichts
in diesem Bereich Nutzung gemeldet hat: Ein Dry-Run ruft nie einen Provider auf, und ein token-loser
Provider (DeepL) meldet nie eine. `RunSummary.usage` ist die Summe der `LocaleSummary.usage` jeder Locale.
`RunSummary.budget` ist nur vorhanden, wenn `maxTokens` konfiguriert ist; gegenüber einem token-losen
Provider oder einem Dry-Run ist es dennoch vorhanden, mit `supported: false`, `tokensUsed: 0` und
`exceeded: false`, sodass die Schutzfunktion sichtbar und ehrlich inaktiv ist, statt stillschweigend
wegzufallen oder fälschlich auszulösen.

`SdkError` ist der eine Fehlertyp, den das SDK für Gesamtlauf-Fehler wirft. Er trägt einen stabilen `code`
und eine geheimnisfreie `message`. Die Codes:

| Code | Wann |
| --- | --- |
| `CONFIG_NOT_FOUND` | keine Konfiguration wurde gefunden, oder ein expliziter `configPath` existiert nicht |
| `CONFIG_INVALID` | eine Konfiguration wurde gefunden, ist aber nicht parsbar oder besteht die Validierung nicht |
| `UNKNOWN_FORMAT` | für das konfigurierte Format ist kein Adapter registriert |
| `PROVIDER_CONSTRUCTION_FAILED` | der Provider konnte nicht gebaut werden (einschließlich eines fehlenden API-Schlüssels) |
| `SOURCE_UNREADABLE` | die Quell-Locale-Datei existiert nicht |
| `SOURCE_INVALID` | die Quell-Locale-Datei konnte nicht gelesen oder geparst werden |
| `LOCK_FILE_INVALID` | die Lock-Datei ist vorhanden, aber korrupt oder zu groß |
| `LOCALE_FAILED` | nicht geworfen: der Fallback-`code`, der auf dem `error` einer fehlgeschlagenen Locale aufgezeichnet wird |

Eine einzelne fehlschlagende Locale wirft nie. Sie landet auf der `LocaleSummary` dieser Locale mit `status:
"failed"` und einem `error`, und der Lauf läuft weiter.

## Eine Konfiguration definieren

Verfasse die Konfiguration in TypeScript und lass `defineConfig` sie typprüfen, während du schreibst.

```ts
import { defineConfig } from "@verbatra/sdk";

export default defineConfig({
  sourceLocale: "en",
  targetLocales: ["de", "fr"],
  format: "i18next-json",
  files: {
    pattern: "locales/{locale}.json",
  },
  provider: {
    id: "anthropic",
    options: {
      model: "claude-sonnet-4-6",
      maxTokens: 4096,
    },
  },
});
```

## Eine einmalige Übersetzung ausführen

`translate` löst zur Lauf-Zusammenfassung auf. Lies `succeeded` und `failed` für die Schlagzeile, dann durchlaufe
`locales` für das Detail und etwaige Hinweise (Provider oder SDK).

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig();
const summary = await translate({ config });

console.log(`${summary.succeeded.length} succeeded, ${summary.failed.length} failed`);

for (const locale of summary.locales) {
  for (const notice of locale.notices) {
    console.warn(`${locale.locale}: ${notice.code} ${notice.message}`);
  }
}
```

Übergib `dryRun: true`, um zu lesen und zu diffen, ohne einen Provider aufzurufen oder Dateien zu schreiben. Übergib `prune: true`,
um verwaiste Keys zu entfernen, oder `generatePlurals: true`, um fehlende Plural-Kategorien zu synthetisieren
(nur i18next-JSON mit einem LLM-Provider). Jedes davon überschreibt die passende Konfigurationsoption für diesen
Lauf. `maxBatchSize` ist die Ausnahme: es ist nur über die Konfiguration möglich (auf dem Konfigurationsobjekt gesetzt, Standard 50),
ohne eine Pro-Lauf-`translate()`-Überschreibung.

## Watch-Modus

`watch` feuert dein `onRun` einmal pro Lauf mit einem diskriminierten `WatchRunResult`, und das `stop()` des Controllers
fährt alles sauber herunter.

```ts
import { loadConfig, watch } from "@verbatra/sdk";

const config = await loadConfig();
const controller = await watch({
  config,
  onRun: (result) => {
    if (result.status === "succeeded") {
      console.log(`${result.summary.succeeded.length} ok, ${result.summary.failed.length} failed`);
    } else {
      console.error(`run failed: ${result.error.code} ${result.error.message}`);
    }
  },
});

// Stop cleanly on Ctrl-C.
process.on("SIGINT", () => {
  void controller.stop();
});
```

## Fehler behandeln

Gesamtlauf-Fehler werfen einen `SdkError`. Schalte auf seinen `code`, um jeden Fall zu seinen eigenen Bedingungen zu behandeln.

```ts
import { loadConfig, SdkError, translate } from "@verbatra/sdk";

try {
  const config = await loadConfig();
  await translate({ config });
} catch (error) {
  if (error instanceof SdkError) {
    switch (error.code) {
      case "CONFIG_NOT_FOUND":
        console.error("No verbatra config found. Run `verbatra init` to scaffold one.");
        break;
      case "SOURCE_UNREADABLE":
      case "SOURCE_INVALID":
        console.error(`Could not read the source locale: ${error.message}`);
        break;
      default:
        console.error(`${error.code}: ${error.message}`);
    }
  } else {
    throw error;
  }
}
```

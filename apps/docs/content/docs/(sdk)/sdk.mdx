title: SDK reference
description: Drive verbatra from your own scripts, CI jobs, and tooling with the @verbatra/sdk API.
---
`@verbatra/sdk` is the engine the `verbatra` command runs on. The CLI is a thin wrapper, so whatever
the command line does, you can do in code: load and validate a config, run a one-shot translation, or
watch the source and re-translate on every change. Reach for the SDK when you want translation inside
a script, a CI job, or your own tooling.

## Install

```bash
pnpm add -D @verbatra/sdk
# npm
npm install -D @verbatra/sdk
# yarn
yarn add -D @verbatra/sdk
```

Requires Node.js `>=22.14.0`.

## The public surface

Eight functions, plus the types you touch when you call them.

- `defineConfig(config): VerbatraConfig` hands your argument straight back, unchanged. Its only job is
  type inference and autocomplete when you write a config in TypeScript. The `model` field is pinned
  to the selected provider's known models, so reaching for another provider's model is a type error.
- `loadConfig(options?: LoadConfigOptions): Promise<VerbatraConfig>` finds and validates the config.
  `LoadConfigOptions` takes `cwd`, an explicit `configPath`, or an in-memory `configOverride`. It
  throws an `SdkError` when there is no config to find or validation fails.
- `translate(input: TranslateInput): Promise<RunSummary>` runs the read, diff, translate, write flow
  across every target locale. `TranslateInput` is `{ config, cwd?, dryRun?, prune?, generatePlurals? }`.
  `prune` (default off) drops orphaned keys from the written file and the lock. `generatePlurals`
  (default off) synthesizes the CLDR plural forms a target language needs but the source lacks
  (i18next-JSON with an LLM provider only). Set either per-run flag and it overrides the matching
  config option for that run. You get back a `RunSummary`: whole-run problems throw an `SdkError`,
  per-locale outcomes ride along on the summary.
- `watch(input: WatchInput): Promise<WatchController>` watches the source file and re-runs on every
  debounced change. `WatchInput` is `{ config, cwd?, debounceMs?, onRun }`, and `onRun` fires once per
  run with a `WatchRunResult`. The returned `WatchController` has one method, `stop()`, which closes
  the watcher and waits for the in-flight run.
- `exportWorkbook(input: ExportWorkbookInput): Promise<ExportWorkbookResult>` writes the strings that
  still need translating into a styled Excel workbook for a human translator. `ExportWorkbookInput` is
  `{ config, cwd?, out?, locales?, includeUnchanged? }`, and you get back an `ExportWorkbookResult`:
  the absolute `path` written plus per-locale `rows` counts. No provider call, no lock file. See
  [Human translation](/docs/manual-translation).
- `importWorkbook(input: ImportWorkbookInput): Promise<RunSummary>` reads a filled workbook back into
  the locale files, running the same placeholder and ICU checks as `translate`. `ImportWorkbookInput`
  is `{ config, workbook, cwd?, dryRun? }`, and it resolves to the same `RunSummary` as `translate`,
  so a per-locale failure is data on the summary, not a throw. `DEFAULT_WORKBOOK_PATH` is the default
  export path (`verbatra-translations.xlsx`).
- `check(input: CheckInput): Promise<CheckSummary>` runs the same read and diff as `translate`, then
  stops short of any provider call: it reports per-locale drift and writes nothing. `CheckInput` is
  `{ config, cwd?, locales? }`, and its `CheckSummary` sets `inSync` to true exactly when every checked
  locale has no missing and no stale keys. No provider, no file write, no lock touch. See
  [`verbatra check`](/docs/cli/check). Want the keys behind the counts? Use `diff`, below.
- `diff(input: DiffInput): Promise<DiffSummary>` is the read-only sibling of `check`: same read and
  diff, but it hands you key lists instead of counts. `DiffInput` is `{ config, cwd?, locales? }`. The
  `DiffSummary` gives each locale `missing` (keys that would be added), `changed` (keys that would be
  re-translated), and `orphaned` (keys in the target but not the source, reported only). Its
  `hasPendingChanges` is true exactly when some locale has a missing or changed key; orphaned keys
  alone never trip it. No provider, no file write, no lock touch. See [`verbatra diff`](/docs/cli/diff).

The result types:

```ts
interface RunSummary {
  dryRun: boolean;          // true when nothing was written and no provider was called
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
  usage?: UsageSummary;     // summed input/output tokens across every locale; absent if none reported usage
  budget?: RunBudget;       // present only when maxTokens is configured
}

interface LocaleSummary {
  locale: string;
  status: "succeeded" | "failed";
  translated: string[];          // keys translated this run (in dry-run, keys that would be)
  unchanged: string[];           // keys already up to date
  orphaned: string[];            // target keys with no matching source key (reported regardless of pruning)
  pruned: string[];              // orphaned keys removed this run; empty unless pruning is on (subset of orphaned)
  invalidIcuSource: string[];    // source keys skipped for invalid ICU
  integrityMismatches: string[]; // translations withheld for a placeholder mismatch, never a provider failure
  providerFailures: string[];    // keys withheld because the provider call itself failed (see the notices for why)
  budgetWithheld: string[];      // keys never sent because a maxTokens budget already tripped in "stop" mode
  generated: string[];           // CLDR plural forms synthesized this run; empty unless generation is on
  notices: LocaleNotice[];       // provider notices (e.g. DeepL degradation) and SDK notices (e.g. PLURAL_CATEGORIES_INCOMPLETE)
  needsReview: { key: string; reasons: string[] }[]; // accepted keys the review heuristics flagged for a second look, sorted by key; empty when nothing was flagged
  usage?: UsageSummary;          // summed input/output tokens for this locale; absent if nothing reported usage
  error?: { code: string; message: string }; // present only when status is "failed"
}

interface UsageSummary {
  inputTokens: number;
  outputTokens: number;
}

interface RunBudget {
  maxTokens: number;             // the configured ceiling
  behavior: "warn" | "stop";     // the configured behavior once the ceiling is reached
  supported: boolean;            // false when the configured provider never reported usage this run
  tokensUsed: number;            // cumulative input plus output tokens across the run so far
  exceeded: boolean;             // true once tokensUsed reached or passed maxTokens
}

// A notice is either a provider-emitted notice or an SDK-emitted one.
type LocaleNotice = ProviderNotice | SdkNotice;

type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };

interface CheckSummary {
  inSync: boolean;                 // true exactly when every locale is in sync
  locales: LocaleCheckSummary[];   // one entry per checked locale, in config order
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;                 // in source, absent from target
  stale: number;                   // source changed since last translated
  upToDate: number;                // target matches the current source
  inSync: boolean;                 // missing === 0 && stale === 0
}

interface DiffSummary {
  hasPendingChanges: boolean;      // true when some locale has a missing or changed key
  locales: LocaleDiff[];           // one entry per checked locale, in config order
}

interface LocaleDiff {
  locale: string;
  missing: string[];               // in source, absent from target: would be added
  changed: string[];               // source changed since last translated: would be re-translated
  orphaned: string[];              // in target, absent from source: reported only
  hasPendingChanges: boolean;      // missing.length > 0 || changed.length > 0
}
```

Notices live per locale, on `LocaleSummary.notices`. They cover provider notices (say, a DeepL
graceful degradation) and SDK notices (say, a target language that needs more CLDR plural categories
than the source supplies, code `PLURAL_CATEGORIES_INCOMPLETE`, or a token budget crossed, code
`BUDGET_TOKENS_EXCEEDED`). There is no top-level notices list on `RunSummary`, so always read them off
the locale.

`needsReview` lists every accepted key the review heuristics flagged for a second look, each with the
reason codes that applied: `LENGTH_RATIO_OUTLIER`, `EQUALS_SOURCE`, `GLOSSARY_TERM_MISSED`,
`INTEGRITY_REORDERED`, and `PROVIDER_DEGRADED`. This is a distinct, non-blocking signal, separate from
the placeholder and ICU integrity gate: a review flag never withholds a key, it only marks one worth a
look, so a key never appears in both `needsReview` and `integrityMismatches` for the same reason. It is
always empty on the `RunSummary` returned by `importWorkbook`, since a workbook import never calls a
provider or recomputes flags on its own path. See [Translation safety](/docs/translation-safety) for
what each code means, and [Human translation](/docs/manual-translation) for the same signal surfaced as
the workbook's `Review status` / `Review reasons` columns.

`usage` is `undefined`, never a fabricated `{ inputTokens: 0, outputTokens: 0 }`, whenever nothing in
that scope reported usage: a dry-run never calls a provider, and a token-less provider (DeepL) never
reports one. `RunSummary.usage` is the sum of every locale's `LocaleSummary.usage`. `RunSummary.budget`
is present only when `maxTokens` is configured; against a token-less provider or a dry-run it is still
present with `supported: false`, `tokensUsed: 0`, and `exceeded: false`, so the guardrail is visibly and
honestly inert rather than silently omitted or falsely tripped.

`SdkError` is the one error type the SDK throws for whole-run failures. It carries a stable `code`
and a secret-free `message`. The codes:

| Code | When |
| --- | --- |
| `CONFIG_NOT_FOUND` | no config was found, or an explicit `configPath` does not exist |
| `CONFIG_INVALID` | a config was found but is unparseable or fails validation |
| `UNKNOWN_FORMAT` | no adapter is registered for the configured format |
| `PROVIDER_CONSTRUCTION_FAILED` | the provider could not be built (includes a missing API key) |
| `SOURCE_UNREADABLE` | the source locale file does not exist |
| `SOURCE_INVALID` | the source locale file could not be read or parsed |
| `LOCK_FILE_INVALID` | the lock file is present but corrupt or too large |
| `UNKNOWN_KEY` | the requested key does not exist in the source resource; thrown by the `retranslateEntry` seam behind [Studio's retranslate action](/docs/cli/studio) |
| `LOCALE_FAILED` | not thrown: the fallback `code` recorded on a failed locale's `error` |

A single locale failing never throws. It lands on that locale's `LocaleSummary` with `status:
"failed"` and an `error`, and the run keeps going.

## Define a config

Author the config in TypeScript and let `defineConfig` type-check it as you go.

```ts
import { defineConfig } from "@verbatra/sdk";

export default defineConfig({
  sourceLocale: "en",
  targetLocales: ["de", "fr"],
  format: "i18next-json",
  files: {
    pattern: "locales/{locale}.json",
  },
  provider: {
    id: "anthropic",
    options: {
      model: "claude-sonnet-4-6",
      maxTokens: 4096,
    },
  },
});
```

## Run a one-shot translation

`translate` resolves to the run summary. Read `succeeded` and `failed` for the headline, then walk
`locales` for the detail and any notices (provider or SDK).

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig();
const summary = await translate({ config });

console.log(`${summary.succeeded.length} succeeded, ${summary.failed.length} failed`);

for (const locale of summary.locales) {
  for (const notice of locale.notices) {
    console.warn(`${locale.locale}: ${notice.code} ${notice.message}`);
  }
}
```

Pass `dryRun: true` to read and diff without calling a provider or writing files. Pass `prune: true`
to drop orphaned keys, or `generatePlurals: true` to synthesize missing plural categories
(i18next-JSON with an LLM provider only). Each of these overrides the matching config option for that
run. `maxBatchSize` is the exception: it is config-only (set it on the config object, default 50),
with no per-run `translate()` override.

## Watch mode

`watch` fires your `onRun` once per run with a discriminated `WatchRunResult`, and the controller's
`stop()` tears everything down cleanly.

```ts
import { loadConfig, watch } from "@verbatra/sdk";

const config = await loadConfig();
const controller = await watch({
  config,
  onRun: (result) => {
    if (result.status === "succeeded") {
      console.log(`${result.summary.succeeded.length} ok, ${result.summary.failed.length} failed`);
    } else {
      console.error(`run failed: ${result.error.code} ${result.error.message}`);
    }
  },
});

// Stop cleanly on Ctrl-C.
process.on("SIGINT", () => {
  void controller.stop();
});
```

## Handle errors

Whole-run failures throw an `SdkError`. Switch on its `code` to handle each case on its own terms.

```ts
import { loadConfig, SdkError, translate } from "@verbatra/sdk";

try {
  const config = await loadConfig();
  await translate({ config });
} catch (error) {
  if (error instanceof SdkError) {
    switch (error.code) {
      case "CONFIG_NOT_FOUND":
        console.error("No verbatra config found. Run `verbatra init` to scaffold one.");
        break;
      case "SOURCE_UNREADABLE":
      case "SOURCE_INVALID":
        console.error(`Could not read the source locale: ${error.message}`);
        break;
      default:
        console.error(`${error.code}: ${error.message}`);
    }
  } else {
    throw error;
  }
}
```

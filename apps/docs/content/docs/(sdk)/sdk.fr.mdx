title: Référence du SDK
description: Pilote verbatra depuis tes propres scripts, tâches CI et outils avec l'API @verbatra/sdk.
---
`@verbatra/sdk` est le moteur sur lequel tourne la commande `verbatra`. La CLI est un mince wrapper, donc tout ce que
fait la ligne de commande, tu peux le faire en code : charger et valider une config, exécuter une traduction unique, ou
surveiller la source et retraduire à chaque changement. Tourne-toi vers le SDK quand tu veux de la traduction à l'intérieur
d'un script, d'une tâche CI ou de tes propres outils.

## Installation

```bash
pnpm add -D @verbatra/sdk
# npm
npm install -D @verbatra/sdk
# yarn
yarn add -D @verbatra/sdk
```

Nécessite Node.js `>=22.14.0`.

## La surface publique

Huit fonctions, plus les types que tu manipules quand tu les appelles.

- `defineConfig(config): VerbatraConfig` te renvoie ton argument tel quel, inchangé. Son seul rôle est
  l'inférence de type et l'autocomplétion quand tu rédiges une config en TypeScript. Le champ `model` est restreint
  aux modèles connus du provider choisi, de sorte que le modèle d'un autre provider est une erreur de type.
- `loadConfig(options?: LoadConfigOptions): Promise<VerbatraConfig>` découvre et valide la config.
  `LoadConfigOptions` accepte `cwd`, un `configPath` explicite, ou un `configOverride` en mémoire. Elle
  lève une `SdkError` quand il n'y a aucune config à trouver ou que la validation échoue.
- `translate(input: TranslateInput): Promise<RunSummary>` exécute le flux lire, diffuser, traduire, écrire
  sur chaque locale cible. `TranslateInput` est `{ config, cwd?, dryRun?, prune?, generatePlurals? }`.
  `prune` (désactivé par défaut) supprime les clés orphelines du fichier écrit et du verrouillage. `generatePlurals`
  (désactivé par défaut) synthétise les formes plurielles CLDR dont une langue cible a besoin mais que la source ne fournit pas
  (i18next-JSON avec un provider LLM uniquement). Définis l'un ou l'autre de ces flags par exécution et il remplace l'option
  de config correspondante pour cette exécution. Tu récupères un `RunSummary` : les problèmes sur toute l'exécution lèvent une `SdkError`,
  les résultats par locale sont transportés sur le résumé.
- `watch(input: WatchInput): Promise<WatchController>` surveille le fichier source et se réexécute à chaque
  changement débondi. `WatchInput` est `{ config, cwd?, debounceMs?, onRun }`, et `onRun` se déclenche une fois par
  exécution avec un `WatchRunResult`. Le `WatchController` renvoyé a une seule méthode, `stop()`, qui ferme
  le surveillant et attend l'exécution en cours.
- `exportWorkbook(input: ExportWorkbookInput): Promise<ExportWorkbookResult>` écrit les chaînes qui
  ont encore besoin d'être traduites dans un classeur Excel stylisé pour un traducteur humain. `ExportWorkbookInput` est
  `{ config, cwd?, out?, locales?, includeUnchanged? }`, et tu récupères un `ExportWorkbookResult` :
  le `path` absolu écrit plus les comptes de `rows` par locale. Aucun appel de provider, aucun fichier de verrouillage. Voir
  [Traduction humaine](/docs/manual-translation).
- `importWorkbook(input: ImportWorkbookInput): Promise<RunSummary>` relit un classeur rempli dans les
  fichiers de locale, en exécutant les mêmes vérifications de placeholders et d'ICU que `translate`. `ImportWorkbookInput`
  est `{ config, workbook, cwd?, dryRun? }`, et elle se résout en le même `RunSummary` que `translate`,
  de sorte qu'un échec par locale est une donnée sur le résumé, pas une exception. `DEFAULT_WORKBOOK_PATH` est le chemin
  d'export par défaut (`verbatra-translations.xlsx`).
- `check(input: CheckInput): Promise<CheckSummary>` exécute la même lecture et le même diff que `translate`, puis
  s'arrête avant tout appel de provider : il signale la dérive par locale et n'écrit rien. `CheckInput` est
  `{ config, cwd?, locales? }`, et son `CheckSummary` définit `inSync` à vrai exactement quand chaque locale vérifiée
  n'a aucune clé manquante ni obsolète. Aucun provider, aucune écriture de fichier, aucune touche au verrouillage. Voir
  [`verbatra check`](/docs/cli/check). Tu veux les clés derrière les décomptes ? Utilise `diff`, ci-dessous.
- `diff(input: DiffInput): Promise<DiffSummary>` est la fonction sœur en lecture seule de `check` : même lecture et
  même diff, mais elle te donne des listes de clés au lieu de décomptes. `DiffInput` est `{ config, cwd?, locales? }`. Le
  `DiffSummary` donne à chaque locale `missing` (clés qui seraient ajoutées), `changed` (clés qui seraient
  retraduites), et `orphaned` (clés dans la cible mais pas dans la source, seulement signalées). Son
  `hasPendingChanges` est vrai exactement quand une locale a une clé manquante ou modifiée ; les clés orphelines
  seules ne le déclenchent jamais. Aucun provider, aucune écriture de fichier, aucune touche au verrouillage. Voir [`verbatra diff`](/docs/cli/diff).

Les types de résultat :

```ts
interface RunSummary {
  dryRun: boolean;          // true when nothing was written and no provider was called
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
  usage?: UsageSummary;     // summed input/output tokens across every locale; absent if none reported usage
  budget?: RunBudget;       // present only when maxTokens is configured
}

interface LocaleSummary {
  locale: string;
  status: "succeeded" | "failed";
  translated: string[];          // keys translated this run (in dry-run, keys that would be)
  unchanged: string[];           // keys already up to date
  orphaned: string[];            // target keys with no matching source key (reported regardless of pruning)
  pruned: string[];              // orphaned keys removed this run; empty unless pruning is on (subset of orphaned)
  invalidIcuSource: string[];    // source keys skipped for invalid ICU
  integrityMismatches: string[]; // translations withheld for a placeholder mismatch, never a provider failure
  providerFailures: string[];    // keys withheld because the provider call itself failed (see the notices for why)
  budgetWithheld: string[];      // keys never sent because a maxTokens budget already tripped in "stop" mode
  generated: string[];           // CLDR plural forms synthesized this run; empty unless generation is on
  notices: LocaleNotice[];       // provider notices (e.g. DeepL degradation) and SDK notices (e.g. PLURAL_CATEGORIES_INCOMPLETE)
  needsReview: { key: string; reasons: string[] }[]; // accepted keys the review heuristics flagged for a second look, sorted by key; empty when nothing was flagged
  usage?: UsageSummary;          // summed input/output tokens for this locale; absent if nothing reported usage
  error?: { code: string; message: string }; // present only when status is "failed"
}

interface UsageSummary {
  inputTokens: number;
  outputTokens: number;
}

interface RunBudget {
  maxTokens: number;             // the configured ceiling
  behavior: "warn" | "stop";     // the configured behavior once the ceiling is reached
  supported: boolean;            // false when the configured provider never reported usage this run
  tokensUsed: number;            // cumulative input plus output tokens across the run so far
  exceeded: boolean;             // true once tokensUsed reached or passed maxTokens
}

// A notice is either a provider-emitted notice or an SDK-emitted one.
type LocaleNotice = ProviderNotice | SdkNotice;

type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };

interface CheckSummary {
  inSync: boolean;                 // true exactly when every locale is in sync
  locales: LocaleCheckSummary[];   // one entry per checked locale, in config order
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;                 // in source, absent from target
  stale: number;                   // source changed since last translated
  upToDate: number;                // target matches the current source
  inSync: boolean;                 // missing === 0 && stale === 0
}

interface DiffSummary {
  hasPendingChanges: boolean;      // true when some locale has a missing or changed key
  locales: LocaleDiff[];           // one entry per checked locale, in config order
}

interface LocaleDiff {
  locale: string;
  missing: string[];               // in source, absent from target: would be added
  changed: string[];               // source changed since last translated: would be re-translated
  orphaned: string[];              // in target, absent from source: reported only
  hasPendingChanges: boolean;      // missing.length > 0 || changed.length > 0
}
```

Les notifications sont par locale, sur `LocaleSummary.notices`. Elles couvrent les notifications du provider (par exemple une
dégradation gracieuse de DeepL) et les notifications du SDK (par exemple une langue cible qui a besoin de plus de catégories de pluriel CLDR
que la source n'en fournit, code `PLURAL_CATEGORIES_INCOMPLETE`, ou un budget de tokens dépassé, code
`BUDGET_TOKENS_EXCEEDED`). Il n'y a pas de liste de notifications de niveau supérieur sur
`RunSummary`, donc lis-les toujours sur la locale.

`needsReview` liste chaque clé acceptée que les heuristiques de révision ont signalée pour un second
regard, chacune avec les codes de motif qui s'appliquent : `LENGTH_RATIO_OUTLIER`, `EQUALS_SOURCE`,
`GLOSSARY_TERM_MISSED`, `INTEGRITY_REORDERED` et `PROVIDER_DEGRADED`. C'est un signal distinct et non
bloquant, séparé du verrou d'intégrité des placeholders et de l'ICU : un signal de révision ne retient
jamais une clé, il la signale seulement comme digne d'être regardée, de sorte qu'une clé n'apparaît
jamais à la fois dans `needsReview` et dans `integrityMismatches` pour le même motif. Il est toujours
vide sur la `RunSummary` renvoyée par `importWorkbook`, puisqu'un import de classeur n'appelle jamais de
provider ni ne recalcule de signaux sur son propre chemin. Voir [Sûreté de la traduction](/docs/translation-safety)
pour ce que signifie chaque code, et [Traduction humaine](/docs/manual-translation) pour le même signal
dans les colonnes `Review status` / `Review reasons` du classeur.

`usage` vaut `undefined`, jamais un `{ inputTokens: 0, outputTokens: 0 }` inventé, dès que rien dans cette
portée n'a signalé d'usage : un dry-run n'appelle jamais de provider, et un provider sans tokens (DeepL)
n'en signale jamais. `RunSummary.usage` est la somme du `LocaleSummary.usage` de chaque locale.
`RunSummary.budget` n'est présent que quand `maxTokens` est configuré ; face à un provider sans tokens ou
à un dry-run, il est tout de même présent, avec `supported: false`, `tokensUsed: 0` et `exceeded: false`,
de sorte que la protection reste visiblement et honnêtement inactive plutôt que d'être omise en silence
ou de se déclencher faussement.

`SdkError` est le type d'erreur unique que le SDK lève pour les échecs sur toute l'exécution. Il porte un `code` stable
et un `message` sans secret. Les codes :

| Code | Quand |
| --- | --- |
| `CONFIG_NOT_FOUND` | aucune config n'a été trouvée, ou un `configPath` explicite n'existe pas |
| `CONFIG_INVALID` | une config a été trouvée mais est non analysable ou échoue à la validation |
| `UNKNOWN_FORMAT` | aucun adaptateur n'est enregistré pour le format configuré |
| `PROVIDER_CONSTRUCTION_FAILED` | le provider n'a pas pu être construit (inclut une clé API manquante) |
| `SOURCE_UNREADABLE` | le fichier de locale source n'existe pas |
| `SOURCE_INVALID` | le fichier de locale source n'a pas pu être lu ou analysé |
| `LOCK_FILE_INVALID` | le fichier de verrouillage est présent mais corrompu ou trop volumineux |
| `LOCALE_FAILED` | non levé : le `code` de repli enregistré sur l'`error` d'une locale échouée |

L'échec d'une seule locale ne lève jamais d'erreur. Il atterrit sur le `LocaleSummary` de cette locale avec `status:
"failed"` et une `error`, et l'exécution continue.

## Définir une config

Rédige la config en TypeScript et laisse `defineConfig` la vérifier au fur et à mesure.

```ts
import { defineConfig } from "@verbatra/sdk";

export default defineConfig({
  sourceLocale: "en",
  targetLocales: ["de", "fr"],
  format: "i18next-json",
  files: {
    pattern: "locales/{locale}.json",
  },
  provider: {
    id: "anthropic",
    options: {
      model: "claude-sonnet-4-6",
      maxTokens: 4096,
    },
  },
});
```

## Exécuter une traduction unique

`translate` se résout en le résumé de l'exécution. Lis `succeeded` et `failed` pour l'essentiel, puis parcours
`locales` pour le détail et toute notification (du provider ou du SDK).

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig();
const summary = await translate({ config });

console.log(`${summary.succeeded.length} succeeded, ${summary.failed.length} failed`);

for (const locale of summary.locales) {
  for (const notice of locale.notices) {
    console.warn(`${locale.locale}: ${notice.code} ${notice.message}`);
  }
}
```

Passe `dryRun: true` pour lire et diffuser sans appeler de provider ni écrire de fichiers. Passe `prune: true`
pour supprimer les clés orphelines, ou `generatePlurals: true` pour synthétiser les catégories de pluriel manquantes
(i18next-JSON avec un provider LLM uniquement). Chacun de ces flags remplace l'option de config correspondante pour cette
exécution. `maxBatchSize` est l'exception : elle est réservée à la config (définis-la sur l'objet de config, par défaut 50),
sans surcharge par exécution dans `translate()`.

## Mode surveillance

`watch` déclenche ton `onRun` une fois par exécution avec un `WatchRunResult` discriminé, et le `stop()` du contrôleur
démonte tout proprement.

```ts
import { loadConfig, watch } from "@verbatra/sdk";

const config = await loadConfig();
const controller = await watch({
  config,
  onRun: (result) => {
    if (result.status === "succeeded") {
      console.log(`${result.summary.succeeded.length} ok, ${result.summary.failed.length} failed`);
    } else {
      console.error(`run failed: ${result.error.code} ${result.error.message}`);
    }
  },
});

// Stop cleanly on Ctrl-C.
process.on("SIGINT", () => {
  void controller.stop();
});
```

## Gérer les erreurs

Les échecs sur toute l'exécution lèvent une `SdkError`. Branche-toi sur son `code` pour gérer chaque cas à sa façon.

```ts
import { loadConfig, SdkError, translate } from "@verbatra/sdk";

try {
  const config = await loadConfig();
  await translate({ config });
} catch (error) {
  if (error instanceof SdkError) {
    switch (error.code) {
      case "CONFIG_NOT_FOUND":
        console.error("No verbatra config found. Run `verbatra init` to scaffold one.");
        break;
      case "SOURCE_UNREADABLE":
      case "SOURCE_INVALID":
        console.error(`Could not read the source locale: ${error.message}`);
        break;
      default:
        console.error(`${error.code}: ${error.message}`);
    }
  } else {
    throw error;
  }
}
```

---
title: Referencia del SDK
description: Usa la API de @verbatra/sdk para ejecutar verbatra desde tus propios scripts, trabajos de CI y herramientas.
---
`@verbatra/sdk` es el motor sobre el que corre el comando `verbatra`. La CLI es un envoltorio ligero, así que
cualquier cosa que haga la línea de comandos, puedes hacerla en código: cargar y validar una configuración, ejecutar
una traducción única, u observar el origen y volver a traducir con cada cambio. Recurre al SDK cuando quieras
traducción dentro de un script, un trabajo de CI o tus propias herramientas.

## Instalación

```bash
pnpm add -D @verbatra/sdk
# npm
npm install -D @verbatra/sdk
# yarn
yarn add -D @verbatra/sdk
```

Requiere Node.js `>=22.14.0`.

## La superficie pública

Ocho funciones, además de los tipos que tocas cuando las llamas.

- `defineConfig(config): VerbatraConfig` te devuelve tu argumento tal cual, sin cambios. Su único trabajo es
  la inferencia de tipos y el autocompletado cuando escribes una configuración en TypeScript. El campo `model` está
  fijado a los modelos conocidos del proveedor seleccionado, así que recurrir al modelo de otro proveedor es un error de tipo.
- `loadConfig(options?: LoadConfigOptions): Promise<VerbatraConfig>` encuentra y valida la configuración.
  `LoadConfigOptions` acepta `cwd`, un `configPath` explícito o un `configOverride` en memoria. Lanza un
  `SdkError` cuando no hay ninguna configuración que encontrar o la validación falla.
- `translate(input: TranslateInput): Promise<RunSummary>` ejecuta el flujo de leer, comparar, traducir y escribir
  sobre cada locale de destino. `TranslateInput` es `{ config, cwd?, dryRun?, prune?, generatePlurals? }`.
  `prune` (desactivado por defecto) elimina las claves huérfanas del archivo escrito y del bloqueo. `generatePlurals`
  (desactivado por defecto) sintetiza las formas plurales CLDR que una lengua de destino necesita pero el origen no tiene
  (solo i18next-JSON con un proveedor de LLM). Establece cualquiera de las banderas por ejecución y anula la opción de
  configuración correspondiente para esa ejecución. Recibes un `RunSummary`: los problemas de toda la ejecución lanzan un
  `SdkError`, los resultados por locale van en el resumen.
- `watch(input: WatchInput): Promise<WatchController>` observa el archivo de origen y se vuelve a ejecutar con cada
  cambio con anti-rebote. `WatchInput` es `{ config, cwd?, debounceMs?, onRun }`, y `onRun` se dispara una vez por ejecución
  con un `WatchRunResult`. El `WatchController` devuelto tiene un único método, `stop()`, que cierra el observador
  y espera la ejecución en curso.
- `exportWorkbook(input: ExportWorkbookInput): Promise<ExportWorkbookResult>` escribe las cadenas que
  aún necesitan traducción en un libro de Excel con estilo para un traductor humano. `ExportWorkbookInput` es
  `{ config, cwd?, out?, locales?, includeUnchanged? }`, y recibes un `ExportWorkbookResult`:
  la `path` absoluta escrita más los recuentos de `rows` por locale. Sin llamada al proveedor, sin archivo de bloqueo. Consulta
  [Traducción humana](/docs/manual-translation).
- `importWorkbook(input: ImportWorkbookInput): Promise<RunSummary>` lee un libro rellenado de vuelta a
  los archivos de locale, ejecutando las mismas comprobaciones de marcadores de posición e ICU que `translate`. `ImportWorkbookInput`
  es `{ config, workbook, cwd?, dryRun? }`, y se resuelve al mismo `RunSummary` que `translate`,
  así que un fallo por locale es un dato en el resumen, no un throw. `DEFAULT_WORKBOOK_PATH` es la ruta
  de exportación por defecto (`verbatra-translations.xlsx`).
- `check(input: CheckInput): Promise<CheckSummary>` ejecuta la misma lectura y el mismo diff que `translate`, luego
  se detiene antes de cualquier llamada al proveedor: informa la desviación por locale y no escribe nada. `CheckInput` es
  `{ config, cwd?, locales? }`, y su `CheckSummary` pone `inSync` en true exactamente cuando cada locale
  comprobado no tiene claves faltantes ni desactualizadas. Sin proveedor, sin escritura de archivos, sin tocar el bloqueo. Consulta
  [`verbatra check`](/docs/cli/check). ¿Quieres las claves detrás de los recuentos? Usa `diff`, abajo.
- `diff(input: DiffInput): Promise<DiffSummary>` es el hermano de solo lectura de `check`: la misma lectura y el mismo
  diff, pero te da listas de claves en lugar de recuentos. `DiffInput` es `{ config, cwd?, locales? }`. El
  `DiffSummary` da a cada locale `missing` (claves que se añadirían), `changed` (claves que se
  volverían a traducir) y `orphaned` (claves en el destino pero no en el origen, solo informadas). Su
  `hasPendingChanges` es true exactamente cuando algún locale tiene una clave faltante o modificada; las claves huérfanas
  por sí solas nunca lo activan. Sin proveedor, sin escritura de archivos, sin tocar el bloqueo. Consulta [`verbatra diff`](/docs/cli/diff).

Los tipos de resultado:

```ts
interface RunSummary {
  dryRun: boolean;          // true when nothing was written and no provider was called
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
  usage?: UsageSummary;     // summed input/output tokens across every locale; absent if none reported usage
  budget?: RunBudget;       // present only when maxTokens is configured
}

interface LocaleSummary {
  locale: string;
  status: "succeeded" | "failed";
  translated: string[];          // keys translated this run (in dry-run, keys that would be)
  unchanged: string[];           // keys already up to date
  orphaned: string[];            // target keys with no matching source key (reported regardless of pruning)
  pruned: string[];              // orphaned keys removed this run; empty unless pruning is on (subset of orphaned)
  invalidIcuSource: string[];    // source keys skipped for invalid ICU
  integrityMismatches: string[]; // translations withheld for a placeholder mismatch, never a provider failure
  providerFailures: string[];    // keys withheld because the provider call itself failed (see the notices for why)
  budgetWithheld: string[];      // keys never sent because a maxTokens budget already tripped in "stop" mode
  generated: string[];           // CLDR plural forms synthesized this run; empty unless generation is on
  notices: LocaleNotice[];       // provider notices (e.g. DeepL degradation) and SDK notices (e.g. PLURAL_CATEGORIES_INCOMPLETE)
  needsReview: { key: string; reasons: string[] }[]; // accepted keys the review heuristics flagged for a second look, sorted by key; empty when nothing was flagged
  usage?: UsageSummary;          // summed input/output tokens for this locale; absent if nothing reported usage
  error?: { code: string; message: string }; // present only when status is "failed"
}

interface UsageSummary {
  inputTokens: number;
  outputTokens: number;
}

interface RunBudget {
  maxTokens: number;             // the configured ceiling
  behavior: "warn" | "stop";     // the configured behavior once the ceiling is reached
  supported: boolean;            // false when the configured provider never reported usage this run
  tokensUsed: number;            // cumulative input plus output tokens across the run so far
  exceeded: boolean;             // true once tokensUsed reached or passed maxTokens
}

// A notice is either a provider-emitted notice or an SDK-emitted one.
type LocaleNotice = ProviderNotice | SdkNotice;

type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };

interface CheckSummary {
  inSync: boolean;                 // true exactly when every locale is in sync
  locales: LocaleCheckSummary[];   // one entry per checked locale, in config order
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;                 // in source, absent from target
  stale: number;                   // source changed since last translated
  upToDate: number;                // target matches the current source
  inSync: boolean;                 // missing === 0 && stale === 0
}

interface DiffSummary {
  hasPendingChanges: boolean;      // true when some locale has a missing or changed key
  locales: LocaleDiff[];           // one entry per checked locale, in config order
}

interface LocaleDiff {
  locale: string;
  missing: string[];               // in source, absent from target: would be added
  changed: string[];               // source changed since last translated: would be re-translated
  orphaned: string[];              // in target, absent from source: reported only
  hasPendingChanges: boolean;      // missing.length > 0 || changed.length > 0
}
```

Los avisos viven por locale, en `LocaleSummary.notices`. Cubren avisos del proveedor (por ejemplo, una
degradación elegante de DeepL) y avisos del SDK (por ejemplo, una lengua de destino que necesita más categorías plurales
CLDR de las que aporta el origen, código `PLURAL_CATEGORIES_INCOMPLETE`, o un presupuesto de tokens superado,
código `BUDGET_TOKENS_EXCEEDED`). No hay una lista de avisos de nivel superior en
`RunSummary`, así que léelos siempre desde el locale.

`needsReview` lista cada clave aceptada que las heurísticas de revisión marcaron para una segunda
mirada, cada una con los códigos de motivo que aplicaron: `LENGTH_RATIO_OUTLIER`, `EQUALS_SOURCE`,
`GLOSSARY_TERM_MISSED`, `INTEGRITY_REORDERED` y `PROVIDER_DEGRADED`. Es una señal distinta y no
bloqueante, separada de la puerta de integridad de marcadores de posición e ICU: una señal de revisión
nunca retiene una clave, solo la marca como digna de mirar, así que una clave nunca aparece a la vez en
`needsReview` y en `integrityMismatches` por el mismo motivo. Siempre está vacía en el `RunSummary` que
devuelve `importWorkbook`, ya que una importación de libro nunca llama a un proveedor ni recalcula
señales en su propio camino. Consulta [Seguridad de la traducción](/docs/translation-safety) para saber
qué significa cada código, y [Traducción humana](/docs/manual-translation) para la misma señal en las
columnas `Review status` / `Review reasons` del libro.

`usage` es `undefined`, nunca un `{ inputTokens: 0, outputTokens: 0 }` inventado, siempre que nada en ese
ámbito haya informado uso: una ejecución en seco nunca llama a un proveedor, y un proveedor sin tokens
(DeepL) nunca informa uno. `RunSummary.usage` es la suma del `LocaleSummary.usage` de cada locale.
`RunSummary.budget` solo está presente cuando `maxTokens` está configurado; frente a un proveedor sin
tokens o a una ejecución en seco, sigue presente con `supported: false`, `tokensUsed: 0` y
`exceeded: false`, de modo que la salvaguarda se muestra de forma visible y honesta como inactiva, en
lugar de omitirse en silencio o dispararse falsamente.

`SdkError` es el único tipo de error que el SDK lanza para los fallos de toda la ejecución. Lleva un `code` estable
y un `message` sin secretos. Los códigos:

| Código | Cuándo |
| --- | --- |
| `CONFIG_NOT_FOUND` | no se encontró ninguna configuración, o un `configPath` explícito no existe |
| `CONFIG_INVALID` | se encontró una configuración pero no se puede analizar o falla la validación |
| `UNKNOWN_FORMAT` | no hay ningún adaptador registrado para el formato configurado |
| `PROVIDER_CONSTRUCTION_FAILED` | el proveedor no se pudo construir (incluye una clave de API faltante) |
| `SOURCE_UNREADABLE` | el archivo de locale de origen no existe |
| `SOURCE_INVALID` | el archivo de locale de origen no se pudo leer o analizar |
| `LOCK_FILE_INVALID` | el archivo de bloqueo está presente pero corrupto o demasiado grande |
| `UNKNOWN_KEY` | la clave solicitada no existe en el recurso de origen; lanzado por la vía `retranslateEntry` que hay detrás de la [acción de retraducción de Studio](/docs/cli/studio) |
| `LOCALE_FAILED` | no lanzado: el `code` de reserva registrado en el `error` de un locale fallido |

Que un solo locale falle nunca lanza. Cae en el `LocaleSummary` de ese locale con `status:
"failed"` y un `error`, y la ejecución sigue adelante.

## Definir una configuración

Escribe la configuración en TypeScript y deja que `defineConfig` la verifique de tipos a medida que avanzas.

```ts
import { defineConfig } from "@verbatra/sdk";

export default defineConfig({
  sourceLocale: "en",
  targetLocales: ["de", "fr"],
  format: "i18next-json",
  files: {
    pattern: "locales/{locale}.json",
  },
  provider: {
    id: "anthropic",
    options: {
      model: "claude-sonnet-4-6",
      maxTokens: 4096,
    },
  },
});
```

## Ejecutar una traducción única

`translate` se resuelve al resumen de la ejecución. Lee `succeeded` y `failed` para el titular, luego recorre
`locales` para el detalle y cualquier aviso (del proveedor o del SDK).

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig();
const summary = await translate({ config });

console.log(`${summary.succeeded.length} succeeded, ${summary.failed.length} failed`);

for (const locale of summary.locales) {
  for (const notice of locale.notices) {
    console.warn(`${locale.locale}: ${notice.code} ${notice.message}`);
  }
}
```

Pasa `dryRun: true` para leer y hacer diff sin llamar a un proveedor ni escribir archivos. Pasa `prune: true`
para eliminar las claves huérfanas, o `generatePlurals: true` para sintetizar las categorías plurales faltantes
(solo i18next-JSON con un proveedor de LLM). Cada una de estas anula la opción de configuración correspondiente para esa
ejecución. `maxBatchSize` es la excepción: solo se controla por configuración (establécela en el objeto de configuración, por
defecto 50), sin anulación por ejecución en `translate()`.

## Modo watch

`watch` dispara tu `onRun` una vez por ejecución con un `WatchRunResult` discriminado, y el `stop()` del
controlador lo desmonta todo de forma limpia.

```ts
import { loadConfig, watch } from "@verbatra/sdk";

const config = await loadConfig();
const controller = await watch({
  config,
  onRun: (result) => {
    if (result.status === "succeeded") {
      console.log(`${result.summary.succeeded.length} ok, ${result.summary.failed.length} failed`);
    } else {
      console.error(`run failed: ${result.error.code} ${result.error.message}`);
    }
  },
});

// Stop cleanly on Ctrl-C.
process.on("SIGINT", () => {
  void controller.stop();
});
```

## Manejar errores

Los fallos de toda la ejecución lanzan un `SdkError`. Conmuta sobre su `code` para manejar cada caso en sus propios términos.

```ts
import { loadConfig, SdkError, translate } from "@verbatra/sdk";

try {
  const config = await loadConfig();
  await translate({ config });
} catch (error) {
  if (error instanceof SdkError) {
    switch (error.code) {
      case "CONFIG_NOT_FOUND":
        console.error("No verbatra config found. Run `verbatra init` to scaffold one.");
        break;
      case "SOURCE_UNREADABLE":
      case "SOURCE_INVALID":
        console.error(`Could not read the source locale: ${error.message}`);
        break;
      default:
        console.error(`${error.code}: ${error.message}`);
    }
  } else {
    throw error;
  }
}
```

---
title: verbatra export
description: Exportiert unübersetzte Strings in eine gestylte Excel-Arbeitsmappe für einen menschlichen Übersetzer.
---
Schreibt die Strings, die übersetzt werden müssen, in eine gestylte Excel-Arbeitsmappe, die ein Mensch ausfüllen und dann für [`verbatra import`](/docs/cli/import) zurücksenden kann. Die Arbeitsmappe hat ein Blatt pro Ziel-Locale plus ein Anweisungsblatt. Der Übersetzer bearbeitet eine einzige Spalte; alles andere ist gesperrt.

```bash
verbatra export
```

Standardmäßig schreibt dies `verbatra-translations.xlsx` in das Arbeitsverzeichnis und enthält nur die **fehlenden und geänderten** Strings: dieselben Schlüssel, die ein automatischer `translate`-Lauf an einen Provider senden würde. Siehe [Menschliche Übersetzung](/docs/manual-translation) für den vollständigen Hin- und Rückweg.

Jede Zeile mit einer aktuellen Übersetzung trägt auch ein nur lesbares Paar `Review status` / `Review reasons`: verbatras eigene beratende Heuristiken, um eine Übersetzung zu erkennen, die einen zweiten Blick wert ist, frisch aus den aktuellen Quell- und Zielwerten zum Exportzeitpunkt neu berechnet. Sie beeinflussen nie, was `import` akzeptiert. Siehe [Menschliche Übersetzung](/docs/manual-translation) für die Spalten und [Übersetzungssicherheit](/docs/translation-safety) für die Grund-Codes.

## Optionen

| Option | Beschreibung |
| --- | --- |
| `--cwd <path>` | löst Konfigurations- und Locale-Dateien aus diesem Verzeichnis auf |
| `--config <path>` | lädt diese Konfigurationsdatei, anstatt nach einer zu suchen |
| `--out <path>` | schreibt die Arbeitsmappe an diesen Pfad (Standard `verbatra-translations.xlsx`) |
| `--locales <list>` | kommagetrennte Teilmenge der Ziel-Locales (Standard alle konfigurierten) |
| `--include-unchanged` | exportiert auch bereits aktuelle Strings (standardmäßig aus) |
| `--json` | gibt das Export-Ergebnis als JSON aus |

## Ausgabe

Ohne `--json` gibt `export` den Ausgabepfad und eine Zeile pro Locale mit dessen Zeilenanzahl aus. Mit `--json` gibt es das Export-Ergebnis (den geschriebenen Pfad und die Zeilenanzahlen pro Locale) als ein JSON-Objekt auf stdout aus. Ein Fehler des gesamten Laufs wird auf stderr geschrieben, sodass stdout sauber und unter `--json` pipebar bleibt.

## Beispiele

```bash
# write the workbook with missing and changed strings
verbatra export

# only the German and French sheets
verbatra export --locales de,fr

# include already up-to-date strings for full context
verbatra export --include-unchanged

# write to a specific path
verbatra export --out handoff/round-2.xlsx

# export a project in another directory
verbatra export --cwd packages/app

# load a specific config instead of searching for one
verbatra export --config config/verbatra.config.ts

# machine-readable result on stdout
verbatra export --json
```

## Exit-Codes

| Code | Bedeutung |
| --- | --- |
| `0` | die Arbeitsmappe wurde geschrieben |
| `2` | konnte nicht ausgeführt werden: ein Fehler des gesamten Laufs (fehlende oder ungültige Konfiguration, nicht lesbare Quelle, unbekanntes Format, eine beschädigte Lock-Datei oder eine angeforderte Locale, die nicht unter den konfigurierten Ziel-Locales ist) oder ein Nutzungsfehler (einschließlich einer leeren `--locales`-Liste) |

`export` ruft niemals einen Provider auf und benötigt keinen API-Schlüssel. Ein unbekannter oder leerer `--locales`-Wert lässt den Lauf mit Exit `2` fehlschlagen, statt still nichts zu exportieren, sodass ein Tippfehler in einem Locale-Namen abgefangen wird, statt durchzugehen. Siehe [`verbatra import`](/docs/cli/import) für das Zurücklesen der ausgefüllten Arbeitsmappe.

Verbatra Studio is a local web dashboard over your verbatra project: a live view of translation status, pending changes, flagged reviews, configuration, lock-file drift, and history. By default it never calls a provider or writes a file. Start it with `--allow-write` and it also gains a write action in the needs-review queue, editing one flagged key at a time (see "Needs-review queue" below); add `--allow-spend` as well and it also gains the ability to retranslate a failing key and to translate every pending change across all locales at once from a live-refresh toast (see "Retranslate a failing key" and "Translate pending changes from a toast" below). Start it with:

```bash
verbatra studio
```

<Callout title="Prerelease only">
Verbatra Studio is currently available only as a prerelease, not yet a stable release. To use it, install `@verbatra/cli` explicitly under the `next` tag: `pnpm add -D @verbatra/cli@next @verbatra/studio`. `@verbatra/studio` itself has not had a stable release yet, so its own `latest` tag already carries the current prerelease build, no tag needed. A plain `pnpm add -D @verbatra/cli` (no tag) resolves to the current stable release and does not include this command.
</Callout>

## Local-only, and read-only by default

Studio binds to `127.0.0.1` only: it is never reachable from another machine, and there is no option to change that. Every request is checked against a strict Host and Origin allowlist, and every page and API call requires the per-session token printed in the startup URL, compared in constant time so a wrong guess cannot be timed. Studio sends no CORS headers, so a browser tab on another origin cannot read it even if it tried. By default there is no state-changing endpoint anywhere in Studio: it never calls `translate`, never runs `watch`, and never imports or exports a workbook. Everything you see, including the needs-review queue, comes from the same read-only computations as [`verbatra check`](/docs/cli/check) and [`verbatra diff`](/docs/cli/diff), or from data a prior `translate`/`watch` run already persisted to disk. Pass `--allow-write` at startup and the needs-review queue gains its edit action, described in "Needs-review queue" below. Add `--allow-spend` as well and Studio also gains the retranslate action described in "Retranslate a failing key" below and the translate-pending action described in "Translate pending changes from a toast" below; leave `--allow-write` off and Studio stays exactly this read-only.

## Command reference

| Option | Description |
| --- | --- |
| `--cwd <path>` | resolve config and locale files from this directory |
| `--config <path>` | load this config file instead of searching for one |
| `--port <n>` | override the default Studio port (must be an integer from 1 to 65535) |
| `--allow-spend` | allow Studio to call a translation provider (also: `VERBATRA_STUDIO_ALLOW_SPEND`) |
| `--allow-write` | allow Studio to write locale files and the lock (also: `VERBATRA_STUDIO_ALLOW_WRITE`) |

`--allow-spend` and `--allow-write` both default to off. Each also reads its environment variable equivalent when the flag itself is absent (`"1"`, `"true"`, `"yes"`, or `"on"`, case-insensitive, counts as on); an explicit CLI flag always wins over the environment variable.

`verbatra studio` loads `.env.local` and `.env` exactly like every other command, then resolves your config once, before it ever touches the network. A missing or invalid config exits `2` with the same rendered error every other command uses; Studio itself is never started.

## The printed URL

Once Studio is listening, it prints one line to stdout:

```
Verbatra Studio running at http://127.0.0.1:5849/?token=<token>
```

Open that exact URL. The token in the query string bootstraps a session cookie for the browser tab; without it (or with the wrong one) every request is rejected. Nothing is ever printed before the server has successfully bound its port, so a failed startup never leaks a URL or a token.

## Restart on config change

The config is resolved once, at startup, and every view for the life of that process is built from that same resolved config. If you change `verbatra.config.ts` (a new target locale, a different provider, a different file pattern), stop Studio and start it again to pick up the change. Project files are different: Studio watches your source locale file, every target locale file, and the lock file, and pushes a live refresh signal to your open browser tab the moment one of them changes on disk (rapid successive writes are debounced into a single signal). The tab re-fetches the view you're currently looking at automatically, no reload needed. If a re-fetch fails, the view keeps showing its last good data with a subtle stale indicator instead of going blank, and tries again on the next signal. A dropped connection reconnects on its own with backoff; if Studio itself shuts down, the tab shows a full-screen notice to restart it from the terminal.

A change that adds, changes, or removes at least one key also shows a dismissable toast summarizing what changed, for example "1 added, 3 changed"; a lock-only change, or a change with no net effect on any key, shows no toast at all. See "Translate pending changes from a toast" below for the action a source-change toast can carry.

## Translate pending changes from a toast

When your source locale file changes on disk, its toast (see "Restart on config change" above) can carry a "Translate pending changes across all locales" action, gated the same way as retranslating a failing key: both `--allow-spend` and `--allow-write` (or their environment variable equivalents). Leave either flag off and the toast still appears, only without that action, not merely disabled.

```bash
verbatra studio --allow-spend --allow-write
```

Clicking it makes the exact same whole-project call [`verbatra translate`](/docs/cli/translate) itself makes: every configured target locale brought current against the source, at the same provider cost. A toast for a target locale's own content change never carries this action: `translate`'s diff mostly cannot see a target file's own content changing, so the action would either do nothing or spend on unrelated drift; use the key-scoped Retranslate action instead for a bad target value.

The toast reports the outcome: "Translated" on full success, "Failed for" followed by the locales that failed while others succeeded, or "Failed: ..." for anything else, for example the whole call itself failing. A second overlapping call, for example from another open tab, is refused immediately instead of left to wait, so you never trigger two whole-project runs at once.

## Error messages

When something goes wrong, for example the source locale file cannot be read, a target locale file fails to parse for its configured format, or your session has expired, Studio shows a specific, actionable description of what happened instead of one generic message. An error Studio does not specifically recognize still falls back to the same generic message as before; you never see a raw stack trace or an internal error object.

## The views

- **Status**: per-locale missing, stale, and up-to-date counts, the same computation as `verbatra check`.
- **Diff**: the exact keys that would be added, re-translated, or are orphaned per locale, the same computation as `verbatra diff`. Long key lists are capped in the view (with a "showing N of total" note) so a very large catalog stays responsive; filtering still searches the full list. A "Copy as review report" button turns the complete diff, including everything past the cap, into a Markdown summary on your clipboard, with a brief "Copied" confirmation, ready to paste into a pull request description or a chat message.
- **Review**: every key and locale pair the most recent `translate` or `watch` run flagged for a second look, with a label per reason. Before any run has been recorded it shows a message to run `translate` or `watch` first, never an error; once a run exists but nothing is flagged, it shows there is nothing to review. With write access, each row also gets its own actions; see "Needs-review queue" below.
- **Usage**: the most recently persisted `translate` or `watch` run's token totals and, when a budget was configured for that run, its ceiling, behavior, and whether it was reached, alongside the timestamp of that run. Before any run has been recorded it shows a message to run `translate` or `watch` first, never an error. A provider that does not report token usage shows "not reported by this provider" instead of a fabricated `0`; a budget configured against such a token-less provider still shows its ceiling but only a "not tracked for this provider" status, no behavior or reached state. A run with no budget configured shows only the token totals, with no budget section at all.
- **Overview**: a redacted projection of your resolved config (source and target locales, format, file pattern, provider id, and the safe scalar options) plus your glossary terms, if any. Free-form values that could carry a pasted secret are never included.
- **Lock drift**: whether a lock file exists, its version, and its per-locale drift against the current source and target files, without writing to it.
- **History**: the commit history for your watched locale files, read via `git log` with a bounded commit count and no `--follow`. A project with no git repository, or with git unavailable, shows history as unavailable rather than an error. A shallow clone's truncated history is also not an error: it simply shows what the clone has.

## Command palette

Press Cmd+K (Ctrl+K on Windows and Linux) anywhere in Studio to open a command palette. Type to filter its list: it always offers all seven views above as jump targets, plus, once the Diff view has loaded data during the current session, every key and locale combination currently shown there. Selecting a view switches to it. Selecting a key and locale switches to Diff and opens that key's detail view directly, the same view a manual click on that key opens. Press Esc, or click outside the palette, to close it and return focus to where you were.

Like the rest of Studio, the palette is read-only: no entry, in any state, ever triggers a network request, a file write, or a provider call. It is pure navigation.

## Key detail view

In Diff, click a changed key to open its detail view, with one row per target locale. Each row also shows a placeholder and ICU integrity indicator: green means that locale's translation still has the same placeholders as the source (things like `{name}` or `{{count}}`, depending on your format); red means a placeholder is missing or an unexpected one was added, and names the specific token so you know what to fix. A neutral, unlabeled indicator means either the format has no placeholders to check, or the check has not loaded yet, since it fetches only once you open the detail view, not eagerly. This only applies to changed keys: a missing key has no target value yet, and an orphaned key has no current source value, so neither has a second value worth comparing.

## Retranslate a failing key

Start Studio with both `--allow-spend` and `--allow-write` (or their environment variable equivalents) and the key detail view gains one write action: a "Retranslate" button next to any locale row whose integrity indicator is red. Leave either flag off and the button is absent from the dashboard, not merely disabled.

```bash
verbatra studio --allow-spend --allow-write
```

Clicking it makes one live call to your configured translation provider for exactly that key and locale, the same provider cost as retranslating one key with [`verbatra translate`](/docs/cli/translate). The candidate it gets back runs through the same placeholder and ICU integrity check before anything is written: an accepted candidate updates the target locale file and the lock entry for that single key, leaving every other key and every other locale untouched; a rejected candidate writes nothing. The button reports the outcome: "Retranslated" on success, "Rejected: placeholder mismatch" or "Rejected: invalid message syntax" when the new candidate itself fails the check, or "Failed: ..." for anything else, for example the provider call itself failing. A successful retranslate reaches your browser tab through the same live-refresh signal as a change made outside Studio.

## Needs-review queue

Start Studio with `--allow-write` (or its environment variable equivalent) and every row in the Review view gains three actions: "Edit", "Approve", and "Reject". This one needs only `--allow-write`; unlike retranslating a key above, `--allow-spend` has no effect on it, since editing a value never calls a translation provider. Leave `--allow-write` off and all three actions are absent from the Review view, not merely disabled.

```bash
verbatra studio --allow-write
```

Approve and Reject dismiss a row for the rest of the browser tab's session: neither one ever makes a network request, writes a file, or touches the lock file. A dismissed row stays dismissed across a live refresh and comes back only if you reload the page. Edit opens a dialog pre-filled with the key's current source and target value, fetched live so you always start from what's actually on disk, never a stale copy. Type a correction and click "Save": it runs through the same placeholder and ICU integrity check every write in Studio goes through, and never calls a provider. An accepted correction updates the target locale file and the lock entry for that single key, leaving every other key and every other locale untouched, and dismisses the row the same way Approve does. The dialog reports the outcome: "Saved" on success, "Rejected: placeholder mismatch" or "Rejected: invalid message syntax" when the correction itself fails the check, or "Failed: ..." for anything else.

## Installation

`@verbatra/studio` is Studio's own package. It has not had a stable release yet, so a plain install already gets the current prerelease build, no dist-tag needed. If it is not installed, `verbatra studio` exits `2` with:

```
Verbatra Studio requires @verbatra/studio. Install it with: pnpm add -D @verbatra/studio
```

## Exit codes

| Code | Meaning |
| --- | --- |
| `0` | Studio was stopped cleanly by a single interrupt (Ctrl-C) |
| `2` | could not run: a whole-run error (missing or invalid config, `@verbatra/studio` not installed, or the port is already in use), or a usage error (an out-of-range or non-integer `--port`) |
| `130` | force-stopped by a second interrupt |

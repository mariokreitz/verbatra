Verbatra Studio est un tableau de bord web local sur ton projet verbatra : une vue en direct du statut de traduction, des changements en attente, de la configuration, de la dérive du fichier de verrouillage et de l'historique. Par défaut, il n'appelle jamais de fournisseur ni n'écrit de fichier. Démarre-le avec deux flags supplémentaires et il gagne aussi une action d'écriture : retraduire une seule clé en échec (voir "Retraduire une clé en échec" plus bas). Démarre-le avec :

```bash
verbatra studio
```

<Callout title="Disponible uniquement en préversion">
Verbatra Studio n'est actuellement disponible qu'en préversion, il n'y a pas encore de version stable. Pour l'utiliser, installe `@verbatra/cli` explicitement sous le tag `next` : `pnpm add -D @verbatra/cli@next @verbatra/studio`. `@verbatra/studio` lui-même n'a pas encore eu de version stable, donc son propre tag `latest` porte déjà le build de préversion actuel, aucun tag requis. Un simple `pnpm add -D @verbatra/cli` (sans tag) résout vers la version stable actuelle et n'inclut pas cette commande.
</Callout>

## Uniquement local, et en lecture seule par défaut

Studio se lie uniquement à `127.0.0.1` : il n'est jamais accessible depuis une autre machine, et il n'y a aucune option pour changer cela. Chaque requête est vérifiée contre une liste stricte de Host et Origin autorisés, et chaque page et chaque appel API nécessitent le jeton de session affiché dans l'URL de démarrage, comparé en temps constant pour qu'une tentative incorrecte ne puisse pas être devinée par chronométrage. Studio n'envoie aucun en-tête CORS, si bien qu'un onglet de navigateur sur une autre origine ne pourrait pas le lire même en essayant. Par défaut, il n'y a aucun endpoint qui modifie l'état dans Studio : il n'appelle jamais `translate`, n'exécute jamais `watch`, et n'importe ni n'exporte jamais un classeur. Tout ce que tu vois provient des mêmes calculs en lecture seule que [`verbatra check`](/docs/cli/check) et [`verbatra diff`](/docs/cli/diff). Passe `--allow-spend` et `--allow-write` ensemble au démarrage et Studio gagne exactement une action d'écriture, décrite dans "Retraduire une clé en échec" plus bas ; laisse l'un des deux flags de côté et Studio reste tout aussi en lecture seule.

## Référence de la commande

| Option | Description |
| --- | --- |
| `--cwd <path>` | résoudre la configuration et les fichiers de locale depuis ce répertoire |
| `--config <path>` | charger ce fichier de configuration au lieu d'en rechercher un |
| `--port <n>` | remplacer le port par défaut de Studio (doit être un entier de 1 à 65535) |
| `--allow-spend` | autoriser Studio à appeler un fournisseur de traduction (aussi : `VERBATRA_STUDIO_ALLOW_SPEND`) |
| `--allow-write` | autoriser Studio à écrire les fichiers de locale et le fichier de verrouillage (aussi : `VERBATRA_STUDIO_ALLOW_WRITE`) |

`--allow-spend` et `--allow-write` sont tous deux désactivés par défaut. Chacun lit aussi sa variable d'environnement équivalente quand le flag lui-même est absent (`"1"`, `"true"`, `"yes"` ou `"on"`, sans distinction de casse, compte comme activé) ; un flag CLI explicite l'emporte toujours sur la variable d'environnement.

`verbatra studio` charge `.env.local` et `.env` exactement comme toute autre commande, puis résout ta configuration une seule fois, avant même de toucher le réseau. Une configuration manquante ou invalide quitte avec `2` et la même erreur affichée que toute autre commande ; Studio lui-même n'est alors jamais démarré.

## L'URL affichée

Une fois que Studio écoute, il affiche une ligne sur stdout :

```
Verbatra Studio running at http://127.0.0.1:5849/?token=<token>
```

Ouvre exactement cette URL. Le jeton dans la chaîne de requête amorce un cookie de session pour l'onglet du navigateur ; sans lui (ou avec le mauvais) toute requête est rejetée. Rien n'est jamais affiché avant que le serveur n'ait réussi à lier son port, si bien qu'un démarrage échoué ne divulgue jamais d'URL ni de jeton.

## Redémarrage lors d'un changement de configuration

La configuration est résolue une seule fois, au démarrage, et chaque vue, pendant toute la durée de vie de ce processus, s'appuie sur cette même configuration résolue. Si tu modifies `verbatra.config.ts` (une nouvelle locale cible, un fournisseur différent, un motif de fichier différent), arrête Studio et redémarre-le pour appliquer le changement. Les fichiers du projet, eux, sont différents : Studio surveille ton fichier de locale source, chaque fichier de locale cible et le fichier de verrouillage, et pousse un signal de rafraîchissement en direct vers ton onglet de navigateur ouvert dès que l'un d'eux change sur le disque (les écritures rapprochées successives sont regroupées en un seul signal). L'onglet recharge automatiquement la vue que tu regardes en ce moment, sans qu'il soit besoin de recharger la page. Si un rechargement échoue, la vue continue d'afficher ses dernières bonnes données avec un discret indicateur de données obsolètes plutôt que de s'effacer, et réessaie au signal suivant. Une connexion coupée se reconnecte d'elle-même avec un délai croissant ; si Studio lui-même s'arrête, l'onglet affiche un avis plein écran invitant à le redémarrer depuis le terminal.

## Messages d'erreur

Quand quelque chose ne va pas, par exemple si le fichier de locale source ne peut pas être lu, si un fichier de locale cible ne peut pas être analysé pour son format configuré, ou si ta session a expiré, Studio affiche une description précise et utile de ce qui s'est passé au lieu d'un simple message générique. Une erreur que Studio ne reconnaît pas spécifiquement continue de s'afficher avec le même message générique qu'avant ; tu ne vois jamais de trace de pile brute ni d'objet d'erreur interne.

## Les vues

- **Statut** : les décomptes de clés manquantes, obsolètes et à jour par locale, le même calcul que `verbatra check`.
- **Diff** : les clés exactes qui seraient ajoutées, retraduites ou orphelines par locale, le même calcul que `verbatra diff`. Les longues listes de clés sont plafonnées dans la vue (avec une mention "affiche N sur le total") pour qu'un très grand catalogue reste réactif ; le filtrage continue de parcourir la liste complète. Un bouton "Copier comme rapport de révision" transforme le diff complet, y compris tout ce qui dépasse la limite, en un résumé Markdown dans ton presse-papiers, avec une brève confirmation "Copié" - prêt à coller dans la description d'une pull request ou dans un message de chat.
- **Aperçu** : une projection expurgée de ta configuration résolue (locales source et cible, format, motif de fichier, id du fournisseur et les options scalaires sûres) ainsi que tes termes de glossaire, s'il y en a. Les valeurs en texte libre qui pourraient contenir un secret collé ne sont jamais incluses.
- **Dérive du verrouillage** : si un fichier de verrouillage existe, sa version et sa dérive par locale par rapport aux fichiers source et cible actuels, sans y écrire.
- **Historique** : l'historique des commits pour tes fichiers de locale surveillés, lu via `git log` avec un nombre de commits plafonné et sans `--follow`. Un projet sans dépôt git, ou avec git indisponible, affiche l'historique comme non disponible plutôt qu'une erreur. L'historique tronqué d'un clone superficiel n'est pas non plus une erreur : il affiche simplement ce que le clone possède.

## Palette de commandes

Appuie sur Cmd+K (Ctrl+K sous Windows et Linux) n'importe où dans Studio pour ouvrir une palette de commandes. Tape pour filtrer sa liste : elle propose toujours les cinq vues ci-dessus comme cibles de navigation, plus, dès que la vue Diff a chargé des données pendant la session en cours, chaque combinaison de clé et de locale actuellement affichée. Sélectionner une vue bascule vers elle. Sélectionner une clé et une locale bascule vers Diff et ouvre directement la vue détaillée de cette clé, la même vue qu'ouvre un clic manuel sur cette clé. Appuie sur Échap, ou clique en dehors de la palette, pour la fermer et rendre le focus à l'endroit où tu étais.

Comme le reste de Studio, la palette est en lecture seule : aucune entrée, quel que soit son état, ne déclenche jamais de requête réseau, d'écriture de fichier ni d'appel à un fournisseur. C'est de la navigation pure.

## Vue détaillée d'une clé

Dans Diff, clique sur une clé modifiée pour ouvrir sa vue détaillée, avec une ligne par locale cible. Chaque ligne affiche aussi un indicateur d'intégrité des placeholders et ICU : vert signifie que la traduction de cette locale a toujours les mêmes placeholders que la source (des choses comme `{name}` ou `{{count}}`, selon ton format) ; rouge signifie qu'un placeholder manque ou qu'un placeholder inattendu a été ajouté, et nomme le jeton précis pour que tu saches quoi corriger. Un indicateur neutre et sans étiquette signifie soit que le format n'a aucun placeholder à vérifier, soit que la vérification n'est pas encore chargée, puisqu'elle n'est récupérée qu'à l'ouverture de la vue détaillée, pas de façon anticipée. Cela ne s'applique qu'aux clés modifiées : une clé manquante n'a pas encore de valeur cible, et une clé orpheline n'a pas de valeur source actuelle, donc aucune des deux n'a de seconde valeur à comparer.

## Retraduire une clé en échec

Démarre Studio avec `--allow-spend` et `--allow-write` ensemble (ou leurs variables d'environnement équivalentes) et la vue détaillée d'une clé gagne une action d'écriture : un bouton "Retranslate" à côté de toute ligne de locale dont l'indicateur d'intégrité est rouge. Laisse l'un des deux flags de côté et le bouton est absent du tableau de bord, pas simplement désactivé.

```bash
verbatra studio --allow-spend --allow-write
```

Cliquer dessus déclenche un seul appel en direct à ton fournisseur de traduction configuré pour exactement cette clé et ce locale, le même coût fournisseur que retraduire une clé avec [`verbatra translate`](/docs/cli/translate). Le résultat obtenu passe par la même vérification d'intégrité des placeholders et ICU avant que quoi que ce soit ne soit écrit : un résultat accepté met à jour le fichier de locale cible et l'entrée de verrouillage pour cette seule clé, laissant toute autre clé et tout autre locale intacts ; un résultat rejeté n'écrit rien. Le bouton affiche le résultat : "Retranslated" en cas de succès, "Rejected: placeholder mismatch" ou "Rejected: invalid message syntax" quand le nouveau résultat lui-même échoue à la vérification, ou "Failed: ..." pour tout le reste, par exemple si l'appel au fournisseur lui-même échoue. Une retraduction réussie atteint ton onglet de navigateur via le même signal de rafraîchissement en direct qu'un changement effectué en dehors de Studio.

## Installation

`@verbatra/studio` est le paquet propre de Studio. Il n'a pas encore eu de version stable, donc une installation simple apporte déjà le build de préversion actuel, aucun dist-tag requis. S'il n'est pas installé, `verbatra studio` quitte avec `2` et :

```
Verbatra Studio requires @verbatra/studio. Install it with: pnpm add -D @verbatra/studio
```

## Codes de sortie

| Code | Signification |
| --- | --- |
| `0` | Studio s'est arrêté proprement suite à un seul interrupt (Ctrl-C) |
| `2` | impossible de s'exécuter : une erreur d'exécution globale (configuration manquante ou invalide, `@verbatra/studio` non installé, ou le port est déjà utilisé), ou une erreur d'usage (un `--port` hors plage ou qui n'est pas un entier) |
| `130` | arrêté de force par un second interrupt |

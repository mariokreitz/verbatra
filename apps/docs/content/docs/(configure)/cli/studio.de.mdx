Verbatra Studio ist ein lokales Web-Dashboard über dein verbatra-Projekt: eine Live-Ansicht des Übersetzungsstatus, ausstehender Änderungen, der Konfiguration, der Lock-Datei-Abweichung und der Historie. Standardmäßig ruft es niemals einen Provider auf und schreibt niemals eine Datei. Startest du es mit zwei zusätzlichen Flags, erhält es zusätzlich eine Schreibaktion: das erneute Übersetzen eines einzelnen fehlschlagenden Schlüssels (siehe "Einen fehlschlagenden Schlüssel erneut übersetzen" weiter unten). Starte es mit:

```bash
verbatra studio
```

<Callout title="Nur als Prerelease verfügbar">
Verbatra Studio ist aktuell nur als Prerelease verfügbar, noch nicht als stabile Version. Um es zu nutzen, installiere `@verbatra/cli` explizit unter dem `next`-Tag: `pnpm add -D @verbatra/cli@next @verbatra/studio`. `@verbatra/studio` selbst hatte noch keine stabile Version, daher liefert schon sein eigener `latest`-Tag den aktuellen Prerelease-Build, kein Tag nötig. Ein einfaches `pnpm add -D @verbatra/cli` (ohne Tag) löst die aktuelle stabile Version auf und enthält diesen Befehl nicht.
</Callout>

## Nur lokal, und standardmäßig schreibgeschützt

Studio bindet sich ausschließlich an `127.0.0.1`: Es ist nie von einem anderen Rechner erreichbar, und es gibt keine Option, das zu ändern. Jede Anfrage wird gegen eine strikte Host- und Origin-Allowlist geprüft, und jede Seite und jeder API-Aufruf benötigt das im Start-URL ausgegebene Session-Token, das zeitkonstant verglichen wird, sodass ein falscher Versuch nicht per Timing erraten werden kann. Studio sendet keine CORS-Header, sodass ein Browser-Tab auf einem anderen Origin es nicht lesen könnte, selbst wenn er es versuchen würde. Standardmäßig gibt es keinen zustandsändernden Endpunkt in Studio: Es ruft niemals `translate` auf, führt niemals `watch` aus und importiert oder exportiert niemals ein Workbook. Alles, was du siehst, stammt aus denselben schreibgeschützten Berechnungen wie [`verbatra check`](/docs/cli/check) und [`verbatra diff`](/docs/cli/diff). Übergibst du beim Start sowohl `--allow-spend` als auch `--allow-write`, erhält Studio genau eine Schreibaktion, beschrieben in "Einen fehlschlagenden Schlüssel erneut übersetzen" weiter unten; lässt du eines der beiden Flags weg, bleibt Studio genauso schreibgeschützt wie bisher.

## Befehlsreferenz

| Option | Beschreibung |
| --- | --- |
| `--cwd <path>` | löst Konfigurations- und Locale-Dateien aus diesem Verzeichnis auf |
| `--config <path>` | lädt diese Konfigurationsdatei, anstatt nach einer zu suchen |
| `--port <n>` | überschreibt den Standard-Port von Studio (muss eine Ganzzahl von 1 bis 65535 sein) |
| `--allow-spend` | erlaubt Studio, einen Übersetzungs-Provider aufzurufen (auch: `VERBATRA_STUDIO_ALLOW_SPEND`) |
| `--allow-write` | erlaubt Studio, Locale-Dateien und die Lock-Datei zu schreiben (auch: `VERBATRA_STUDIO_ALLOW_WRITE`) |

`--allow-spend` und `--allow-write` sind beide standardmäßig aus. Jedes liest zusätzlich seine entsprechende Umgebungsvariable, wenn das Flag selbst fehlt (`"1"`, `"true"`, `"yes"` oder `"on"`, ohne Groß- und Kleinschreibung, zählt als an); ein explizites CLI-Flag gewinnt immer gegenüber der Umgebungsvariable.

`verbatra studio` lädt `.env.local` und `.env` genau wie jeder andere Befehl, löst dann deine Konfiguration einmal auf, bevor es überhaupt das Netzwerk berührt. Eine fehlende oder ungültige Konfiguration beendet sich mit `2` und demselben gerenderten Fehler wie jeder andere Befehl; Studio selbst wird nie gestartet.

## Der ausgegebene URL

Sobald Studio lauscht, gibt es eine Zeile auf stdout aus:

```
Verbatra Studio running at http://127.0.0.1:5849/?token=<token>
```

Öffne genau diesen URL. Das Token im Query-String bootstrapt ein Session-Cookie für den Browser-Tab; ohne es (oder mit dem falschen) wird jede Anfrage abgelehnt. Es wird niemals etwas ausgegeben, bevor der Server seinen Port erfolgreich gebunden hat, sodass ein fehlgeschlagener Start nie einen URL oder ein Token preisgibt.

## Neustart bei Konfigurationsänderung

Die Konfiguration wird einmal beim Start aufgelöst, und jede Ansicht für die Lebensdauer dieses Prozesses baut auf derselben aufgelösten Konfiguration auf. Wenn du `verbatra.config.ts` änderst (ein neues Ziel-Locale, ein anderer Provider, ein anderes Datei-Muster), stoppe Studio und starte es erneut, um die Änderung zu übernehmen. Bei Projektdateien ist das anders: Studio beobachtet deine Quell-Locale-Datei, jede Ziel-Locale-Datei und die Lock-Datei und pusht ein Live-Aktualisierungssignal an deinen offenen Browser-Tab, sobald sich eine davon auf der Festplatte ändert (schnell aufeinanderfolgende Schreibvorgänge werden zu einem einzigen Signal zusammengefasst). Der Tab lädt die gerade angezeigte Ansicht automatisch neu, kein Neuladen der Seite nötig. Schlägt ein Nachladen fehl, zeigt die Ansicht weiterhin ihre letzten guten Daten mit einem dezenten Hinweis auf veraltete Daten, statt leer zu werden, und versucht es beim nächsten Signal erneut. Eine unterbrochene Verbindung verbindet sich von selbst mit Backoff neu; fährt Studio selbst herunter, zeigt der Tab einen Vollbild-Hinweis, es vom Terminal aus neu zu starten.

## Fehlermeldungen

Wenn etwas schiefgeht, zum Beispiel wenn sich die Quell-Locale-Datei nicht lesen lässt, sich eine Ziel-Locale-Datei nicht für ihr konfiguriertes Format parsen lässt oder deine Sitzung abgelaufen ist, zeigt Studio eine konkrete, hilfreiche Beschreibung dessen, was passiert ist, statt einer einzigen allgemeinen Meldung. Einen Fehler, den Studio nicht eigens erkennt, zeigt es weiterhin mit derselben allgemeinen Meldung wie bisher; du siehst dabei nie einen rohen Stack-Trace oder ein internes Fehlerobjekt.

## Die Ansichten

- **Status**: pro Locale die Anzahl fehlender, veralteter und aktueller Schlüssel, dieselbe Berechnung wie `verbatra check`.
- **Diff**: die genauen Schlüssel, die pro Locale hinzugefügt, neu übersetzt oder verwaist wären, dieselbe Berechnung wie `verbatra diff`. Lange Schlüssellisten sind in der Ansicht begrenzt (mit einem Hinweis "zeige N von insgesamt"), damit ein sehr großer Katalog reaktionsfähig bleibt; das Filtern durchsucht weiterhin die vollständige Liste. Eine Schaltfläche "Als Prüfbericht kopieren" verwandelt den vollständigen Diff, einschließlich allem, was über die Begrenzung hinausgeht, in eine Markdown-Zusammenfassung in deiner Zwischenablage, mit einer kurzen Bestätigung "Kopiert" - bereit zum Einfügen in eine Pull-Request-Beschreibung oder eine Chat-Nachricht.
- **Übersicht**: eine geschwärzte Projektion deiner aufgelösten Konfiguration (Quell- und Ziel-Locales, Format, Datei-Muster, Provider-ID und die sicheren skalaren Optionen) sowie deine Glossar-Begriffe, falls vorhanden. Freitext-Werte, die ein eingefügtes Geheimnis enthalten könnten, sind niemals enthalten.
- **Lock-Abweichung**: ob eine Lock-Datei existiert, ihre Version und ihre Abweichung pro Locale gegenüber den aktuellen Quell- und Zieldateien, ohne sie zu schreiben.
- **Historie**: die Commit-Historie für deine beobachteten Locale-Dateien, gelesen über `git log` mit einer begrenzten Commit-Anzahl und ohne `--follow`. Ein Projekt ohne Git-Repository oder mit nicht verfügbarem Git zeigt die Historie als nicht verfügbar an, statt eines Fehlers. Die abgeschnittene Historie eines Shallow-Clones ist ebenfalls kein Fehler: Sie zeigt einfach, was der Clone hat.

## Befehlspalette

Drücke Cmd+K (unter Windows und Linux: Strg+K) an einer beliebigen Stelle in Studio, um die Befehlspalette zu öffnen. Tippe, um ihre Liste zu filtern: Sie bietet immer alle fünf Ansichten von oben als Sprungziele an, dazu, sobald die Diff-Ansicht in der aktuellen Sitzung schon einmal Daten geladen hat, jede darin aktuell angezeigte Kombination aus Schlüssel und Locale. Die Auswahl einer Ansicht wechselt zu ihr. Die Auswahl eines Schlüssels und einer Locale wechselt zu Diff und öffnet direkt die Detailansicht dieses Schlüssels, dieselbe Ansicht, die auch ein manueller Klick auf diesen Schlüssel öffnet. Drücke Esc, oder klicke außerhalb der Palette, um sie zu schließen und den Fokus dorthin zurückzugeben, wo du warst.

Wie der Rest von Studio ist auch die Palette schreibgeschützt: Kein Eintrag löst, in welchem Zustand auch immer, jemals eine Netzwerkanfrage, einen Dateischreibvorgang oder einen Provider-Aufruf aus. Sie ist reine Navigation.

## Detailansicht eines Schlüssels

Klicke in Diff auf einen geänderten Schlüssel, um seine Detailansicht zu öffnen, mit einer Zeile pro Ziel-Locale. Jede Zeile zeigt außerdem einen Hinweis zur Platzhalter- und ICU-Integrität: Grün bedeutet, dass die Übersetzung dieses Locales weiterhin dieselben Platzhalter wie die Quelle hat (etwa `{name}` oder `{{count}}`, je nach Format); Rot bedeutet, dass ein Platzhalter fehlt oder ein unerwarteter hinzugekommen ist, und nennt den genauen Token, damit du weißt, was zu beheben ist. Ein neutraler, unbeschrifteter Hinweis bedeutet entweder, dass das Format keine Platzhalter zu prüfen hat, oder dass die Prüfung noch nicht geladen ist, da sie erst beim Öffnen der Detailansicht abgerufen wird, nicht im Voraus. Das gilt nur für geänderte Schlüssel: Ein fehlender Schlüssel hat noch keinen Zielwert, und ein verwaister Schlüssel hat keinen aktuellen Quellwert, sodass keiner von beiden einen zweiten Wert zum Vergleichen hat.

## Einen fehlschlagenden Schlüssel erneut übersetzen

Startest du Studio mit sowohl `--allow-spend` als auch `--allow-write` (oder ihren Umgebungsvariablen-Entsprechungen), erhält die Detailansicht eines Schlüssels eine Schreibaktion: eine Schaltfläche "Retranslate" neben jeder Locale-Zeile, deren Integritätsanzeige rot ist. Lässt du eines der beiden Flags weg, fehlt die Schaltfläche im Dashboard vollständig, statt nur deaktiviert zu sein.

```bash
verbatra studio --allow-spend --allow-write
```

Ein Klick darauf löst genau einen Live-Aufruf an deinen konfigurierten Übersetzungs-Provider für exakt diesen Schlüssel und dieses Locale aus, mit denselben Provider-Kosten wie das erneute Übersetzen eines Schlüssels mit [`verbatra translate`](/docs/cli/translate). Das zurückgegebene Ergebnis durchläuft dieselbe Platzhalter- und ICU-Integritätsprüfung, bevor irgendetwas geschrieben wird: Ein akzeptiertes Ergebnis aktualisiert die Ziel-Locale-Datei und den Lock-Eintrag für genau diesen einen Schlüssel, alle anderen Schlüssel und Locales bleiben unberührt; ein abgelehntes Ergebnis schreibt nichts. Die Schaltfläche zeigt das Ergebnis an: "Retranslated" bei Erfolg, "Rejected: placeholder mismatch" oder "Rejected: invalid message syntax", wenn das neue Ergebnis selbst die Prüfung nicht besteht, oder "Failed: ..." für alles andere, zum Beispiel wenn schon der Provider-Aufruf selbst fehlschlägt. Ein erfolgreiches erneutes Übersetzen erreicht deinen Browser-Tab über dasselbe Live-Aktualisierungssignal wie eine Änderung außerhalb von Studio.

## Installation

`@verbatra/studio` ist das eigene Paket von Studio. Es hatte noch keine stabile Version, daher liefert eine einfache Installation schon den aktuellen Prerelease-Build, kein Dist-Tag nötig. Wenn es nicht installiert ist, beendet sich `verbatra studio` mit `2` und:

```
Verbatra Studio requires @verbatra/studio. Install it with: pnpm add -D @verbatra/studio
```

## Exit-Codes

| Code | Bedeutung |
| --- | --- |
| `0` | Studio wurde sauber durch einen einzelnen Interrupt (Ctrl-C) gestoppt |
| `2` | konnte nicht ausgeführt werden: ein Fehler des gesamten Laufs (fehlende oder ungültige Konfiguration, `@verbatra/studio` nicht installiert, oder der Port ist bereits belegt) oder ein Nutzungsfehler (ein `--port` außerhalb des Bereichs oder keine Ganzzahl) |
| `130` | durch einen zweiten Interrupt zwangsweise gestoppt |

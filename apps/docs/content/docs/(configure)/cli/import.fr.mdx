---
title: verbatra import
description: Importe un classeur rempli dans tes fichiers de locale, en exécutant les mêmes vérifications de sécurité qu'une exécution automatisée.
---
Relis dans tes fichiers de locale un classeur qu'un traducteur a rempli. `import` exécute les mêmes vérifications d'espaces réservés et d'ICU que [`verbatra translate`](/docs/cli/translate) : chaque valeur est validée, celles qui passent sont écrites et le [fichier de verrouillage](/docs/the-lock-file) est mis à jour, et toute valeur qui échoue à une vérification est retenue et signalée plutôt qu'écrite.

```bash
verbatra import verbatra-translations.xlsx
```

L'argument est le chemin vers le classeur produit par [`verbatra export`](/docs/cli/export) et rempli par le traducteur. Seule la colonne `Translation` est lue ; les cellules vides sont ignorées, jamais écrites comme des chaînes vides. Voir [Traduction humaine](/docs/manual-translation) pour l'aller-retour complet.

## Arguments

| Argument | Description |
| --- | --- |
| `<workbook>` | chemin vers le classeur rempli à importer |

## Options

| Option | Description |
| --- | --- |
| `--cwd <path>` | résoudre la configuration et les fichiers de locale depuis ce répertoire |
| `--config <path>` | charger ce fichier de configuration au lieu d'en rechercher un |
| `--dry-run` | valider et signaler sans écrire les fichiers de locale ni mettre à jour le verrouillage |
| `--json` | afficher le résumé de l'exécution au format JSON |

## Sortie

`import` réutilise le résumé d'exécution de `translate` tel quel. Sans `--json`, il affiche un résumé lisible par un humain : une ligne par locale, puis un agrégat. Chaque ligne de locale indique combien de valeurs étaient `translated`, `unchanged`, `orphaned`, `invalid-ICU skipped` et `integrity-withheld`. Avec `--json`, il affiche la même [structure `RunSummary`](/docs/sdk) sous forme d'un seul objet JSON sur stdout.

Une valeur est **retenue** lorsqu'elle casse un espace réservé, qu'elle est de l'ICU invalide, ou que sa source a changé depuis l'export du classeur (détecté via la colonne cachée de hachage de source). Une valeur retenue laisse le fichier de locale intact afin que tu puisses la corriger et la réimporter. Voir [Sûreté de la traduction](/docs/translation-safety) pour les vérifications.

Les colonnes `Review status` et `Review reasons` d'un classeur, quand elles sont présentes, sont en lecture seule et consultatives : elles ne conditionnent jamais `import`. Une ligne marquée `review` avec un `Translation` rempli est acceptée ou retenue selon exactement les mêmes règles que n'importe quelle autre ligne. Un classeur exporté avant l'existence de ces deux colonnes continue d'importer chaque ligne sans changement. Voir [Traduction humaine](/docs/manual-translation) pour ce que signifient les colonnes.

## Exemples

```bash
# import the filled workbook
verbatra import verbatra-translations.xlsx

# validate and report, write nothing
verbatra import verbatra-translations.xlsx --dry-run

# import against a project in another directory
verbatra import verbatra-translations.xlsx --cwd packages/app

# load a specific config instead of searching for one
verbatra import verbatra-translations.xlsx --config config/verbatra.config.ts

# machine-readable summary on stdout
verbatra import verbatra-translations.xlsx --json
```

## Codes de sortie

| Code | Signification |
| --- | --- |
| `0` | chaque locale cible a réussi |
| `1` | l'exécution s'est terminée mais une ou plusieurs locales ont échoué |
| `2` | impossible de s'exécuter : une erreur d'exécution globale (configuration manquante ou invalide, un classeur illisible ou invalide, format inconnu, ou un fichier de verrouillage corrompu), ou une erreur d'usage |

Le contrat des codes de sortie correspond à celui de [`verbatra translate`](/docs/cli/translate), de sorte qu'un transfert manuel s'embranche de la même manière en CI : voir [CI et automatisation](/docs/ci-and-exit-codes).

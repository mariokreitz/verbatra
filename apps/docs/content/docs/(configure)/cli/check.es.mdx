---
title: verbatra check
description: Informa qué locales tienen cadenas faltantes o desactualizadas, de solo lectura, sin llamar a un proveedor ni escribir archivos.
---
Informa, por cada locale de destino, cuántas cadenas faltan, están desactualizadas o están al día, y luego sale con un código sobre el que un paso de CI puede bifurcar. `check` ejecuta la misma lectura y el mismo diff que [`verbatra translate`](/docs/cli/translate) hace justo antes de llamar a un proveedor, y luego se detiene: no llama a ningún proveedor, no escribe archivos y nunca toca el [archivo de bloqueo](/docs/the-lock-file). Es la forma barata y de solo lectura de preguntar "¿están mis traducciones sincronizadas?" antes de que alguien pague a un proveedor.

```bash
verbatra check
```

Una clave está **faltante** cuando está en el locale de origen pero ausente del destino. Una clave está **desactualizada** cuando el destino la tiene pero la cadena de origen cambió desde la última vez que se tradujo. Una clave está **al día** cuando el destino coincide con el origen actual. Un locale está sincronizado cuando no tiene claves faltantes ni desactualizadas.

`check` te da recuentos. ¿Quieres las claves reales detrás de esos recuentos (cuáles se añadirían, se volverían a traducir o están huérfanas)? Recurre a [`verbatra diff`](/docs/cli/diff), su hermano detallado que ejecuta el mismo cálculo de solo lectura.

## Opciones

| Opción | Descripción |
| --- | --- |
| `--cwd <path>` | resuelve la configuración y los archivos de locale desde este directorio |
| `--config <path>` | carga este archivo de configuración en lugar de buscar uno |
| `--locales <list>` | subconjunto de locales de destino separados por comas (por defecto todos los configurados) |
| `--json` | imprime el resumen de check como JSON |

No hay `--dry-run`: `check` ya es una lectura en seco. No necesita ninguna clave de API.

## Salida

Sin `--json`, `check` imprime una línea por locale con sus recuentos y un marcador de sincronización, y luego una línea general:

```
verbatra check
  de: 3 missing, 1 stale, 120 up-to-date (out of sync)
  fr: 0 missing, 0 stale, 124 up-to-date (in sync)
out of sync (run verbatra translate to update)
```

Con `--json`, imprime el estado como un único objeto JSON en stdout. El `inSync` de nivel superior es true exactamente cuando todos los locales están sincronizados (y exactamente cuando el comando sale con `0`):

```json
{
  "inSync": false,
  "locales": [
    { "locale": "de", "missing": 3, "stale": 1, "upToDate": 120, "inSync": false },
    { "locale": "fr", "missing": 0, "stale": 0, "upToDate": 124, "inSync": true }
  ]
}
```

Un error de toda la ejecución va a stderr, así que stdout queda limpio y canalizable con `--json`.

## Ejemplos

```bash
# report missing and stale keys per locale (exit 1 if drifted)
verbatra check

# only check the German and French locales
verbatra check --locales de,fr

# check a project in another directory
verbatra check --cwd packages/app

# load a specific config instead of searching for one
verbatra check --config config/verbatra.config.ts

# machine-readable status on stdout for CI
verbatra check --json
```

## Códigos de salida

| Código | Significado |
| --- | --- |
| `0` | sincronizado: cada locale de destino no tiene claves faltantes ni desactualizadas |
| `1` | fuera de sincronización: al menos un locale tiene una clave faltante o desactualizada (el informe completo se sigue imprimiendo) |
| `2` | no se pudo ejecutar: un error de toda la ejecución (configuración ausente o no válida, origen ilegible, formato desconocido, un archivo de bloqueo corrupto o un locale solicitado que no está entre los locales de destino configurados), o un error de uso (incluida una lista `--locales` vacía) |

Para `check`, el código de salida `1` significa "fuera de sincronización", no un fallo por locale: la ejecución en sí tuvo éxito, solo que el resultado no está todo en verde. Eso convierte a `check` en una puerta de CI de una línea que falla cuando las traducciones se han desviado. Un valor `--locales` desconocido o vacío hace que la ejecución falle con el código de salida `2` en lugar de no comprobar nada en silencio, así que un error tipográfico en un nombre de locale se detecta en lugar de pasar. Consulta [CI y automatización](/docs/ci-and-exit-codes) para bifurcar sobre estos códigos.

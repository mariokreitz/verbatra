title: Fournisseurs
description: Les cinq fournisseurs de traduction, leurs options et la clé API que chacun lit.
---
verbatra propose cinq fournisseurs derrière une seule interface : Anthropic, OpenAI, Gemini et openai-compatible (un serveur local ou autohébergé compatible OpenAI) sont des modèles de langage étendus, et DeepL est un service de traduction automatique. Tu en sélectionnes un dans le bloc `provider` de ta configuration, par `id`.

## Lequel choisir ?

N'importe quel fournisseur fonctionne avec n'importe quel format et n'importe quelle commande ; le choix porte sur le coût, la qualité, la confidentialité et la façon d'obtenir une clé.

- **Petit projet ou projet perso, tu le veux gratuit ?** Utilise **Gemini**. Il dispose d'un vrai palier d'API gratuit, tu peux donc traduire une application entière sans aucun coût. Récupère une clé sur [Google AI Studio](https://aistudio.google.com/apikey) et définis `GEMINI_API_KEY`. Le palier gratuit est soumis à des limites de débit (voir la note ci-dessous), ce qui ne compte que lors d'une grande première exécution.
- **Meilleure qualité de traduction pour une application en production ?** Utilise **Anthropic** ou **OpenAI**. Ce sont tous deux des LLM payants qui lisent `ANTHROPIC_API_KEY` ou `OPENAI_API_KEY`.
- **Gros volume, piloté par glossaire, ou pas de LLM dans la boucle ?** Utilise **DeepL**. C'est une API de traduction automatique dédiée avec un palier gratuit et un palier payant, et elle prend en charge un glossaire natif par id. Elle ne traduit pas les chaînes contenant des espaces réservés (voir l'avis ci-dessous) ; pour celles-ci, utilise un fournisseur LLM.
- **Hors ligne, sans coût, ou tes chaînes ne doivent jamais quitter ton réseau ?** Utilise **openai-compatible**. Il pointe verbatra vers un serveur d'inférence local ou autohébergé (LM Studio, Ollama, vLLM) au lieu d'une API hébergée. Aucun compte, aucune clé API, et aucun trafic réseau au-delà de ta propre machine ou de ton LAN.

Pas sûr ? Commence sur le palier gratuit de Gemini, puis change de fournisseur plus tard en modifiant un seul `id` dans ta configuration. Rien d'autre ne change.

<Callout title="Autocomplétion des modèles en TypeScript">
Lorsque tu écris la configuration en TypeScript avec `defineConfig`, le champ `model` est restreint aux identifiants de modèle connus du fournisseur choisi, issus du SDK du fournisseur lui-même : un modèle d'un autre fournisseur (par exemple un modèle Claude avec `id: "gemini"`) est une erreur de type. À l'exécution, toute chaîne non vide reste acceptée, de sorte qu'un modèle que le SDK n'a pas encore listé fonctionne même si l'éditeur le signale. DeepL n'a pas de champ `model`.
</Callout>

## Les clés proviennent de l'environnement

verbatra ne lit jamais une clé API depuis la configuration. Chaque fournisseur hébergé lit exactement une variable d'environnement :

| ID du fournisseur | Variable d'environnement |
| --- | --- |
| `anthropic` | `ANTHROPIC_API_KEY` |
| `openai` | `OPENAI_API_KEY` |
| `gemini` | `GEMINI_API_KEY` |
| `deepl` | `DEEPL_API_KEY` |

Définis la variable dans `.env` (que `verbatra init` ajoute au gitignore) ou exporte-la dans ton shell ou ton coffre de secrets CI. Obtiens une clé depuis le tableau de bord du fournisseur, puis place sa valeur derrière la variable ci-dessus. Ne commite jamais une vraie clé.

`openai-compatible` est différent : la plupart des serveurs locaux n'ont besoin d'aucune clé, il n'apparaît donc pas dans ce tableau. Voir sa propre section ci-dessous pour le fonctionnement de la résolution de sa clé.

## Anthropic

```ts
provider: {
  id: "anthropic",
  options: {
    model: "claude-sonnet-4-6",
    maxTokens: 4096,
  },
}
```

Lit `ANTHROPIC_API_KEY`. `model` et `maxTokens` sont tous deux requis.

## OpenAI

```ts
provider: {
  id: "openai",
  options: {
    model: "gpt-5.4-mini",
    maxOutputTokens: 4096,
  },
}
```

Lit `OPENAI_API_KEY`. `model` et `maxOutputTokens` sont tous deux requis.

## Gemini

```ts
provider: {
  id: "gemini",
  options: {
    model: "gemini-2.5-flash",
    maxOutputTokens: 4096,
  },
}
```

Lit `GEMINI_API_KEY`. `model` et `maxOutputTokens` sont tous deux requis.

<Callout title="Le palier gratuit et ses limites">
Gemini est le point de départ recommandé pour les petits projets parce que son API a un palier gratuit. Crée une clé sur [Google AI Studio](https://aistudio.google.com/apikey), sans facturation requise. Le palier gratuit plafonne les requêtes par minute et par jour (les chiffres exacts dépendent du modèle et changent au fil du temps, alors vérifie les limites actuelles de Google), et verbatra découpe le travail en sous-lots séquentiels, de sorte qu'une première exécution sur de nombreuses langues peut atteindre ces plafonds. Une brève erreur de limite de débit ou de serveur est retentée automatiquement avec une courte attente, mais cela ne lisse que les accrocs momentanés, pas un plafond soutenu. Si tu atteins le plafond, traduis une langue à la fois ou baisse `maxBatchSize`, puis relance : verbatra ne reprend que ce qui manque encore, les exécutions sont donc sûres à répéter.
</Callout>

## DeepL

```ts
provider: {
  id: "deepl",
  options: {},
}
```

Lit `DEEPL_API_KEY`. DeepL n'a besoin d'aucun modèle. La seule option est un `glossaryId` facultatif qui nomme un glossaire que tu as déjà créé dans DeepL :

```ts
provider: {
  id: "deepl",
  options: {
    glossaryId: "<your-glossary-id>",
  },
}
```

<Callout title="DeepL et les espaces réservés">
DeepL ne peut pas conserver les espaces réservés ni les tokens ICU, donc il n'envoie pas les chaînes qui en contiennent : ces entrées restent non traduites (retenues) et sont signalées par un avis `PLACEHOLDER_UNSUPPORTED` sur le résultat. Les chaînes sans espaces réservés sont traduites normalement. Pour traduire les chaînes contenant des espaces réservés, utilise un fournisseur LLM (Anthropic, OpenAI, Gemini ou openai-compatible).
</Callout>

## openai-compatible

Pointe verbatra vers un serveur local ou autohébergé qui parle l'API Chat Completions d'OpenAI : LM Studio, Ollama, vLLM et autres fonctionnent tous. L'id nomme le protocole de transport, pas l'endroit où le serveur tourne, donc un futur serveur hébergé qui parle le même protocole s'y intègre aussi.

```ts
provider: {
  id: "openai-compatible",
  options: {
    baseUrl: "http://192.168.178.74:1234/v1",
    model: "google/gemma-4-26b-a4b-qat",
    maxOutputTokens: 1024,
  },
}
```

`baseUrl`, `model` et `maxOutputTokens` sont requis. Contrairement à tous les autres fournisseurs, `baseUrl` vit dans la configuration plutôt que dans l'environnement : c'est une adresse réseau que tu connais déjà (une IP de LAN ou `localhost`), pas un secret. Elle doit être une URL absolue valide utilisant le schéma `http` ou `https` ; tout le reste, y compris une faute de frappe comme un schéma manquant, échoue immédiatement à la validation de la configuration avec une erreur claire, de la même façon qu'un `model` mal formé le ferait pour n'importe quel autre fournisseur.

`baseUrl` doit inclure le segment de chemin d'API de ton serveur, typiquement `/v1` : LM Studio, Ollama et vLLM servent tous leurs routes compatibles OpenAI sous `/v1`, la même convention que le client sous-jacent utilise déjà pour le fournisseur hébergé `openai`. Un `baseUrl` sans ce segment atteint le mauvais chemin sur le serveur et échoue au moment de la requête, pas à la validation de la configuration, puisque l'URL elle-même reste syntaxiquement valide.

### Les clés sont facultatives

La plupart des serveurs d'inférence locaux n'ont besoin d'aucune clé API, donc `openai-compatible` résout sa clé en trois niveaux, dans l'ordre :

1. **`apiKeyEnvVar`**, un champ de configuration facultatif nommant une variable d'environnement dont lire une vraie clé. Utilise ce champ quand la clé de ton serveur vit déjà sous un nom de variable existant (un secret d'entreprise partagé, un `LM_STUDIO_KEY` que tu exportes déjà, et ainsi de suite). Si tu définis ce champ et que sa variable nommée est absente ou vide, verbatra échoue clairement au lieu de retomber silencieusement, puisque tu as explicitement dit qu'une clé était requise.
2. **`OPENAI_COMPATIBLE_API_KEY`**, une variable d'environnement de convention, lue quand `apiKeyEnvVar` n'est pas défini. Si elle est définie et non vide, verbatra l'utilise. Si elle est absente ou vide, verbatra passe au niveau suivant sans erreur.
3. **Le marqueur `"local"`**, envoyé quand aucun des deux niveaux ci-dessus ne résout quoi que ce soit. C'est une chaîne fixe, non secrète, pas une vraie clé, et c'est exactement ce qui fait fonctionner un serveur local sans clé (le cas courant) avec une configuration nulle.

`apiKeyEnvVar` ne peut jamais nommer `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GEMINI_API_KEY` ou `DEEPL_API_KEY` : la validation de la configuration le rejette d'emblée, donc l'auteur d'une configuration ne peut pas pointer ce fournisseur vers la clé d'un des fournisseurs hébergés par son nom. Le client ne lit non plus jamais `OPENAI_API_KEY` lui-même et transmet toujours une valeur de clé explicite au SDK sous-jacent, donc une clé hébergée ne peut jamais atteindre un `baseUrl` personnalisé, ni par accident ni volontairement.

<Callout type="warn" title="Une vraie clé en http en clair voyage en clair">
verbatra autorise le `http://` en clair vers n'importe quel hôte, y compris une adresse LAN comme `http://192.168.178.74:1234`, sans restriction au-delà du schéma http ou https. C'est intentionnel et sûr dans le cas courant sans clé : rien de secret ne transite sur la ligne.

Ce n'est un risque réel que lorsque tu configures aussi une clé (via `apiKeyEnvVar` ou `OPENAI_COMPATIBLE_API_KEY`) **et** que `baseUrl` est du `http://` en clair vers un hôte non loopback (autre chose que `localhost` ou `127.0.0.1`) : cette clé voyage alors en clair sur ton réseau. Si ton serveur exige une vraie clé et est accessible via ton LAN ou au-delà, utilise `https://` pour ce serveur, ou restreins-le à `localhost`. verbatra ne détecte ni ne signale cette combinaison aujourd'hui ; considère-la comme ta propre responsabilité en attendant.
</Callout>

### Analyse tolérante pour les modèles locaux et plus petits

`openai-compatible` demande le même format de réponse strict et lié à un schéma que le fournisseur hébergé `openai`, donc un serveur conforme reçoit exactement la forme que verbatra attend. La seule différence se situe au retour : les modèles locaux et plus petits respectent moins fiablement cette contrainte que les fournisseurs LLM hébergés, et ont plus tendance à envelopper une réponse par ailleurs correcte dans du texte environnant ou un bloc de code Markdown malgré la consigne de ne pas le faire. `openai-compatible` analyse de façon tolérante : il extrait le premier objet JSON équilibré n'importe où dans la réponse avant de l'analyser, donc le texte environnant ou un bloc de code est ignoré, ce que le fournisseur hébergé `openai` ne fait pas. Sa sortie passe toujours exactement par les mêmes vérifications de validation de schéma et d'intégrité des espaces réservés et ICU que n'importe quel autre fournisseur : la sortie d'un modèle local n'est pas fiable, comme celle de n'importe quel fournisseur, et une réponse mal formée ou incohérente sur les espaces réservés est rejetée de la même façon.

<Callout title="Pas encore dans verbatra init">
`verbatra init` ne propose pas actuellement `openai-compatible` comme option d'échafaudage, puisqu'il n'a pas de variable d'environnement unique et requise à demander. Ajoute le bloc de fournisseur ci-dessus à ta configuration à la main ; tout le reste de ce guide (glossaire, ton, limites de tokens en sortie) s'applique à lui exactement comme à n'importe quel fournisseur LLM.
</Callout>

## Ton et glossaire selon les fournisseurs

Les champs de configuration facultatifs `tone` et `glossary` (voir [Configuration](/docs/config-file)) sont appliqués différemment selon le fournisseur :

- **Ton.** Les fournisseurs LLM reçoivent le ton avec la requête et sont instruits de le respecter. DeepL le mappe sur la formalité : `formal` devient plus formel, `informal` devient moins formel, et `neutral` ou un ton absent est omis. Avec une clé DeepL au palier gratuit, un ton autre que par défaut n'est pas pris en charge et retombe sur la valeur par défaut, avec un avis sur le résultat.
- **Glossaire.** Les fournisseurs LLM reçoivent la table de termes `glossary` et sont instruits de traiter ses termes comme contraignants. DeepL n'applique pas de table de termes générique ; il enregistre un avis et utilise à la place l'option `glossaryId` ci-dessus.

## Limites de tokens en sortie

Les quatre fournisseurs LLM (Anthropic, OpenAI, Gemini et openai-compatible) plafonnent le nombre de tokens qu'une seule réponse peut produire. Lorsqu'un fournisseur s'arrête parce qu'il a atteint ce plafond (Anthropic `stop_reason` `max_tokens`, Gemini `MAX_TOKENS`, ou `finish_reason` `length` pour OpenAI et openai-compatible, qui partagent le même traitement de réponse), verbatra lève une erreur structurée distincte avec le code `OUTPUT_TRUNCATED` au lieu d'un échec générique de réponse mal formée. Le message est fixe :

> The provider stopped because the output-token limit was reached. Reduce the batch size or raise the configured max output tokens.

Tu as deux leviers, et l'un ou l'autre le résout :

- Baisse `maxBatchSize` dans ta configuration (voir [Configuration](/docs/config-file)) pour que chaque requête traduise moins d'entrées et produise une réponse plus petite. C'est purement de la configuration et s'applique à tous les fournisseurs.
- Augmente le budget de tokens par réponse chez le fournisseur : `maxTokens` pour Anthropic, ou `maxOutputTokens` pour OpenAI, Gemini et openai-compatible (les options montrées dans les blocs ci-dessus).

DeepL n'est pas concerné. C'est un service de traduction automatique qui ne passe pas par la couche LLM, il n'a donc aucun cas de troncature de tokens en sortie.

# How it works

A verbatra run is a short pipeline: load the config, read the source locale, then for each target
locale diff against what was translated last time, send only what changed to the provider, check the
result, and write. This page walks through each step so you know exactly what a run does.

## Config discovery and load

A run starts by loading the configuration. verbatra searches upward from the working directory and
uses the first source it finds: a `"verbatra"` key in `package.json`, then one of the
`.verbatrarc.*` files, then a `verbatra.config.*` file. The config is validated before any work begins. If nothing is
found the run fails with `CONFIG_NOT_FOUND`; if a config is present but invalid it fails with
`CONFIG_INVALID`. The config carries no API key; the provider reads its key from the environment.

## Reading the source locale

verbatra resolves the source file from `files.pattern` by substituting the `{locale}` token with
`sourceLocale`, relative to the working directory. If that file is missing the run fails with
`SOURCE_UNREADABLE`; if it cannot be parsed it fails with `SOURCE_INVALID`. While reading, the format
adapter also notes any source keys whose values are invalid for the format's message syntax (invalid
ICU). Those keys are carried along and skipped for translation rather than sent in a broken state.

## The per-locale run

Each target locale is handled on its own. A failure in one locale does not abort the others.

### Diff against the lock baseline

verbatra reads the existing target file (an absent file is treated as empty) and diffs the source
against it, using the lock file as the baseline of what was translated before. The diff sorts every
source key into one of four buckets:

- new keys that the target does not have yet,
- changed keys whose source text differs from the recorded baseline,
- unchanged keys that are already up to date,
- orphaned keys present in the target but no longer in the source.

Only the new and changed keys are candidates for translation. Unchanged keys are left alone, and
orphaned keys are reported but not removed. Any candidate flagged invalid ICU is set aside as well.
This is what keeps a run incremental: a provider is called only for the strings that actually changed.

### Calling the provider

The candidate entries are sent to the configured provider in one batch, along with the optional
`glossary` and `tone` from your config. How those are applied depends on the provider.

- The LLM providers (Anthropic, OpenAI, Gemini) receive the entries, the glossary, and the tone in a
  structured data payload, and are instructed to treat the glossary terms as binding and to honor the
  tone. The model is constrained to return one value per requested key.
- DeepL is a machine-translation service and maps the same inputs to its own options. Tone becomes
  formality: `formal` asks for a more formal result, `informal` for a less formal one, and `neutral`
  or an absent tone is omitted. A configured `glossaryId` is passed through natively.

DeepL can also report notices when it gracefully degrades. On a free-tier key, a non-default tone is
not supported, so it falls back to the default and emits a `FORMALITY_DOWNGRADED` notice. A generic
`glossary` term map is not applied by DeepL (it uses a glossary by id instead), so it emits a
`GLOSSARY_IGNORED` notice. These are data on the result, not errors. The LLM providers emit no
notices.

### Checking placeholders and writing

After translation, verbatra compares the placeholder set of each translated value against its source.
A translation that drops or alters a placeholder fails this integrity check and is withheld rather
than written. The accepted translations are merged into the target locale and written atomically, so
an interrupted write never leaves a half-finished file.

## The lock file and incremental work

verbatra keeps a committed lock file (`verbatra.lock.json`) that records, per locale, a content hash
for each translated source key. That record is the baseline the next run diffs against. After a
locale is written, the lock is refreshed with the current source hashes, with one deliberate
exception: keys that were withheld this run (a placeholder mismatch or invalid ICU) keep their prior
hash, so the next run sees them as still needing work and retries them. Orphaned keys get no entry.
A missing lock file is treated as a first run; a corrupt one fails with `LOCK_FILE_INVALID` rather
than being silently overwritten.

## The run summary

A run returns a summary across all target locales: the locales that succeeded, the ones that failed,
and a per-locale breakdown. Each locale entry reports the keys translated, the keys left unchanged,
the orphaned keys, any source keys skipped for invalid ICU, any translations withheld for a
placeholder mismatch, and any provider notices (such as the DeepL cases above). A dry run produces the
same summary shape, listing the keys that would be translated, without calling a provider or writing
anything.

## Errors

verbatra draws a clear line between two kinds of failure. A whole-run problem, such as a missing or
invalid config, an unreadable source, an unknown format, a provider that cannot be constructed
(including a missing API key), or a corrupt lock file, stops the run with a structured error. A
single locale failing is not fatal: it is recorded on that locale's entry with a code and a message,
and the rest of the locales still run.

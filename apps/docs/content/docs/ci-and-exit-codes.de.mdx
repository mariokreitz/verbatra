# CI und Exit-Codes

verbatra ist eine Entwicklungsabhängigkeit, daher laufen dieselben Befehle, die du lokal ausführst, auch in CI. Installiere es, stelle den Provider-Schlüssel aus deinem CI-Geheimnisspeicher als die passende Umgebungsvariable bereit und führe `verbatra translate` aus.

```bash
pnpm add -D @verbatra/cli
# ANTHROPIC_API_KEY is provided by the CI secret store as an environment variable
verbatra translate --json
```

Um eine Pipeline fehlschlagen zu lassen, wenn Übersetzungen abgewichen sind, ohne einen Provider aufzurufen oder einen Schlüssel zu benötigen, führe [`verbatra check`](/docs/cli/check) aus: Es beendet sich mit `1`, wenn ein Locale nicht synchron ist.

```bash
# read-only gate: exit 1 if any locale is missing or stale
verbatra check
```

Um die genauen Schlüssel hinter dieser Abweichung zu sehen (zum Beispiel um sie in einem Pull-Request-Kommentar zu posten), führe [`verbatra diff`](/docs/cli/diff) aus: dasselbe schreibgeschützte Gate, aber es listet die Schlüssel auf, die hinzugefügt, neu übersetzt oder verwaist würden, anstatt sie nur zu zählen.

```bash
# read-only gate with the key lists: exit 1 if any locale has pending changes
verbatra diff
```

## Exit-Codes

verbatra verwendet seinen Exit-Code als den Vertrag, auf den ein CI-Schritt verzweigt:

| Code | Bedeutung |
| --- | --- |
| `0` | Erfolg (bei `check` ist jedes Locale synchron; bei `diff` keine ausstehenden Änderungen) |
| `1` | `translate` oder `import` wurde beendet, aber einige Locales sind fehlgeschlagen, `check` fand ein Locale nicht synchron, oder `diff` fand ausstehende Änderungen |
| `2` | konnte nicht ausgeführt werden (ein Fehler des gesamten Laufs oder ein Nutzungsfehler) |
| `130` | `watch` wurde durch eine zweite Unterbrechung erzwungen gestoppt |

Ein Fehlschlag eines einzelnen Laufs in `watch` wird als Stream-Datensatz gemeldet, nicht als Exit-Code; eine einzelne Unterbrechung stoppt `watch` ordnungsgemäß und beendet sich mit `0`.

## JSON-Ausgabe

Die Befehle `translate`, `watch`, `check`, `diff`, `export` und `import` akzeptieren `--json` für maschinenlesbare Ausgabe.

`verbatra translate --json` gibt die Lauf-Zusammenfassung als ein JSON-Objekt mit dieser Struktur aus:

```ts
interface RunSummary {
  dryRun: boolean;          // whether this was a dry run (no provider calls, no writes)
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
}
```

`verbatra watch --json` gibt einen Datensatz pro Lauf als NDJSON aus (ein JSON-Objekt pro Zeile). Jeder Datensatz ist entweder ein Erfolg oder ein Fehlschlag:

```ts
type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };
```

Ein fehlgeschlagener Lauf während `watch` ist einer dieser Datensätze im Stream; er stoppt den Watcher nicht und setzt keinen Exit-Code ungleich null.

`verbatra check --json` gibt ein schreibgeschütztes Statusdokument als ein JSON-Objekt aus. Das oberste `inSync` ist genau dann wahr, wenn jedes Locale synchron ist, was genau dann der Fall ist, wenn sich der Befehl mit `0` beendet:

```ts
interface CheckSummary {
  inSync: boolean;             // true exactly when the command exits 0
  locales: LocaleCheckSummary[];
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;             // in source, absent from target
  stale: number;               // source changed since last translated
  upToDate: number;            // target matches the current source
  inSync: boolean;             // missing === 0 && stale === 0
}
```

`verbatra diff --json` gibt dieselbe schreibgeschützte Berechnung als Schlüssellisten statt als Anzahlen aus. Das oberste `hasPendingChanges` ist genau dann wahr, wenn mindestens ein Locale einen Schlüssel zum Hinzufügen oder Neuübersetzen hat, was genau dann der Fall ist, wenn sich der Befehl mit `1` beendet. Verwaiste Schlüssel werden aufgelistet, setzen aber `hasPendingChanges` nicht:

```ts
interface DiffSummary {
  hasPendingChanges: boolean;  // true exactly when the command exits 1
  locales: LocaleDiff[];
}

interface LocaleDiff {
  locale: string;
  missing: string[];           // in source, absent from target: would be added
  changed: string[];           // source changed since last translated: would be re-translated
  orphaned: string[];          // in target, absent from source: reported only
  hasPendingChanges: boolean;  // missing.length > 0 || changed.length > 0
}
```

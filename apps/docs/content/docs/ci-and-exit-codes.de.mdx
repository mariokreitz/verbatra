# CI und Exit-Codes

verbatra ist eine dev-Abhängigkeit, also laufen die Befehle, die du lokal ausführst, in CI genauso. Installiere es, übergib den Provider-Schlüssel aus deinem CI-Secret-Store als passende Umgebungsvariable und führe `verbatra translate` aus.

```bash
pnpm add -D @verbatra/cli
# ANTHROPIC_API_KEY is provided by the CI secret store as an environment variable
verbatra translate --json
```

Willst du eine Pipeline scheitern lassen, wenn Übersetzungen abgedriftet sind, ohne einen Provider aufzurufen oder auch nur einen Schlüssel zu brauchen? Führe [`verbatra check`](/docs/cli/check) aus: es beendet sich mit `1`, wenn irgendeine Locale nicht synchron ist.

```bash
# read-only gate: exit 1 if any locale is missing or stale
verbatra check
```

Brauchst du die genauen Keys hinter diesem Drift (etwa um sie in einem Pull-Request-Kommentar zu posten)? Führe [`verbatra diff`](/docs/cli/diff) aus: dasselbe read-only Gate, aber statt nur zu zählen, listet es die Keys auf, die hinzugefügt, neu übersetzt oder verwaist wären.

```bash
# read-only gate with the key lists: exit 1 if any locale has pending changes
verbatra diff
```

## Exit-Codes

Der Exit-Code ist der Vertrag, auf den dein CI-Schritt verzweigt:

| Code | Bedeutung |
| --- | --- |
| `0` | Erfolg (bei `check` ist jede Locale synchron; bei `diff` gibt es keine ausstehenden Änderungen) |
| `1` | `translate` oder `import` ist fertig, aber einige Locales sind fehlgeschlagen, `check` hat eine nicht synchrone Locale gefunden, oder `diff` hat ausstehende Änderungen gefunden |
| `2` | konnte nicht laufen (ein Gesamtlauf-Fehler oder ein Nutzungsfehler) |
| `130` | `watch` wurde durch einen zweiten Interrupt zwangsgestoppt |

`watch` ist die Ausnahme: ein Fehler pro Lauf erscheint als Stream-Eintrag, nicht als Exit-Code, und ein einzelner Interrupt stoppt `watch` sauber und beendet sich mit `0`.

## JSON-Ausgabe

Jeder Befehl (`translate`, `watch`, `check`, `diff`, `export` und `import`) akzeptiert `--json` für maschinenlesbare Ausgabe.

`verbatra translate --json` gibt die Lauf-Zusammenfassung als ein JSON-Objekt mit dieser Form aus:

```ts
interface RunSummary {
  dryRun: boolean;          // whether this was a dry run (no provider calls, no writes)
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
}
```

`verbatra watch --json` gibt einen Eintrag pro Lauf als NDJSON aus (ein JSON-Objekt pro Zeile). Jeder Eintrag ist ein Erfolg oder ein Fehler:

```ts
type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };
```

Ein fehlgeschlagener Lauf während `watch` landet als einer dieser Einträge auf dem Stream. Er stoppt den Watcher nicht und setzt keinen Exit-Code ungleich null.

`verbatra check --json` gibt ein read-only Statusdokument als ein JSON-Objekt aus. Das oberste `inSync` ist genau dann wahr, wenn jede Locale synchron ist, was genau dann ist, wenn der Befehl sich mit `0` beendet:

```ts
interface CheckSummary {
  inSync: boolean;             // true exactly when the command exits 0
  locales: LocaleCheckSummary[];
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;             // in source, absent from target
  stale: number;               // source changed since last translated
  upToDate: number;            // target matches the current source
  inSync: boolean;             // missing === 0 && stale === 0
}
```

`verbatra diff --json` führt dieselbe read-only Berechnung aus, gibt dir aber Key-Listen statt Zählungen. Das oberste `hasPendingChanges` ist genau dann wahr, wenn mindestens eine Locale einen Key zum Hinzufügen oder Neuübersetzen hat, was genau dann ist, wenn der Befehl sich mit `1` beendet. Verwaiste Keys erscheinen in den Listen, setzen aber nie `hasPendingChanges`:

```ts
interface DiffSummary {
  hasPendingChanges: boolean;  // true exactly when the command exits 1
  locales: LocaleDiff[];
}

interface LocaleDiff {
  locale: string;
  missing: string[];           // in source, absent from target: would be added
  changed: string[];           // source changed since last translated: would be re-translated
  orphaned: string[];          // in target, absent from source: reported only
  hasPendingChanges: boolean;  // missing.length > 0 || changed.length > 0
}
```

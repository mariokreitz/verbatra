# SDK

`@verbatra/sdk` ist der programmatische Kern, auf dem der `verbatra`-Befehl aufbaut. Das CLI ist ein dünner
Wrapper darüber, sodass du alles, was die Kommandozeile tut, auch im Code tun kannst: eine Konfiguration laden und
validieren, eine einmalige Übersetzung ausführen oder die Quelle überwachen und bei jeder Änderung neu übersetzen. Greife zum SDK,
wenn du Übersetzung in einem Skript, einem CI-Job oder deinem eigenen Werkzeug möchtest.

## Installation

```bash
pnpm add -D @verbatra/sdk
# npm
npm install -D @verbatra/sdk
# yarn
yarn add -D @verbatra/sdk
```

Erfordert Node.js `>=22.14.0`.

## Die öffentliche Oberfläche

Das Paket exportiert acht Funktionen und die Typen, die ein Aufrufer berührt.

- `defineConfig(config): VerbatraConfig` gibt sein Argument unverändert zurück. Es existiert nur, um dir
  Typinferenz und Autovervollständigung zu geben, wenn du eine Konfiguration in TypeScript schreibst; das `model`-Feld ist
  auf die bekannten Modelle des gewählten Providers beschränkt, sodass das Modell eines anderen Providers ein Typfehler ist.
- `loadConfig(options?: LoadConfigOptions): Promise<VerbatraConfig>` erkennt und validiert die
  Konfiguration. `LoadConfigOptions` akzeptiert `cwd`, einen expliziten `configPath` oder ein In-Memory-
  `configOverride`. Es wirft einen `SdkError`, wenn keine Konfiguration gefunden wird oder die Validierung fehlschlägt.
- `translate(input: TranslateInput): Promise<RunSummary>` führt den Ablauf aus Lesen, Diff, Übersetzen, Schreiben
  über jedes Ziel-Locale aus. `TranslateInput` ist `{ config, cwd?, dryRun?, prune?, generatePlurals? }`.
  `prune` (standardmäßig aus) entfernt verwaiste Schlüssel aus der geschriebenen Datei und der Lock-Datei; `generatePlurals`
  (standardmäßig aus) synthetisiert die CLDR-Pluralformen, die eine Zielsprache benötigt, der Quelle aber fehlen
  (nur i18next-JSON mit einem LLM-Provider). Jedes Pro-Lauf-Flag überschreibt, wenn gesetzt, die passende Konfigurations-
  option für diesen Lauf. Es wird zu einer `RunSummary` aufgelöst; Probleme des gesamten Laufs werfen einen `SdkError`, während
  Ergebnisse pro Locale Daten in der Zusammenfassung sind.
- `watch(input: WatchInput): Promise<WatchController>` überwacht die Quelldatei und läuft bei jeder
  entprellten Änderung neu. `WatchInput` ist `{ config, cwd?, debounceMs?, onRun }`, wobei `onRun` ein
  `WatchRunResult` pro Lauf erhält. Der zurückgegebene `WatchController` hat eine einzige Methode, `stop()`, die
  den Watcher schließt und auf den laufenden Lauf wartet.
- `exportWorkbook(input: ExportWorkbookInput): Promise<ExportWorkbookResult>` schreibt die Strings, die
  übersetzt werden müssen, in eine gestaltete Excel-Arbeitsmappe für einen menschlichen Übersetzer. `ExportWorkbookInput` ist
  `{ config, cwd?, out?, locales?, includeUnchanged? }`. Es wird zu einem `ExportWorkbookResult` aufgelöst:
  dem absoluten geschriebenen `path` und den `rows`-Anzahlen pro Locale. Es wird kein Provider aufgerufen und keine Lock-
  Datei geschrieben. Siehe [Manuelle Übersetzung](/docs/manual-translation).
- `importWorkbook(input: ImportWorkbookInput): Promise<RunSummary>` liest eine ausgefüllte Arbeitsmappe zurück in
  die Locale-Dateien und führt dieselben Platzhalter- und ICU-Prüfungen aus wie `translate`. `ImportWorkbookInput`
  ist `{ config, workbook, cwd?, dryRun? }`. Es wird zu derselben `RunSummary` wie `translate` aufgelöst, sodass ein
  Fehlschlag pro Locale Daten in der Zusammenfassung ist statt eines Wurfs. `DEFAULT_WORKBOOK_PATH` ist der
  standardmäßige Export-Pfad (`verbatra-translations.xlsx`).
- `check(input: CheckInput): Promise<CheckSummary>` führt dasselbe Lesen und Diff aus wie `translate`, hört aber
  vor jedem Provider-Aufruf auf: Es meldet die Abweichung pro Locale und schreibt nichts. `CheckInput` ist
  `{ config, cwd?, locales? }`. Es wird zu einer `CheckSummary` aufgelöst, deren `inSync` genau dann wahr ist, wenn
  jedes geprüfte Locale keine fehlenden und keine veralteten Schlüssel hat. Es wird kein Provider aufgerufen, keine Datei geschrieben,
  und die Lock-Datei wird nicht berührt. Siehe [`verbatra check`](/docs/cli/check). Für die Schlüssel hinter diesen
  Anzahlen verwende `diff` unten.
- `diff(input: DiffInput): Promise<DiffSummary>` ist das schreibgeschützte Geschwister von `check`: Es führt
  dasselbe Lesen und Diff aus, meldet aber Schlüssellisten statt Anzahlen. `DiffInput` ist `{ config, cwd?, locales? }`.
  Es wird zu einer `DiffSummary` aufgelöst, deren Einträge pro Locale `missing` (Schlüssel, die hinzugefügt würden),
  `changed` (Schlüssel, die neu übersetzt würden) und `orphaned` (Schlüssel im Ziel, aber nicht in der Quelle,
  nur gemeldet) tragen. Sein `hasPendingChanges` ist genau dann wahr, wenn ein Locale einen fehlenden oder geänderten Schlüssel hat;
  verwaiste Schlüssel allein setzen es nicht. Es wird kein Provider aufgerufen, keine Datei geschrieben, und die Lock-Datei wird nicht
  berührt. Siehe [`verbatra diff`](/docs/cli/diff).

Die Ergebnistypen:

```ts
interface RunSummary {
  dryRun: boolean;          // true when nothing was written and no provider was called
  locales: LocaleSummary[]; // one entry per target locale, in config order
  succeeded: string[];      // locales whose run succeeded
  failed: string[];         // locales whose run failed
}

interface LocaleSummary {
  locale: string;
  status: "succeeded" | "failed";
  translated: string[];          // keys translated this run (in dry-run, keys that would be)
  unchanged: string[];           // keys already up to date
  orphaned: string[];            // target keys with no matching source key (reported regardless of pruning)
  pruned: string[];              // orphaned keys removed this run; empty unless pruning is on (subset of orphaned)
  invalidIcuSource: string[];    // source keys skipped for invalid ICU
  integrityMismatches: string[]; // translations withheld for a placeholder mismatch
  generated: string[];           // CLDR plural forms synthesized this run; empty unless generation is on
  notices: LocaleNotice[];       // provider notices (e.g. DeepL degradation) and SDK notices (e.g. PLURAL_CATEGORIES_INCOMPLETE)
  error?: { code: string; message: string }; // present only when status is "failed"
}

// A notice is either a provider-emitted notice or an SDK-emitted one.
type LocaleNotice = ProviderNotice | SdkNotice;

type WatchRunResult =
  | { status: "succeeded"; summary: RunSummary }
  | { status: "failed"; error: { code: string; message: string } };

interface CheckSummary {
  inSync: boolean;                 // true exactly when every locale is in sync
  locales: LocaleCheckSummary[];   // one entry per checked locale, in config order
}

interface LocaleCheckSummary {
  locale: string;
  missing: number;                 // in source, absent from target
  stale: number;                   // source changed since last translated
  upToDate: number;                // target matches the current source
  inSync: boolean;                 // missing === 0 && stale === 0
}

interface DiffSummary {
  hasPendingChanges: boolean;      // true when some locale has a missing or changed key
  locales: LocaleDiff[];           // one entry per checked locale, in config order
}

interface LocaleDiff {
  locale: string;
  missing: string[];               // in source, absent from target: would be added
  changed: string[];               // source changed since last translated: would be re-translated
  orphaned: string[];              // in target, absent from source: reported only
  hasPendingChanges: boolean;      // missing.length > 0 || changed.length > 0
}
```

Hinweise sind pro Locale, auf `LocaleSummary.notices`, und decken sowohl Provider-Hinweise (zum Beispiel eine
DeepL-Herabstufung) als auch SDK-Hinweise ab (zum Beispiel eine Zielsprache, die mehr CLDR-Plural-
Kategorien benötigt, als die Quelle liefert, Code `PLURAL_CATEGORIES_INCOMPLETE`). Es gibt keine Hinweis-Liste auf oberster
Ebene in `RunSummary`.

`SdkError` ist der eine Fehlertyp, den das SDK für Fehler des gesamten Laufs wirft. Er trägt einen stabilen `code`
und eine geheimnisfreie `message`. Die Codes sind:

| Code | Wann |
| --- | --- |
| `CONFIG_NOT_FOUND` | keine Konfiguration wurde gefunden, oder ein expliziter `configPath` existiert nicht |
| `CONFIG_INVALID` | eine Konfiguration wurde gefunden, ist aber nicht parsebar oder besteht die Validierung nicht |
| `UNKNOWN_FORMAT` | für das konfigurierte Format ist kein Adapter registriert |
| `PROVIDER_CONSTRUCTION_FAILED` | der Provider konnte nicht gebaut werden (einschließlich eines fehlenden API-Schlüssels) |
| `SOURCE_UNREADABLE` | die Quell-Locale-Datei existiert nicht |
| `SOURCE_INVALID` | die Quell-Locale-Datei konnte nicht gelesen oder geparst werden |
| `LOCK_FILE_INVALID` | die Lock-Datei ist vorhanden, aber beschädigt oder zu groß |
| `LOCALE_FAILED` | nicht geworfen: der Fallback-`code`, der auf dem `error` eines fehlgeschlagenen Locales aufgezeichnet wird |

Ein einzelnes fehlschlagendes Locale wirft nicht; es wird auf der `LocaleSummary` dieses Locales mit `status:
"failed"` und einem `error` aufgezeichnet, und der Lauf wird fortgesetzt.

## Eine Konfiguration definieren

```ts
import { defineConfig } from "@verbatra/sdk";

export default defineConfig({
  sourceLocale: "en",
  targetLocales: ["de", "fr"],
  format: "i18next-json",
  files: {
    pattern: "locales/{locale}.json",
  },
  provider: {
    id: "anthropic",
    options: {
      model: "claude-sonnet-4-6",
      maxTokens: 4096,
    },
  },
});
```

## Eine einmalige Übersetzung ausführen

`translate` wird zur Lauf-Zusammenfassung aufgelöst. Lies `succeeded` und `failed` für die Schlagzeile und durchlaufe
`locales` für das Detail, einschließlich etwaiger Hinweise (Provider oder SDK).

```ts
import { loadConfig, translate } from "@verbatra/sdk";

const config = await loadConfig();
const summary = await translate({ config });

console.log(`${summary.succeeded.length} succeeded, ${summary.failed.length} failed`);

for (const locale of summary.locales) {
  for (const notice of locale.notices) {
    console.warn(`${locale.locale}: ${notice.code} ${notice.message}`);
  }
}
```

Übergib `dryRun: true`, um zu lesen und zu vergleichen, ohne einen Provider aufzurufen oder Dateien zu schreiben. Übergib
`prune: true`, um verwaiste Schlüssel zu entfernen, oder `generatePlurals: true`, um fehlende Plural-
Kategorien zu synthetisieren (nur i18next-JSON mit einem LLM-Provider); jedes überschreibt die passende Konfigurationsoption für
diesen Lauf. Anders als `prune` und `generatePlurals` ist `maxBatchSize` nur über die Konfiguration möglich (auf dem Konfigurations-
objekt gesetzt, Standard 50): Es gibt keine Pro-Lauf-`translate()`-Überschreibung dafür.

## Watch-Modus

`watch` ruft dein `onRun` einmal pro Lauf mit einem diskriminierten `WatchRunResult` auf. Das `stop()` des Controllers
fährt es sauber herunter.

```ts
import { loadConfig, watch } from "@verbatra/sdk";

const config = await loadConfig();
const controller = await watch({
  config,
  onRun: (result) => {
    if (result.status === "succeeded") {
      console.log(`${result.summary.succeeded.length} ok, ${result.summary.failed.length} failed`);
    } else {
      console.error(`run failed: ${result.error.code} ${result.error.message}`);
    }
  },
});

// Stop cleanly on Ctrl-C.
process.on("SIGINT", () => {
  void controller.stop();
});
```

## Fehler behandeln

Fehler des gesamten Laufs werfen einen `SdkError`. Schalte auf seinen `code`, um auf bestimmte Fälle zu reagieren.

```ts
import { loadConfig, SdkError, translate } from "@verbatra/sdk";

try {
  const config = await loadConfig();
  await translate({ config });
} catch (error) {
  if (error instanceof SdkError) {
    switch (error.code) {
      case "CONFIG_NOT_FOUND":
        console.error("No verbatra config found. Run `verbatra init` to scaffold one.");
        break;
      case "SOURCE_UNREADABLE":
      case "SOURCE_INVALID":
        console.error(`Could not read the source locale: ${error.message}`);
        break;
      default:
        console.error(`${error.code}: ${error.message}`);
    }
  } else {
    throw error;
  }
}
```
